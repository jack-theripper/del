# ESLint-плагин

ESLint-плагин доступен по адресу [@inngest/eslint-plugin](https://www.npmjs.com/package/@inngest/eslint-plugin) и предоставляет правила для соблюдения лучших практик при написании функций Inngest.

## Начало работы

Установите пакет с помощью предпочитаемого менеджера пакетов как [dev-зависимость](https://docs.npmjs.com/cli/v10/configuring-npm/package-json#devdependencies).

```sh
npm install -D @inngest/eslint-plugin
```

Добавьте плагин в файл конфигурации ESLint с рекомендуемыми настройками.

```json
{
  "plugins": ["@inngest"],
  "extends": ["plugin:@inngest/recommended"]
}
```

Вы также можете настроить каждое правило вручную вместо использования конфигурации `plugin:@inngest/recommended`.

```json
{
  "plugins": ["@inngest"],
  "rules": {
    "@inngest/await-inngest-send": "warn"
  }
}
```

Список всех доступных правил смотрите ниже.

### @inngest/await-inngest-send

Перед вызовом `inngest.send()` следует использовать `await` или `return`.

```json
"@inngest/await-inngest-send": "warn" // рекомендуется
```

В бессерверных средах среда выполнения нередко принудительно завершается, как только обработчик запроса завершил работу. Это означает, что незавершённые промисы, не выполненные до окончания работы обработчика, могут быть отменены.

```ts
// ❌ Плохо
inngest.send({ name: "some.event" });
```
```ts
// ✅ Хорошо
await inngest.send({ name: "some.event" });
```

#### Когда не использовать

Есть случаи, когда у вас есть более глубокий контроль над средой выполнения или когда вы безопасно выполните `await` для отправки позже. В таких случаях это правило можно отключить.

### @inngest/no-nested-steps

Использование `step.*` внутри функции `step.run()` не допускается.

```json
"@inngest/no-nested-steps": "error" // рекомендуется
```

Вложенные вызовы `step.run()` не поддерживаются и приведут к ошибке во время выполнения. Если ваши шаги вложены, они, вероятно, зависят друг от друга. В таком случае выделите их в отдельную функцию, которая выполняет их последовательно.

```ts
// ❌ Плохо
await step.run("a", async () => {
  const someValue = "...";
  await step.run("b", () => {
    return use(someValue);
  });
});
```
```ts
// ✅ Хорошо
const aThenB = async () => {
  const someValue = await step.run("a", async () => {
    return "...";
  });

  return step.run("b", async () => {
    return use(someValue);
  });
};

await aThenB();
```

### @inngest/no-variable-mutation-in-step

Не изменяйте переменные внутри `step.run()` — вместо этого возвращайте результат.

```json
"@inngest/no-variable-mutation-in-step": "error" // рекомендуется
```

Inngest выполняет вашу функцию несколько раз в ходе одного запуска, запоминая состояние по мере продвижения. Это означает, что код внутри вызовов `step.run()` вызывается не при каждом выполнении.

Это может сбивать с толку, если вы используете шаги для обновления переменных в замыкании функции:

```ts
// ❌ Плохо
// ЭТО НЕПРАВИЛЬНО! step.run выполняется только один раз и пропускается
// при последующих шагах, поэтому userID не будет определён.
let userId;

// НЕ делайте так! Вместо этого возвращайте данные из step.run.
await step.run("get-user", async () => {
  userId = await getRandomUserId();
});

console.log(userId); // undefined
```

Убедитесь, что все переменные, необходимые для функции в целом, _возвращаются_ из вызовов `step.run()`.

```ts
// ✅ Хорошо
// Это правильный способ задания переменных внутри step.run :)
const userId = await step.run("get-user", () => getRandomUserId());

console.log(userId); // 123
```

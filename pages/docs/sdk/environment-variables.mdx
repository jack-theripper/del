# Переменные окружения

Вы можете задавать переменные окружения для изменения различных параметров конфигурации Inngest.

Здесь рассматриваются все доступные переменные окружения, их значения и рекомендации по использованию.

- [INNGEST_BASE_URL](#inngest-base-url)
- [INNGEST_DEV](#inngest-dev)
- [INNGEST_ENV](#inngest-env)
- [INNGEST_EVENT_KEY](#inngest-event-key)
- [INNGEST_LOG_LEVEL](#inngest-log-level)
- [INNGEST_SERVE_ORIGIN](#inngest-serve-origin)
- [INNGEST_SERVE_PATH](#inngest-serve-path)
- [INNGEST_SIGNING_KEY](#inngest-signing-key)
- [INNGEST_STREAMING](#inngest-streaming)

В некоторых фреймворках и платформах, например Cloudflare Workers, переменные окружения недоступны в глобальной области видимости и вместо этого передаются как аргументы времени выполнения в ваш обработчик. В таком случае вы можете использовать `inngest.setEnvVars()`, чтобы убедиться, что клиент имеет корректную конфигурацию перед взаимодействием с Inngest.

```ts
// Например, в Hono на Cloudflare Workers
app.on("POST", "/my-api/send-some-event", async (c) => {
  inngest.setEnvVars(c.env);

  await inngest.send({ name: "test/event" });

  return c.json({ message: "Done!" });
});

// Также можно объединить вызов в цепочку для краткости
await inngest.setEnvVars(c.env).send({ name: "test/event" });
```

---

## INNGEST_BASE_URL

Используется для указания SDK хоста для взаимодействия с Inngest.

Если задано, должно включать протокол и порт, например `http://localhost:8288` или `https://my.tunnel.com`. Может быть переопределено явным указанием `baseUrl` в `new Inngest()`.

В большинстве случаев рекомендуется оставить это значение не заданным. Распространённый сценарий — принудительное использование production-сборкой приложения Inngest Dev Server вместо Inngest Cloud для локального интеграционного тестирования.

В таком случае предпочтительно использовать [INNGEST_DEV=1](#inngest-dev). Для Docker может быть целесообразно также задать `INNGEST_BASE_URL=http://host.docker.internal:8288`. Подробнее в [руководстве по Docker](/docs/guides/development-with-docker).

---

## INNGEST_DEV

Используется для принудительного перевода SDK в режим разработки с `INNGEST_DEV=1` или в облачный режим с `INNGEST_DEV=0`. URL dev-сервера можно задать одновременно: `INNGEST_DEV=http://localhost:8288`.

Может быть переопределено явным указанием `isDev` в `new Inngest()`.

Явная установка любого из режимов изменяет URL для взаимодействия с Inngest, а также отключает проверку подписи в режиме разработки или включает её в облачном режиме.

Если ни переменная окружения, ни параметр конфигурации не указаны, SDK по умолчанию работает в **облачном режиме**. Для локальной разработки явно установите `INNGEST_DEV=1` или `isDev: true` в конфигурации клиента.

---

## INNGEST_ENV

Используется для указания Inngest, из какой [среды Inngest](/docs/platform/environments?ref=environment-variables) вы хотите отправлять и получать события.

Может быть переопределено явным указанием `env` в `new Inngest()`.

На некоторых платформах это определяется и устанавливается автоматически, на других потребуется ручная настройка. Смотрите [Настройка сред веток](/docs/platform/environments#configuring-branch-environments?ref=environment-variables), чтобы узнать, нужна ли вам эта переменная.

---

## INNGEST_EVENT_KEY

Ключ для отправки событий в Inngest. Подробнее см. [Создание ключа события](/docs/events/creating-an-event-key?ref=environment-variables).

Может быть переопределено явным указанием `eventKey` в `new Inngest()`.

---

## INNGEST_SERVE_ORIGIN

Источник (origin), используемый для доступа к этому приложению из Inngest Cloud.

Если задано, должно включать протокол и порт, например `http://localhost:8288` или `https://my.tunnel.com`. Может быть переопределено явным указанием `serveOrigin` в `serve()`.

По умолчанию SDK пытается определить это значение на основе деталей запроса, таких как заголовок `Host`, но иногда это невозможно (например, при работе в ограниченной среде, такой как AWS Lambda, или при использовании прокси/перенаправлений).

---

## INNGEST_SERVE_PATH

Путь, используемый для доступа к этому приложению из Inngest Cloud.

Если задано, должен быть корректным URL-путём с ведущим `/`, например `/api/inngest`.

По умолчанию SDK пытается определить это значение на основе деталей запроса, но иногда это невозможно (например, при работе в ограниченной среде, такой как AWS Lambda, или при использовании прокси/перенаправлений).

---

## INNGEST_SIGNING_KEY

Ключ для подписи запросов к Inngest и от него, обеспечивающий безопасную связь. Подробнее см. [Serve — Ключ подписи](/docs/learn/serving-inngest-functions#signing-key?ref=environment-variables).

Может быть переопределено явным указанием `signingKey` в `new Inngest()`.

---

## INNGEST_SIGNING_KEY_FALLBACK

Используется только при ротации ключа подписи. Если указан, SDK автоматически повторит аутентификацию с резервным ключом при сбоях проверки основного ключа подписи.

Доступно начиная с версии `3.18.0`.

---

## INNGEST_STREAMING

Устанавливает поддержку потоковой передачи в SDK, потенциально позволяя обойти ограничения по времени ожидания запросов и другие ограничения. Подробнее см. [Streaming](/docs/streaming?ref=environment-variables).

Может принимать значения `true` или `false`.

По умолчанию имеет значение `false`, отключая потоковую передачу. Также может быть переопределено установкой `streaming` в `serve()` с теми же значениями.

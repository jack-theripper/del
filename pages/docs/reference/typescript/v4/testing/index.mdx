# Тестирование


Для программного тестирования функций Inngest используйте библиотеку `@inngest/test`,
доступную на [npm](https://www.npmjs.com/package/@inngest/test) и [JSR](https://jsr.io/@inngest/test).

Это позволяет мокировать состояние функций, инструменты шагов и входные данные с помощью
Jest-совместимого API, поддерживающего все основные тестовые фреймворки, среды выполнения и
библиотеки:

- `jest`
- `vitest`
- `bun:test` (Bun)
- `@std/expect` (Deno)
- `chai`/`expect`

## Установка

Пакет `@inngest/test` требует `inngest@>=4.0.0`.

<CodeGroup>
```shell {{ title: "npm" }}
npm install -D @inngest/test
```
```shell {{ title: "Yarn" }}
yarn add -D @inngest/test
```
```shell {{ title: "pnpm" }}
pnpm add -D @inngest/test
```
```shell {{ title: "Bun" }}
bun add -d @inngest/test
```
```shell {{ title: "Deno" }}
deno add --dev @inngest/test
# or with JSR...
deno add --dev jsr:@inngest/test
```
</CodeGroup>

## Юнит-тесты

Используйте любой поддерживаемый тестовый фреймворк; `@inngest/test` не диктует,
как запускать тесты. Здесь мы продемонстрируем с использованием `jest`.

Импортируйте `InngestTestEngine`, тестируемую функцию и создайте новый
экземпляр `InngestTestEngine`.

```ts
import { InngestTestEngine } from "@inngest/test";
import { helloWorld } from "./helloWorld";

describe("helloWorld function", () => {
  const t = new InngestTestEngine({
    function: helloWorld,
  });
});
```

Теперь можно использовать основной API для тестирования — `t.execute()`:

```ts
test("returns a greeting", async () => {
  const { result } = await t.execute();
  expect(result).toEqual("Hello World!");
});
```

Это выполнит всю функцию (включая шаги) до завершения, затем вернёт
ответ функции, где мы проверяем, что это строка `"Hello
World!"`.

Сериализованная `error` будет возвращена вместо `result`, если функция выбросила исключение:

```ts
test("throws an error", async () => {
  const { error } = await t.execute();
  expect(error).toContain("Some specific error");
});
```

<Tip>
  При использовании шагов, задерживающих выполнение, таких как `step.sleep` или `step.waitForEvent`, их необходимо мокировать. [Узнайте больше о мокировании шагов](#steps).
</Tip>

### Запуск отдельного шага

`t.executeStep()` можно использовать для выполнения функции до тех пор, пока не выполнится определённый шаг.

Это полезно для тестирования отдельного шага внутри функции или для проверки того, что
невыполнимый шаг, например `step.waitForEvent()`, был зарегистрирован с
правильными опциями.

```ts
test("runs the price calculations", async () => {
  const { result } = await t.executeStep("calculate-price");
  expect(result).toEqual(123);
});
```

Утверждения также можно делать для шагов в любой части запуска, независимо от того,
является ли это проверяемым чекпоинтом. Смотрите [Утверждения -> State](#assertions).

### Утверждения

`@inngest/test` по умолчанию добавляет Jest-совместимые моки, которые могут помочь вам проверить
входные и выходные данные функций и шагов. Можно проверять:

- Входные данные функции
- Выходные данные функции
- Выходные данные шага
- Использование инструментов шага

Все эти значения возвращаются из обоих `t.execute()` и `t.executeStep()`;
для простоты здесь показан только один.

Возвращается `result` — вывод запуска или шага:

```ts
const { result } = await t.execute();
expect(result).toEqual("Hello World!");
```

`ctx` — это входные данные, использованные для запуска функции. Это можно использовать для проверки выводов,
основанных на входных данных, таких как `event` или `runId`, или для подтверждения того, что
промежуточное ПО работает правильно и влияет на входные аргументы.

```ts
const { ctx, result } = await t.execute();
expect(result).toEqual(`Run ID was: "${ctx.runId}"`);
```

Инструменты шагов в `ctx.step` — это все Jest-совместимые функции-шпионы, поэтому можно
использовать их для проверки того, что они были вызваны и использованы правильно:

```ts
const { ctx } = await t.execute();
expect(ctx.step.run).toHaveBeenCalledWith("my-step", expect.any(Function));
```

Также возвращается `state` — представление выводов всех шагов в
запуске. Это позволяет тестировать вывод каждого отдельного шага для любых входных данных:

```ts
const { state } = await t.execute();
expect(state["my-step"]).resolves.toEqual("some successful output");
expect(state["dangerous-step"]).rejects.toThrowError("something failed");
```

### Мокирование

Часть мокирования выполняется `@inngest/test` автоматически, но при необходимости может быть перезаписана.

Все описанные ниже моки можно указать как при создании экземпляра
`InngestTestEngine`, так и для каждого отдельного выполнения:

```ts
// Set the events for every execution
const t = new InngestTestEngine({
  function: helloWorld,
  // mocks here
});

// Or for just one, which will overwrite any current event mocks
t.execute({
  // mocks here
});

t.executeStep("my-step", {
  // mocks here
})
```

Также можно клонировать существующий экземпляр `InngestTestEngine` для повторного использования
сложных моков:

```ts
// Make a direct clone, which includes any mocks
const otherT = t.clone();

// Provide some more mocks in addition to any existing ones
const anotherT = t.clone({
  // mocks here
});
```

Для простоты следующие примеры показывают использование `t.execute()`, но
моки можно размещать в любом из этих мест.

#### События

Входящие данные событий можно мокировать. Они всегда указываются в виде массива
событий, чтобы позволить мокировать также батчи.

```ts
t.execute({
  events: [{ name: "demo/event.sent", data: { message: "Hi!" } }],
});
```

<Callout>
Если моки событий вообще не заданы (или явно указано `events: undefined`),
для вас будет автоматически создан мок события `inngest/function.invoked`.
</Callout>

#### Шаги

Мокирование шагов помогает моделировать различные пути и ситуации внутри вашей
функции. Для этого любой шаг можно замокировать, передав опцию `steps`. Всегда следует мокировать шаги `sleep` и `waitForEvent` — [узнайте больше здесь](#sleep-and-wait-for-event).

Здесь мы мокируем два шага: один, который выполнится успешно, и другой, который
смоделирует сбой и выбросит ошибку:

```ts
t.execute({
  steps: [
    {
      id: "successful-step",
      handler() {
        return "We did it!";
      },
    },
    {
      id: "dangerous-step",
      handler() {
        throw new Error("Oh no!");
      },
    },
  ],
});
```

Эти обработчики будут лениво выполняться при обнаружении во время выполнения функции.
Это означает, что можно писать сложные моки, реагирующие на другую информацию:

```ts
let message = "";

t.execute({
  steps: [
    {
      id: "build-greeting",
      handler() {
        message = "Hello, ";
        return message;
      },
    },
    {
      id: "build-name",
      handler() {
        return message + " World!";
      },
    },
  ],
});
```

#### Sleep и waitForEvent

Шаги, приостанавливающие функцию — `step.sleep`, `step.sleepUntil` и `step.waitForEvent` — всегда следует мокировать.

<CodeGroup>
```ts {{ title: 'step.sleep' }}
// Given the following function that sleeps
const myFunction = inngest.createFunction(
  {
    id: "my-function",
    triggers: { event: "user.created" },
  },
  async ({ event, step }) => {
    await step.sleep("one-day-delay", "1d");
    return { message: "success" };
  }
)
// Mock the step to execute a no-op handler to return immediately
t.execute({
  steps: [
    {
      id: "one-day-delay",
      handler() {}, // no return value necessary
    },
  ],
});
```
```ts {{ title: "step.waitForEvent" }}
// Given the following function that sleeps
const myFunction = inngest.createFunction(
  {
    id: "my-function",
    triggers: { event: "time_off.requested" },
  },
  async ({ event, step }) => {
    const evt = await step.waitForEvent("wait-for-approval", {
      event: "manager.approved",
      timeout: "1d",
    });
    return { message: evt?.data.message };
  }
)
// Mock the step to return null to simulate a timeout
t.execute({
  steps: [
    {
      id: "wait-for-approval",
      handler() {
        // A timeout will return null
        return null;
      },
    },
  ],
});
// Mock the step to return an event
t.execute({
  steps: [
    {
      id: "wait-for-approval",
      handler() {
        // If the event is approved, it will be returned
        return {
          name: 'manager.approved',
          data: {
            message: 'This looks great!'
          }
        };
      },
    },
  ],
});
```
</CodeGroup>

#### Модули и импорты

Мокирование модулей или импортов вне Inngest, на которые могут полагаться ваши функции,
следует выполнять за пределами Inngest с помощью используемого вами тестового фреймворка.

Вот ссылки на основные поддерживаемые фреймворки и их рекомендации по
мокированию импортов:

- [`jest`](https://jestjs.io/docs/mock-functions#mocking-modules)
- [`vitest`](https://vitest.dev/guide/mocking#modules)
- [`bun:test` (Bun)](https://bun.sh/docs/test/mocks#module-mocks-with-mock-module)
- [`@std/testing` (Deno)](https://jsr.io/@std/testing/doc/mock/~)

#### Кастомное мокирование

Также можно предоставить собственные кастомные моки для входных данных функции.

При создании экземпляра `InngestTestEngine` или запуске выполнения передайте функцию
`transformCtx`, которая будет добавлять эти моки каждый раз при запуске функции:

```ts
const t = new InngestTestEngine({
  function: helloWorld,
  transformCtx: (ctx) => {
    return {
      ...ctx,
      event: someCustomThing,
    };
  },
});
```

Если вы хотите сохранить автоматическое мокирование из `@inngest/test` (например,
шпионы на `ctx.step.*`), можно импортировать и использовать автоматические преобразования в рамках своего:

```ts
import { InngestTestEngine, mockCtx } from "@inngest/test";

const t = new InngestTestEngine({
  function: helloWorld,
  transformCtx: (ctx) => {
    return {
      ...mockCtx(ctx),
      event: someCustomThing,
    };
  },
});
```

# Обработка сбоев


Определяйте обработчики сбоев для вашей функции с помощью опции [`onFailure`](/docs/reference/typescript/v4/functions/create#configuration). Эта функция будет автоматически вызвана, когда ваша функция завершится ошибкой после максимального количества повторных попыток. Альтернативно, можно использовать системное событие [`"inngest/function.failed"`](/docs/reference/system-events/inngest-function-failed) для обработки сбоев всех функций.

```ts
import { inngest } from "./client";

export default inngest.createFunction(
  {
    id: "import-product-images",
    triggers: { event: "shop/product.imported" },
    onFailure: async ({ error, event, step }) => {
      // Это обработчик сбоя, который можно использовать для
      // отправки уведомления или любых других необходимых действий
    },
  },
  async ({ event, step, runId }) => {
    // Это код основного обработчика функции
  }
);
```

Обработчик сбоев очень полезен для:

* Отправки оповещений команде
* Отправки метрик в сторонний инструмент мониторинга (например, Datadog)
* Отправки уведомления команде или пользователю о сбое задания
* Выполнения откатов транзакций (отмены работы, частично выполненной основным обработчиком)

_Сбои_ не следует путать с _ошибками_, которые будут повторно обработаны. Читайте [документацию по обработке ошибок и повторным попыткам](/docs/functions/retries) для контекста.

---

## Как работает `onFailure`

Обработчик `onFailure` — это вспомогательный механизм, создающий отдельную функцию Inngest специально для обработки сбоев основного обработчика.

Отдельная функция Inngest использует системное событие [`"inngest/function.failed"`](/docs/reference/system-events/inngest-function-failed), которое отправляется в ваш аккаунт при каждом сбое функции. Функция, созданная через `onFailure`, будет отображаться в панели управления как отдельная функция с именем в формате: `"<Имя вашей функции> (failure)"`.

## `onFailure({ error, event, step, runId })`

Обработчик `onFailure` имеет те же аргументы, что и [основной обработчик функции](/docs/reference/typescript/v4/functions/create#handler), но также получает аргумент `error`.

### `error`

Объект JavaScript [`Error`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Error/Error) как брошенный в последней попытке основного обработчика.

<Callout variant="warning">
SDK Inngest пытается сериализовать и десериализовать объект `Error` в меру своих возможностей, и любые пользовательские классы ошибок (например, `Prisma.PrismaClientKnownRequestError` или `MyCustomErrorType`) будут десериализованы как базовый объект `Error`. Это означает, что в `onFailure` нельзя использовать `instanceof` для определения типа ошибки.
</Callout>

### `event`

Объект полезной нагрузки системного события [`"inngest/function.failed"`](/docs/reference/system-events/inngest-function-failed). Похож на обычную полезную нагрузку события, но содержит данные, специфичные для последней попытки неудавшейся функции. [Смотрите полный справочник по этой полезной нагрузке](/docs/reference/system-events/inngest-function-failed).

### `step`

[Смотрите справочник по `step` в документации по созданию функций](/docs/reference/typescript/v4/functions/create#step).

### `runId`

Это будет ID запуска функции обработки ошибки, _а не функции, которая завершилась ошибкой_. Для получения ID неудавшего запуска используйте `event.data.run_id`. [Подробнее о `runId`](/docs/reference/typescript/v4/functions/create#run-id).

## Примеры

### Отправка уведомления в Slack при сбое функции

В этом примере функция пытается синхронизировать все продукты из магазина Shopify, а при сбое отправляет сообщение в канал _#eng-alerts_ команды в Slack, используя метод `chat.postMessage` Slack Web API ([документация](https://api.slack.com/methods/chat.postMessage)).

```ts
import { client } from "@slack/web-api";
import { inngest } from "./client";

export default inngest.createFunction(
  {
    id: "sync-shopify-products",
    triggers: { event: "shop/product_sync.requested" },
    // Обработчик должен быть асинхронной функцией:
    onFailure: async ({ error, event }) => {
      const originalEvent = event.data.event;

      // Отправить сообщение в канал оповещений инженерной команды в Slack:
      const result = await client.chat.postMessage({
        token: process.env.SLACK_TOKEN,
        channel: "C12345",
        blocks: [
          {
            type: "section",
            text: {
              type: "mrkdwn",
              text: `Sync Shopify function failed for Store ${
                originalEvent.storeId
              }: ${error.toString()}`,
            },
          },
        ],
      });

      return result;
    },
  },
  async ({ event, step, runId }) => {
    // Это код основного обработчика функции
    const products = await step.run("fetch-products", async () => {
      const storeId = event.data.storeId;
      // Функция может завершиться ошибкой здесь или...
    });
    await step.run("save-products", async () => {
      // Функция может завершиться ошибкой здесь после максимального числа попыток
    });
  }
);
```

### Захват всех ошибок сбоев с помощью Sentry

Как и в примере выше, можно захватывать ошибки всех неудавшихся функций и отправлять их в единое место. Вот пример с использованием [библиотеки Sentry для Node.js](https://docs.sentry.io) для захвата и отправки всех ошибок сбоев в Sentry.

```ts
import * as Sentry from "@sentry/node";
import { inngest } from "./client";

Sentry.init({
  dsn: "https://examplePublicKey@o0.ingest.sentry.io/0",
});

export default inngest.createFunction(
  {
    name: "Send failures to Sentry",
    id: "send-failed-function-errors-to-sentry",
    triggers: { event: "inngest/function.failed" },
  },
  async ({ event, step }) => {

    // Ошибка сериализована как JSON, поэтому нужно пересоздать её для обработчика ошибок Sentry:
    const error = event.data.error;
    const reconstructedEvent = new Error(error.message);
    // Установить имя в новом событии:
    // Можно даже настроить имя, например:
    // `Function Failure: ${event.data.function_id} - ${error.name}`
    reconstructedEvent.name = error.name;

    // Добавить стек вызовов к ошибке:
    reconstructedEvent.stack = error.stack;

    // Захватить ошибку в Sentry с дополнительными тегами и метаданными:
    Sentry.captureException(reconstructedEvent,{
      extra: {
        function_id: event.data.function_id,
      },
    });

    // Сбросить очередь Sentry, чтобы убедиться, что ошибка отправлена:
    return await Sentry.flush();
  }
);
```

# Serve

Обработчик API `serve()` используется для обслуживания [функций](/docs/reference/typescript/v4/functions/create) вашего приложения через HTTP. Этот обработчик позволяет Inngest удалённо и безопасно читать конфигурацию ваших функций и вызывать код функций. Это позволяет размещать код функций на любой платформе.

```ts
import { serve } from "inngest/next"; // or your preferred framework
import { inngest } from "./client";
import {
  importProductImages,
  sendSignupEmail,
  summarizeText,
} from "./functions";

serve({
  client: inngest,
  functions: [sendSignupEmail, summarizeText, importProductImages],
});

```

Обработчики `serve` импортируются из удобных пакетов для конкретных фреймворков, таких как `"inngest/next"`, `"inngest/express"` или `"inngest/lambda"`. [Нажмите здесь для полного списка официально поддерживаемых фреймворков](/docs/learn/serving-inngest-functions). Для любого неподдерживаемого фреймворка можно [создать кастомный обработчик](#custom-frameworks).

## `serve(options)`

<Properties>
  <Property name="client" type="Inngest client" required>
    Клиент Inngest ([справочник](/docs/reference/typescript/v4/client/create)).
  </Property>
  <Property name="functions" type="InngestFunctions[]" required>
    Массив функций Inngest, созданных с помощью `inngest.createFunction()` ([справочник](/docs/reference/typescript/v4/functions/create)).
  </Property>
  <Property name="serveOrigin" type="string">
    Домен вашего приложения _с_ протоколом, например `https://myapp.com`. SDK пытается определить это через HTTP-заголовки во время выполнения, но это может быть необходимо при использовании таких платформ, как AWS Lambda, или при использовании обратного прокси. Смотрите также [`INNGEST_SERVE_ORIGIN`](/docs/sdk/environment-variables#inngest-serve-origin).
  </Property>
  <Property name="servePath" type="string">
    Путь, по которому размещён ваш обработчик `serve`. SDK пытается определить это через HTTP-заголовки во время выполнения. Рекомендуем `/api/inngest`. Смотрите также [`INNGEST_SERVE_PATH`](/docs/sdk/environment-variables#inngest-serve-path).
  </Property>
  <Property name="streaming" type={`true | false`}>
    Включает потоковые ответы обратно в Inngest, что может обеспечить максимальные тайм-ауты serverless-функций. Смотрите [справочник](/docs/streaming) для получения дополнительной информации о конфигурации. Смотрите также [`INNGEST_STREAMING`](/docs/sdk/environment-variables#inngest-streaming).
  </Property>
  <Property name="id" type="string">
    ID для представления этого приложения вместо ID клиента. Полезно при создании многих эндпоинтов Inngest в одном приложении.
  </Property>
</Properties>

<Callout>
  Опции `signingKey`, `signingKeyFallback`, `logger`, `baseUrl` и `fetch` настраиваются на [клиенте Inngest](/docs/reference/typescript/v4/client/create), а не в `serve()`. Мы всегда рекомендуем устанавливать переменную окружения [`INNGEST_SIGNING_KEY`](/docs/sdk/environment-variables#inngest-signing-key) вместо использования опции `signingKey` напрямую. Как и с любым секретом, хранить signing key в коде — плохая практика.
</Callout>

## Как работает обработчик API `serve`

API работает, предоставляя единый эндпоинт по пути `/api/inngest`, который обрабатывает различные действия с использованием HTTP-методов запросов:

- `GET`: Возвращает метаданные функций и отображает страницу в **только в режиме разработки**.
- `POST`: Вызывает функции с телом запроса в качестве входящего состояния функции.
- `PUT`: Запускает SDK для регистрации всех функций в Inngest с использованием signing key.

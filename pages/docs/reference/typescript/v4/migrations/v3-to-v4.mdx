# Руководство по миграции TypeScript SDK: с v3 на v4

Это руководство поможет вам перейти с Inngest TypeScript SDK v3 на v4.

## Новые возможности

- Промежуточное ПО стало мощнее и интуитивнее ([подробнее](/docs/reference/typescript/v4/middleware/lifecycle))
- Хелперы триггеров ([подробнее](/docs/reference/typescript/v4/functions/triggers))
- Раздельные внутренние и пользовательские логи с опцией `internalLogger` ([подробнее](/docs/reference/typescript/v4/logging#internal-logger))
- Использование воркер-потока в Connect ([подробнее](/docs/reference/typescript/v4/migrations/v3-to-v4#connect-worker-thread))

## Промпт для обновления

Вы можете использовать любой coding-агент для обновления до v4. Вот рекомендуемый промпт:

```plaintext
Upgrade Inngest TypeScript SDK from v3 to v4

Read the migration guide at https://www.inngest.com/docs-markdown/reference/typescript/v4/migrations/v3-to-v4 and apply every breaking change to this codebase. Pay special attention to:

- Triggers moving into the options object (1st arg of createFunction)
- EventSchemas being replaced with eventType() and staticSchema()
- Serve options (signingKey, baseUrl, etc.) moving to the client constructor
- step.invoke() no longer accepting string function IDs
- Set `maxRuntime` option for `checkpointing` if running on serverless.

Install the latest v4 package and verify TypeScript compilation passes afterward.
```

## Критические изменения

### Переработка промежуточного ПО

Система промежуточного ПО была полностью переработана. Ознакомьтесь с документацией по [промежуточному ПО](/docs/reference/typescript/v4/middleware/lifecycle), чтобы узнать о новом API.

Это руководство описывает миграцию между v3 и v4 версиями пакета `inngest`.

### Режим по умолчанию теперь "cloud"

Режим по умолчанию теперь `cloud` вместо `dev`. Это предотвращает случайные деплои в продакшен в режиме разработки и согласуется со всеми другими SDK Inngest.

**Что это означает:**

- В режиме `cloud` требуется signing key (через `INNGEST_SIGNING_KEY` или опцию `signingKey`)
- Для локальной разработки явно установите `isDev: true` на клиенте или задайте `INNGEST_DEV=1`

```typescript
// Local development
const inngest = new Inngest({ id: "my-app", isDev: true });

// Production (signing key required via env or option)
const inngest = new Inngest({ id: "my-app" });
```

или

```sh
INNGEST_DEV=1 pnpm run dev
```


Вы столкнулись с этой проблемой, если ваша ошибка выглядит примерно так:

```
Error: Inngest error: A signing key is required to run in Cloud mode, but no signing key was found.

To fix this, choose one of the following:
  - For local development, set INNGEST_DEV=1 to use the Dev Server (e.g. INNGEST_DEV=1 npm run dev)
  - For production, set the INNGEST_SIGNING_KEY environment variable

Find your keys at https://app.inngest.com
```

### Триггер события теперь в опциях

В предыдущих версиях SDK триггеры для событий указывались вторым аргументом `createFunction`. Мы изменили это, потому что триггеры семантически логичнее как опции, _и_ случай задания функции без триггеров требовал передачи пустого массива, что нам не нравилось.

Например, старый синтаксис:

```ts
// Old (v3)
inngest.createFunction(
  { id: "fn-id" },
  { event: "fn/trigger-event" },
  async ({ event }) => {
    // ...
  }
)

// New (v4)
inngest.createFunction(
  { id: "fn-id", triggers: { event: "fn/trigger-event" } },
  async ({ event }) => {
    // ...
  }
)
```

Несколько триггеров передаются в виде массива:

```ts
triggers: [
  { event: "fn/trigger-event" },
  { cron: "1 */2 * * *" }
]
```

Функция без триггеров — это просто функция без указания триггеров.

### Схемы событий заменены типами событий

Централизованная опция схем на клиенте Inngest (класс `EventSchemas`) удалена. Вместо неё используйте функцию `eventType()` для определения типов событий, которые используются при отправке событий, ожидании событий и триггерах событий. Смотрите [справочник по хелперам триггеров](/docs/reference/typescript/v4/functions/triggers) для полной документации по `eventType()`, `cron()`, `invoke()` и `staticSchema()`.

```ts
// Old (v3) - centralized type-only schemas on the client
const inngest = new Inngest({
  id: "my-app",
  schemas: new EventSchemas().fromRecord<{
    "user/created": { data: { userId: string; email: string } };
  }>(),
});
inngest.createFunction(
  { id: "on-user-created" },
  { event: "user/created" },
  async ({ event }) => {
    // event.data typed from centralized schema
  }
);

// New (v4) - decentralized event types with optional runtime validation
const inngest = new Inngest({ id: "my-app" });
const userCreated = eventType("user/created", {
  schema: z.object({ userId: z.string(), email: z.string() }),
});
inngest.createFunction(
  { id: "on-user-created", triggers: [userCreated] },
  async ({ event }) => {
    // event.data typed as { userId: string; email: string }
  }
);
```

Использование библиотеки схем времени выполнения (например, Zod) приведёт к проверке типов во время выполнения. Если проверка типов во время выполнения не нужна, используйте вместо этого `staticSchema()`:

```ts
  type UserCreatedPayload = { userId: string; email: string };

  const userCreated = eventType("user/created", {
    schema: staticSchema<UserCreatedPayload>(),
  });
```

Типы событий также можно использовать с `step.waitForEvent` и `inngest.send`:

```ts
// Return value is inferred and validated using `userCreated`'s schema
const user = await step.waitForEvent("wait", {
  event: userCreated,
  timeout: "7d",
});

// Sent data is validated using `userCreated`'s schema
await inngest.send(userCreated.create({ userId: "1", email: "a@b.com" }));
```

<Tip>
  Обратите внимание: `staticSchema` ожидает `type`, а не `interface`. Возможно, потребуется преобразовать существующие интерфейсы в типы.
</Tip>

### Переименование `serveHost` в `serveOrigin`

Переименуйте опцию `serveHost` в `serveOrigin`, чтобы лучше отражать её назначение. Использование "host" было фактически неверным, поскольку можно указать схему и порт, тогда как "host" — это только домен или IP.

Переменная окружения `INNGEST_SERVE_HOST` по-прежнему поддерживается для обратной совместимости, но выдаст предупреждение об устаревании. Пожалуйста, перейдите на `INNGEST_SERVE_ORIGIN`.

### Опции serve перенесены на клиент

Многие опции, ранее передававшиеся функции `serve`, были перемещены на уровень `client`. Эти свойства логичнее на этом уровне, и поскольку это связано только с потенциальной реорганизацией места указания значений, миграция должна быть очень простой.

Опции, которые может потребоваться реорганизовать:

- **baseUrl**
- **fetch**
- **signingKey**
- **signingKeyFallback**

Если вы передаёте любое из этих значений функции `serve` или функции `createServer`, вам нужно изменить код так, чтобы они предоставлялись клиенту.

```typescript
// Old (v3)
const inngest = new Inngest({ id: "my-app" });
app.use(
  "/api/inngest",
  serve({
    client: inngest,
    functions,
    signingKey: "my-signing-key",
    signingKeyFallback: "my-fallback-key",
    baseUrl: "https://my-inngest-instance.example.com",
  })
);

// New (v4)
const inngest = new Inngest({
  id: "my-app",
  signingKey: "my-signing-key",
  signingKeyFallback: "my-fallback-key",
  baseUrl: "https://my-inngest-instance.example.com",
});
app.use("/api/inngest", serve({ client: inngest, functions }));
```

Если вы использовали переменные окружения (например, `INNGEST_SIGNING_KEY`) вместо явной передачи этих опций, изменений не требуется — клиент автоматически читает из окружения.

### Удаление опции `logLevel`

Уровень логирования теперь является исключительной ответственностью объекта логгера, передаваемого в опцию `logger` клиента.

Уровень логгера по умолчанию — `info`. Чтобы изменить его, можно вручную передать наш логгер по умолчанию:

```ts
import { ConsoleLogger, Inngest } from "inngest";

export const inngest = new Inngest({
    id: "my-app",
    logger: new ConsoleLogger({ level: "debug" })
});
```

### Упрощение опции `streaming`

Опция `streaming` в `serve()` упрощена с `"allow" | "force" | false` до `true | false`.

- `"force"` → `true` (включить стриминг; выбрасывает ошибку, если обработчик его не поддерживает)
- `"allow"` → удалено (используйте `true`)
- `false` → `false` (без изменений)

```typescript
// Old (v3)
serve({ client, functions, streaming: "force" });

// New (v4)
serve({ client, functions, streaming: true });
```

Если функция `serve` не поддерживает стриминг, она выбрасывает ошибку. Ранее опция игнорировалась молча.

### Оптимизированный параллелизм включён по умолчанию

"Оптимизированный параллелизм" значительно уменьшает количество запросов, необходимых для выполнения параллельных шагов (шаги в `Promise.all`, `Promise.allSettled` и т.д.). Это снижение запросов может значительно улучшить использование CPU, памяти и длительность запуска функций.

Основным недостатком является изменение поведения `Promise.race`. `Promise.race` будет ждать завершения *всех* промисов перед разрешением. Возвращается правильный "победитель", но `Promise.race` *не* разрешится немедленно на первом победителе. Если вы полагались на поведение раннего разрешения, используйте новый хелпер `group.parallel()`. Это отключит оптимизированный параллелизм для групп шагов:

```typescript
// Old behavior (no longer works as expected with optimized parallelism)
const winner = await Promise.race([
  step.run("a", () => "a"),
  step.run("b", () => "b"),
]);

// New approach using group.parallel() from the function context
const winner = await group.parallel(async () => {
  return Promise.race([
    step.run("a", () => "a"),
    step.run("b", () => "b"),
  ]);
});
```

Чтобы полностью отключить оптимизированный параллелизм на уровне клиента или функции, установите `optimizeParallelism: false`.

### Чекпоинтинг включён по умолчанию

[Чекпоинтинг](/docs/setup/checkpointing) теперь включён по умолчанию для всех функций. Это означает, что несколько шагов могут выполняться в рамках одного запроса, что значительно снижает задержку и использование полосы пропускания.

Если ваши функции работают на serverless-платформах, таких как Vercel, следует настроить опцию `maxRuntime` немного ниже максимальной продолжительности функции:

```ts
const inngest = new Inngest({
  id: 'my-app',
  checkpointing: {
    maxRuntime: '50s', // 50s might be a good option if your max duration is 60s
  }
})
```

<Warning>
  Многие платформы, например Vercel, позволяют настроить максимальную продолжительность для каждой функции, например для эндпоинта `/api/inngest`. Рекомендуем устанавливать `maxRuntime` на 60-80% от максимальной продолжительности.

  Для приложений Vercel следует явно задать `maxDuration` ([документация](https://vercel.com/docs/functions/configuring-functions/duration)) для `/api/inngest`. Например:
  ```ts app/api/inngest/route.ts
  import { serve } from 'inngest/next';
  import { inngest, functions } from '../../inngest';

  // This endpoint can run for a maximum of 300 seconds
  export const maxDuration = 300;

  export default serve({ client: inngest, functions });
  ```

  Узнайте больше о настройке Vercel [здесь]().
</Warning>

Чтобы отключить чекпоинтинг, установите `checkpointing: false` на клиенте или отдельных функциях:

```typescript
// Disable for all functions
const inngest = new Inngest({ id: "my-app", checkpointing: false });

// Disable per-function
inngest.createFunction(
  { id: "my-fn", checkpointing: false },
  async ({ step }) => {
    // ...
  }
);
```

### Connect: `rewriteGatewayEndpoint` заменён на `gatewayUrl`

Опция обратного вызова `rewriteGatewayEndpoint` удалена из `connect()`. Используйте строковую опцию `gatewayUrl` или переменную окружения `INNGEST_CONNECT_GATEWAY_URL`.

```typescript
// Old (v3)
const connection = await connect({
  apps: [...],
  rewriteGatewayEndpoint: (url) => {
    const clusterUrl = new URL(url);
    clusterUrl.host = 'my-cluster-host:8289';
    return clusterUrl.toString();
  },
});

// New (v4)
const connection = await connect({
  apps: [...],
  gatewayUrl: "wss://my-cluster-host:8289/v0/connect",
});
```

### Воркер-поток Connect

Внутренние компоненты Connect (например, WebSocket-соединение) теперь находятся в воркер-потоке. Это решает проблему, когда голодание цикла событий (например, CPU-интенсивная работа) блокировало heartbeat'ы, заставляя сервер Inngest думать, что воркер завис.

Это изменение _не должно_ нарушать пользовательское поведение, но его полезно знать. Если вы заметите проблему с Connect, можно переместить WebSocket-соединение обратно в главный поток, установив `isolateExecution: false` в опциях `connect()` или задав переменную окружения `INNGEST_CONNECT_ISOLATE_EXECUTION` в `false`.

### Улучшения для edge-окружений

Fetch и конфигурация теперь разрешаются лениво при первом использовании, а не при создании клиента. Это означает, что больше не нужно вручную привязывать `globalThis.fetch` перед созданием клиента Inngest в edge-окружениях (Cloudflare Workers, Vercel Edge, Deno и т.д.).

### Удалена поддержка строковых ID функций в `step.invoke()`

Передача строки напрямую в `step.invoke()` больше не поддерживается. Используйте `referenceFunction()` или передайте импортированный экземпляр функции.

```typescript
// Old (v3) - No longer works
await step.invoke("my-step", {
  function: "my-app-other-fn",
  data: { foo: "bar" },
});

// New (v4) - Use referenceFunction for cross-app invocation
await step.invoke("my-step", {
  function: referenceFunction({ appId: "my-app", functionId: "other-fn" }),
  data: { foo: "bar" },
});

// Or pass an imported function instance directly
await step.invoke("my-step", {
  function: otherFn,
  data: { foo: "bar" },
});
```

Хелпер `referenceFunction()` обеспечивает типобезопасность и избавляет от ошибок ручного формирования строки `appId-functionId`.

# Extended Traces (OpenTelemetry)


Inngest поддерживает OpenTelemetry для распределённой трассировки и наблюдаемости ваших функций. `extendedTracesMiddleware` автоматически инструментирует функции Inngest и интегрируется с существующими провайдерами OpenTelemetry, давая глубокое понимание выполнения функций, времени шагов и производительности. Включает [автоматическую инструментацию](#instrumentation) многих популярных библиотек.

## Базовое использование

<Callout>
  Импортируйте и запустите `extendedTracesMiddleware()` перед любым другим кодом.

  Это гарантирует, что [провайдер трассировщика](https://opentelemetry.io/docs/concepts/signals/traces/#tracer-provider)
  и любая [инструментация](https://opentelemetry.io/docs/concepts/instrumentation/)
  успеют патчить код для сбора трассировок и спанов из всех частей приложения.
  Запуск `extendedTracesMiddleware()` после другого кода рискует не инструментировать его.
</Callout>

Например:

```ts
// Импортировать это первым
import { extendedTracesMiddleware } from "inngest/experimental";
const extendedTraces = extendedTracesMiddleware();

// Затем всё остальное
import { Inngest } from "inngest";

const inngest = new Inngest({
  id: "my-app",
  middleware: [extendedTraces],
});
```

### Serverless

Если вы используете serverless, точкой входа приложения, вероятно, будет файл
конкретного эндпоинта, например `/api/inngest`.

Если ваш клиент настроен как в примере выше, убедитесь, что вы импортируете
его первым, чтобы у провайдера было время для инициализации.

```ts
// Импортировать клиент первым
import { inngest } from "@/inngest";

// Затем всё остальное
import { serve } from "inngest/next";
import { myFn } from "@/inngest/functions";

export const { GET, POST, PUT } = serve({
  client: inngest,
  functions: [myFn],
});
```

### Расширение существующих провайдеров

JavaScript-процесс может иметь только один провайдер OpenTelemetry. Некоторые
библиотеки, например Sentry, также создают собственный провайдер.

`extendedTracesMiddleware()` сначала попытается _расширить_ существующий провайдер и создаст
новый только если не найдёт существующего. При расширении существующего провайдера мы не
будем добавлять автоматическую [инструментацию](#instrumentation).

В случае Sentry `extendedTracesMiddleware()` расширит провайдер Sentry при условии, что
запущен после `Sentry.init()`.

<Note>
  Это расширение также должно работать для провайдеров OpenTelemetry, встроенных в
  среду выполнения, например [OpenTelemetry в Deno](https://docs.deno.com/runtime/fundamentals/open_telemetry/).
</Note>

Это поведение можно изменить:

```ts
extendedTracesMiddleware({
  behaviour: "auto",
});
```

Доступные варианты:

- `"auto"` (по умолчанию): попытаться расширить провайдер, если он существует, иначе создать новый; ошибка если ни то, ни другое не сработало
- `"extendProvider"`: только попытаться расширить провайдер; ошибка если нет существующего
- `"createProvider"`: только попытаться создать провайдер; ошибка если не получилось
- `"off"`: ничего не делать

При намерении использовать `extendedTracesMiddleware()` только для расширения существующего
провайдера больше не нужно убеждаться, что он вызывается раньше остального кода.

### Ручное расширение

Если вы уже вручную создаёте собственный провайдер трассировки и порядок импортов
проблематичен, можно вручную добавить `InngestSpanProcessor` Inngest в
существующую настройку.

Добавьте `InngestSpanExporter` в ваш провайдер:

```ts
// Создайте клиент как обычно
import { Inngest } from "inngest";
import { extendedTracesMiddleware } from "inngest/experimental";

export const inngest = new Inngest({
  id: "my-app",
  middleware: [
    extendedTracesMiddleware({
      // Убедитесь, что промежуточное ПО не пытается инструментировать автоматически
      behaviour: "off",
    }),
  ],
});

// Затем при создании провайдера передайте клиент в него
import { inngest } from "@/inngest";
import { BasicTracerProvider } from "@opentelemetry/sdk-trace-base";
import { InngestSpanProcessor } from "inngest/experimental";

const provider = new BasicTracerProvider({
  // Добавьте процессор спанов при создании провайдера
  spanProcessors: [new InngestSpanProcessor(inngest)],
});

// Зарегистрируйте провайдер глобально
provider.register();
```

### Кастомная инструментация

Можно добавлять дополнительные кастомные инструментации для более глубокого понимания
вашего стека.

Например, добавление [Prisma
OpenTelemetry](https://www.prisma.io/docs/orm/prisma-client/observability-and-logging/opentelemetry-tracing):

```ts
import { PrismaInstrumentation } from "@prisma/instrumentation";
import { extendedTracesMiddleware } from "inngest/experimental";

const extendedTraces = extendedTracesMiddleware({
  instrumentations: [new PrismaInstrumentation()],
});
```

# Примеры промежуточного ПО

Практические примеры использования API промежуточного ПО на основе классов v4. Смотрите [Жизненный цикл](/docs/reference/typescript/v4/middleware/lifecycle) для полного справочника хуков.

## Внедрение зависимостей

Внедрение клиента [Prisma](https://www.prisma.io/) во все функции через `transformFunctionInput`. Типы автоматически выводятся, поэтому функции видят типизированное свойство `prisma`.

```ts
import { Middleware } from "inngest";
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

class PrismaMiddleware extends Middleware.BaseMiddleware {
  transformFunctionInput(args: Middleware.TransformFunctionInputArgs) {
    return {
      ...args,
      ctx: {
        ...args.ctx,
        prisma,
      },
    };
  }
}

const inngest = new Inngest({
  id: "my-app",
  middleware: [PrismaMiddleware],
});
```

```ts
// prisma типизирован и доступен в каждой функции
inngest.createFunction(
  { id: "create-audit-log", triggers: { event: "app/user.loggedin" } },
  async ({ prisma, event }) => {
    await prisma.auditTrail.create({
      data: { userId: event.data.userId, action: "login" },
    });
  }
);
```

## Заголовки запроса в контекст

Используйте `wrapRequest` для захвата входящих HTTP-заголовков, затем `transformFunctionInput` для их предоставления в контексте функции.

```ts
import { Middleware } from "inngest";

class HeadersMiddleware extends Middleware.BaseMiddleware {
  private headers: Record<string, string> = {};

  async wrapRequest({ next, requestInfo }: Middleware.WrapRequestArgs) {
    this.headers = { ...requestInfo.headers };
    return await next();
  }

  transformFunctionInput(args: Middleware.TransformFunctionInputArgs) {
    return {
      ...args,
      ctx: {
        ...args.ctx,
        headers: this.headers,
      },
    };
  }
}
```

## Наблюдаемость

Логирование событий жизненного цикла функций и шагов для мониторинга и метрик.

```ts
import { Middleware } from "inngest";

class ObservabilityMiddleware extends Middleware.BaseMiddleware {
  onRunStart({ functionInfo }: Middleware.OnRunStartArgs) {
    console.log(`[run:start] ${functionInfo.id}`);
  }

  onRunComplete({ functionInfo, output }: Middleware.OnRunCompleteArgs) {
    console.log(`[run:complete] ${functionInfo.id}`, output);
  }

  onRunError({ functionInfo, error, isFinalAttempt }: Middleware.OnRunErrorArgs) {
    console.error(`[run:error] ${functionInfo.id}`, error);
    if (isFinalAttempt) {
      // Отправить в сервис отслеживания ошибок
    }
  }

  onStepComplete({ functionInfo, stepInfo }: Middleware.OnStepCompleteArgs) {
    console.log(`[step:complete] ${functionInfo.id} > ${stepInfo.hashedId}`);
  }

  onStepError({ functionInfo, stepInfo, error }: Middleware.OnStepErrorArgs) {
    console.error(`[step:error] ${functionInfo.id} > ${stepInfo.hashedId}`, error);
  }
}
```

## Обработка ошибок

Используйте `wrapStepHandler` для перехвата ошибок шагов и преобразования их в `NonRetriableError`.

```ts
import { Middleware, NonRetriableError } from "inngest";

class ErrorHandlingMiddleware extends Middleware.BaseMiddleware {
  async wrapStepHandler({ next }: Middleware.WrapStepHandlerArgs) {
    try {
      return await next();
    } catch (err) {
      // Преобразовать конкретные ошибки в не-повторяемые
      if (err instanceof ValidationError) {
        throw new NonRetriableError(err.message, { cause: err });
      }
      throw err;
    }
  }
}
```

Здесь используется `wrapStepHandler` вместо `wrapStep`, потому что он выполняется при каждой неудавшейся попытке, тогда как `wrapStep` выполняется только после исчерпания всех повторных попыток шага.

## Кастомная сериализация

Inngest сериализует и десериализует данные как JSON. Это означает, что не-JSON типы, такие как `Date`, `Map` или `Set`, теряются. Однако их можно сохранить с помощью промежуточного ПО сериализатора.

Промежуточное ПО сериализатора создаёт JSON-совместимые представления не-JSON типов, обеспечивая их сохранение между границами шагов, отправкой событий и вызовами функций. Смотрите руководство по [кастомной сериализации](/docs/reference/typescript/v4/middleware/serialization) для полного описания.

## Вставка шагов

Используйте `wrapFunctionHandler` для выполнения шагов до или после обработчика функции. Поскольку оборачивающие хуки имеют доступ к `ctx`, можно вызывать `step.run()` напрямую.

```ts
import { Middleware } from "inngest";

class InsertStepsMiddleware extends Middleware.BaseMiddleware {
  async wrapFunctionHandler({ ctx, next }: Middleware.WrapFunctionHandlerArgs) {
    // Выполнить шаг перед обработчиком функции
    await ctx.step.run("setup", async () => {
      // например, инициализировать ресурсы
    });

    const functionOutput = await next();

    // Выполнить шаг после обработчика функции
    await ctx.step.run("cleanup", async () => {
      // например, отправить уведомление, обновить запись статуса
    });

    return functionOutput;
  }
}
```

Тот же паттерн работает с `wrapStep` для вставки шагов вокруг отдельных выполнений шагов.

```ts
import { Middleware } from "inngest";

class StepAuditMiddleware extends Middleware.BaseMiddleware {
  async wrapStep({ ctx, next, stepInfo }: Middleware.WrapStepArgs) {
    try {
      return await next();
    } catch (err) {
      // Записать сбой в надёжный шаг
      await ctx.step.run(`audit-failure-${stepInfo.hashedId}`, async () => {
        await auditLog.write({ step: stepInfo.hashedId, error: err });
      });

      throw err;
    }
  }
}
```

<Callout>
Шаги, вставленные `wrapStep` промежуточного ПО, не будут снова вызывать `wrapStep` того же промежуточного ПО. SDK предотвращает это во избежание бесконечных циклов.
</Callout>

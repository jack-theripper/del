# Жизненный цикл промежуточного ПО

Промежуточное ПО позволяет встраиваться в выполнение функций для добавления сквозной функциональности: логирования, обработки ошибок и внедрения зависимостей. Промежуточное ПО основано на классах: нужно расширить `Middleware.BaseMiddleware` и определить методы-хуки.

## Создание промежуточного ПО

Расширьте `Middleware.BaseMiddleware` и переопределите нужные хуки:

```ts
import { Middleware } from "inngest";

class MyMiddleware extends Middleware.BaseMiddleware {
  // Переопределить методы-хуки здесь
}
```

Зарегистрируйте промежуточное ПО на уровне клиента (применяется ко всем функциям) или на уровне функции (применяется к одной функции):

```ts
// На уровне клиента
const inngest = new Inngest({
  id: "my-app",
  middleware: [MyMiddleware],
});

// На уровне функции
inngest.createFunction({
  id: "my-fn",
  middleware: [MyMiddleware],
  triggers: { event: "app/user.created" },
}, async ({ event }) => {
  // ...
});
```

<Callout>
Для каждого запроса создаётся новый экземпляр промежуточного ПО, поэтому можно безопасно использовать свойства экземпляра (`this`) для хранения состояния запроса без риска утечек между запусками.
</Callout>

## Жизненный цикл выполнения

Ниже показан порядок вызова хуков во время запроса. Это ключевая модель для понимания промежуточного ПО:

1. **`wrapRequest()`** — Внешняя обёртка вокруг всего HTTP-запроса
2. **`transformFunctionInput()`** — Изменение `ctx` перед запуском обработчика функции
3. **`wrapFunctionHandler()`** — Обёртка выполнения функции (например, для `AsyncLocalStorage`)
4. **`onMemoizationEnd()`** — После разрешения всех мемоизированных шагов
5. **`onRunStart()`** — Только первая попытка (попытка 0, нет мемоизированных шагов)
6. Для каждого шага:
   - `transformStepInput()` → `wrapStep()` → `onStepStart()` → `wrapStepHandler()` → выполнение → `onStepComplete()` / `onStepError()`
7. **`onRunComplete()`** / **`onRunError()`** — Когда функция завершается

Отправка событий (`transformSendEvent()` → `wrapSendEvent()`) не является частью этой фиксированной последовательности. Выполняется при каждом вызове `inngest.send()` или `step.sendEvent()`.

<Callout>
Неопределённые хуки не создают накладных расходов: SDK полностью пропускает их. Переопределяйте только нужные хуки.
</Callout>

## Наблюдательные хуки

Наблюдательные хуки (`on*`) работают по принципу "вызвал и забыл". Они получают аргументы только для чтения и не возвращают значения. Используйте их для логирования, метрик и побочных эффектов. Ошибки, выброшенные в наблюдательных хуках, перехватываются и логируются, а не распространяются на запуск или шаг.

---

### `onRunStart`

Вызывается один раз при первой попытке запуска (попытка 0, нет мемоизированных шагов). Не вызывается при повторных попытках или воспроизведениях.

Не будет вызван, если первая попытка не достигла приложения (например, из-за сетевой ошибки).

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о выполняемой функции.
  </Property>
</Properties>

---

### `onRunComplete`

Вызывается при успешном завершении функции.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="output">
    Успешное возвращаемое значение функции.
  </Property>
</Properties>

---

### `onRunError`

Вызывается каждый раз, когда функция выбрасывает ошибку.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="error">
    Выброшенная ошибка.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="isFinalAttempt">
    Является ли это последней попыткой повтора перед окончательным сбоем запуска.
  </Property>
</Properties>

---

### `onStepStart`

Вызывается перед запуском обработчика шага. Вызывается только для `step.run` и `step.sendEvent`.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="stepInfo">
    Метаданные о выполняемом шаге.
  </Property>
</Properties>

---

### `onStepComplete`

Вызывается при успешном завершении шага. Вызывается только для `step.run` и `step.sendEvent`. Никогда не вызывается для мемоизированных шагов.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="output">
    Успешное возвращаемое значение шага.
  </Property>
  <Property required name="stepInfo">
    Метаданные о шаге.
  </Property>
</Properties>

---

### `onStepError`

Вызывается, когда шаг выбрасывает ошибку. Вызывается только для `step.run` и `step.sendEvent`. Никогда не вызывается для мемоизированных шагов.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="error">
    Выброшенная ошибка.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="isFinalAttempt">
    Является ли это последней попыткой повтора для данного шага.
  </Property>
  <Property required name="stepInfo">
    Метаданные о шаге.
  </Property>
</Properties>

---

### `onMemoizationEnd`

Вызывается один раз за запрос после разрешения всех мемоизированных шагов. При первом запросе (нет мемоизированных шагов) вызывается немедленно.

Сценарии использования этого хука ограничены. Он в первую очередь полезен для логирования или метрик, которые должны выполняться после разрешения всех мемоизированных шагов.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
</Properties>

## Оборачивающие хуки

Оборачивающие хуки (`wrap*`) следуют модели "луковицы": **необходимо** вызвать `next()` для продолжения обработки. Код перед `next()` выполняется на входе, код после — на выходе.

```ts
class MyMiddleware extends Middleware.BaseMiddleware {
  async wrapFunctionHandler({ next }: Middleware.WrapFunctionHandlerArgs) {
    console.log("before");
    const result = await next();
    console.log("after");
    return result;
  }
}
```

При нескольких промежуточных ПО они вкладываются: промежуточное ПО 1 оборачивает промежуточное ПО 2, которое оборачивает внутренний обработчик.

---

### `wrapRequest`

Оборачивает весь HTTP-запрос.

Примеры использования: аутентификация, метрики верхнего уровня или границы ошибок.

<Properties>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="next">
    Необходимо вызвать для продолжения обработки. Возвращает HTTP-ответ.
  </Property>
  <Property required name="requestInfo">
    Метаданные входящего HTTP-запроса (заголовки, URL, метод).
  </Property>
  <Property required name="runId">
    ID текущего запуска.
  </Property>
</Properties>

---

### `wrapFunctionHandler`

Оборачивает выполнение функции. `next()` разрешается при завершении функции.

Примеры использования: `AsyncLocalStorage`, преобразование ошибок, измерение времени или [вставка шагов](/docs/reference/typescript/v4/middleware/examples#inserting-steps).

<Callout>
`next()` разрешается только при полном завершении или ошибке функции. При обнаружении нового шага `next()` не разрешается для данного запроса. Он намеренно зависает до сборки мусора.
</Callout>

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="next">
    Необходимо вызвать для выполнения обработчика функции.
  </Property>
</Properties>

---

### `wrapStep`

Оборачивает каждый шаг, включая мемоизированные шаги и все виды шагов (`step.run`, `step.sleep`, `step.invoke` и т.д.).

Примеры использования: десериализация мемоизированных данных, [вставка шагов](/docs/reference/typescript/v4/middleware/examples#inserting-steps).

<Callout>
Если шаг не мемоизирован, `next()` не разрешается для данного запроса. Метод намеренно зависает до сборки мусора.
</Callout>

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="next">
    Необходимо вызвать для продолжения обработки шага.
  </Property>
  <Property required name="stepInfo">
    Метаданные о шаге. Проверяйте `stepInfo.memoized` для различения мемоизированных и свежих шагов.
  </Property>
</Properties>

---

### `wrapStepHandler`

Оборачивает выполнение обработчика шага. Вызывается только для `step.run` и `step.sendEvent`.

Примеры использования: сериализация вывода шага, [обработка ошибок](/docs/reference/typescript/v4/middleware/examples#error-handling) или измерение времени.

<Properties>
  <Property required name="ctx">
    Контекст функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="next">
    Необходимо вызвать для выполнения обработчика шага.
  </Property>
  <Property required name="stepInfo">
    Метаданные о шаге.
  </Property>
</Properties>

---

### `wrapSendEvent`

Оборачивает отправку событий через `inngest.send()` или `step.sendEvent()`.

Примеры использования: резервное копирование при сбое отправки, метрики.

<Properties>
  <Property required name="events">
    Отправляемые события.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции или `null` при вызове вне функции (например, `inngest.send()`).
  </Property>
  <Property required name="next">
    Необходимо вызвать для отправки событий.
  </Property>
</Properties>

## Хуки преобразования

Хуки преобразования (`transform*`) получают аргументы и возвращают изменённую копию. Используйте их для внедрения зависимостей, изменения входных данных или обогащения событий.

---

### `transformFunctionInput`

Изменение контекста функции перед запуском обработчика.

Примеры использования: внедрение зависимостей, десериализация данных события.

<Properties>
  <Property required name="ctx">
    Контекст функции. Добавляйте свойства сюда для их внедрения в обработчик функции.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="steps">
    Запись мемоизированных данных шагов с ключами в виде хэшированных ID шагов.
  </Property>
</Properties>

---

### `transformStepInput`

Изменение параметров или входных данных шага перед его запуском.

Примеры использования: сериализация данных `step.invoke`, сброс кэша мемоизации (изменение ID шага).

<Properties>
  <Property required name="functionInfo">
    Метаданные о функции.
  </Property>
  <Property required name="stepInfo">
    Частичные метаданные шага.
  </Property>
  <Property required name="stepOptions">
    Параметры, переданные шагу (первый аргумент).
  </Property>
  <Property required name="input">
    Аргументы, переданные функции шага (после ID и обработчика).
  </Property>
</Properties>

---

### `transformSendEvent`

Изменение событий перед их отправкой.

Примеры использования: сериализация данных событий, добавление метаданных.

<Properties>
  <Property required name="events">
    Отправляемые события. Верните изменённую копию для их модификации.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции или `null` при вызове вне функции.
  </Property>
</Properties>

## Преобразования типов вывода

Преобразования типов вывода — это свойства `declare`, контролирующие способ типизации TypeScript возвращаемых значений функций и шагов. Они не имеют поведения во время выполнения — влияют только на систему типов.

По умолчанию SDK предполагает, что все возвращаемые значения сериализованы в JSON, поэтому типы вроде `Date` становятся `string`. Если промежуточное ПО изменяет поведение сериализации во время выполнения (например, через [`wrapStepHandler`](#wrapstephandler)), объявите соответствующее преобразование типов, чтобы TypeScript отражал фактические типы времени выполнения.

<Callout>
Преобразования типов влияют только на типы. Вы несёте ответственность за соответствие объявленного преобразования вашему фактическому преобразованию во время выполнения.
</Callout>

Оба преобразования используют паттерн `Middleware.StaticTransform` для имитации типов высшего порядка. Например, обычный обобщённый тип, сохраняющий `Date` вместо его JSON-ификации:

```ts
// Обычный обобщённый тип
type PreserveDate<In> = In extends Date ? Date : Jsonify<In>;
```

Как `StaticTransform` это становится интерфейсом, где `this["In"]` заменяет обобщённый параметр, а `Out` — результат:

```ts
import { Middleware } from "inngest";

// Та же логика в виде StaticTransform
interface PreserveDate extends Middleware.StaticTransform {
  Out: this["In"] extends Date ? Date : Jsonify<this["In"]>;
}
```

<Properties>
  <Property required name="In">
    Исходный возвращаемый тип. Устанавливается автоматически SDK — не устанавливайте это самостоятельно.
  </Property>
  <Property required name="Out">
    Преобразованный тип. Определите это для вычисления выходного типа на основе `In`.
  </Property>
</Properties>

При нескольких промежуточных ПО с объявленными преобразованиями они соединяются в цепочку в порядке регистрации: `Out` одного становится `In` следующего.

---

### `functionOutputTransform`

Объявляет, как преобразуются возвращаемые типы функций. По умолчанию типы возврата JSON-ифицируются (например, `Date` становится `string`).

```ts
import { Middleware } from "inngest";

interface PreserveDate extends Middleware.StaticTransform {
  Out: this["In"] extends Date ? Date : Jsonify<this["In"]>;
}

class MyMiddleware extends Middleware.BaseMiddleware {
  declare functionOutputTransform: PreserveDate;
}
```

---

### `stepOutputTransform`

Объявляет, как преобразуются типы вывода шагов. По умолчанию типы вывода JSON-ифицируются (например, `Date` становится `string`).

```ts
import { Middleware } from "inngest";

interface PreserveDate extends Middleware.StaticTransform {
  Out: this["In"] extends Date ? Date : Jsonify<this["In"]>;
}

class MyMiddleware extends Middleware.BaseMiddleware {
  declare stepOutputTransform: PreserveDate;
}
```

---

## Статические хуки

### `onRegister`

Вызывается один раз при регистрации класса промежуточного ПО в клиенте или функции.

Примеры использования: одноразовая настройка.

```ts
class MyMiddleware extends Middleware.BaseMiddleware {
  static onRegister({ client, functionInfo }: Middleware.OnRegisterArgs) {
    // Одноразовая настройка
  }
}
```

<Properties>
  <Property required name="client">
    Экземпляр клиента Inngest.
  </Property>
  <Property required name="functionInfo">
    Метаданные о функции или `null` для промежуточного ПО на уровне клиента.
  </Property>
</Properties>

## Важные замечания

- **Производительность** — Неопределённые хуки не создают накладных расходов. SDK проверяет наличие хука и полностью пропускает его если не определён.
- **Иммутабельность** — Аргументы наблюдательных и оборачивающих хуков строго доступны только для чтения. Не мутируйте их.
- **`next()` обязателен** — Оборачивающие хуки должны вызывать `next()`, иначе запрос зависнет.
- **Экземпляр на запрос** — Для каждого запроса создаётся новый экземпляр промежуточного ПО, поэтому состояние экземпляра безопасно использовать в рамках одного запроса.

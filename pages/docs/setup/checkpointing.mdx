# Чекпоинтинг

Чекпоинтинг — это оптимизация производительности для функций Inngest, которая выполняет шаги сразу (eager execution) вместо ожидания внутренней оркестрации. Результат — значительно меньшая задержка, что идеально подходит для AI-рабочих процессов реального времени.

В v4 TypeScript SDK чекпоинтинг **включён по умолчанию**. Настройка нужна только если вы хотите изменить параметры, например `maxRuntime`, или отключить его.

## Минимальные требования

### Язык

- **TypeScript**: SDK `3.51.0` или выше (включён по умолчанию в v4).
- **Go**: SDK версии `v0.15.0`.

### Максимальное время выполнения для бессерверных сред

Если ваше приложение развёрнуто на бессерверных платформах, таких как [Vercel](/docs/deploy/vercel), следует настроить параметр `maxRuntime` чуть ниже максимальной длительности вашей функции. Подробнее в разделе [конфигурация](#configuration).

```ts
const inngest = new Inngest({
  id: 'my-app',
  checkpointing: {
    maxRuntime: '50s', // 50с — хороший вариант, если максимальная длительность 60с
  }
})
```

<Tip>
  Совет: Многие платформы, например Vercel, позволяют настраивать максимальную длительность для каждой функции отдельно, например для эндпоинта `/api/inngest`. Подробнее о настройке на Vercel [здесь](https://vercel.com/docs/functions/configuring-functions/duration).
</Tip>

### Отключение чекпоинтинга

Чтобы отключить чекпоинтинг для всех функций, установите `checkpointing: false` на клиенте:

```ts
import { Inngest } from "inngest";

export const inngest = new Inngest({
  id: "my-app",
  checkpointing: false,
});
```

Также можно отключить его для конкретной функции:

```ts
export const myFunction = inngest.createFunction(
  {
    id: "my-function",
    checkpointing: false,
  },
  async ({ step }) => {
    // шаги здесь будут использовать стандартную оркестрацию
  }
);
```

### Конфигурация

Настройте поведение чекпоинтинга с помощью следующих параметров:

* `maxRuntime` — по умолчанию: `0` (без ограничений): Максимальное время непрерывного выполнения функции и записи чекпоинтов шагов перед возвратом ответа на запрос. Установите значение чуть меньше максимально допустимого тайм-аута запроса для вашей платформы или сервера. Например, если платформа допускает `900с`, можно задать `maxRuntime` равным `800s`.
* `bufferedSteps` — по умолчанию: `1` (без буферизации): Количество шагов, накапливаемых перед записью чекпоинта. Позволяет сократить количество запросов к Inngest при выполнении множества шагов подряд. Учтите, что буферизованные шаги, для которых не был записан чекпоинт, могут быть потеряны при некорректном завершении сервера.
* `maxInterval`: Максимальный интервал ожидания перед записью чекпоинта, даже если количество `bufferedSteps` не достигнуто.

```ts {{ title: "Пример конфигурации" }}
checkpointing: {
  maxRuntime: '300s',
  bufferedSteps: 2,
  maxInterval: "10s",
}
```

Для включения чекпоинтинга:

1. Установите пакет checkpoint:

```shell
go get github.com/inngest/inngestgo/pkg/checkpoint
```

2. Задайте `Checkpoint` в параметрах функции:

```go
import (
  "github.com/inngest/inngestgo"
  "github.com/inngest/inngestgo/pkg/checkpoint"
)

_, err := inngestgo.CreateFunction(
  client,
  inngestgo.FunctionOpts{
    ID:         "my-function",
    Name:       "My Function",
    Checkpoint: checkpoint.ConfigSafe,
  },
  // ... триггеры и обработчик
)
```

## Как это работает?

[Стандартная модель выполнения Inngest](/docs/learn/how-functions-are-executed) предполагает полную передачу управления платформе Inngest: при завершении каждого шага выполняется HTTP-запрос для сохранения состояния выполнения, что создаёт задержку между шагами.

![С чекпоинтингом и без него](/assets/docs/checkpointing/checkpointing_with_without.png)

При чекпоинтинге SDK оркестрирует шаги на стороне клиента (_на вашем сервере_) и выполняет их немедленно. По мере завершения шагов в Inngest отправляются сообщения-чекпоинты для отслеживания прогресса. Результат — значительно меньшая задержка, что идеально подходит для AI-рабочих процессов реального времени.

![Выполнение рабочего процесса Inngest](/assets/docs/checkpointing/checkpointing_inngest_workflow.jpg)

### Сбои и повторные попытки

Что происходит при ошибке? Если шаг завершается сбоем и требует повторной попытки, движок выполнения переключается на стандартную оркестрацию для корректной обработки. Вы получаете скорость при успешной работе и надёжность при сбоях.

## Примечания и ограничения

- **Параллельное выполнение шагов** — Когда функция разветвляется на параллельные шаги, выполнение переключается на стандартную оркестрацию до конца запуска. Чекпоинтинг не возобновляется после параллельного выполнения.

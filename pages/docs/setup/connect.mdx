# Connect

<Info>
  Connect в настоящее время находится в публичной бета-версии. Подробнее о [фазе публичной беты здесь](/docs/release-phases#public-beta).
</Info>

API `connect` позволяет вашему приложению создавать исходящее постоянное соединение с Inngest. Каждое приложение может устанавливать несколько соединений с Inngest, что позволяет горизонтально масштабироваться на нескольких воркерах. Ключевые преимущества использования `connect` по сравнению с [`serve`](/docs/learn/serving-inngest-functions):

- **Минимальная задержка** — Постоянные соединения обеспечивают наименьшую задержку между вашим приложением и Inngest.
- **Эластичное горизонтальное масштабирование** — Легко добавляйте мощности, запуская дополнительные воркеры.
- **Идеально для контейнерных сред** — Развёртывание в Kubernetes или ECS без необходимости в балансировщике нагрузки для входящего трафика.
- **Упрощённые долгоработающие шаги** — Выполнение шагов не ограничено тайм-аутами HTTP платформы.

Количество доступных одновременных соединений воркеров зависит от вашего тарифного плана. Для новых планов ограничения следующие:

* Бесплатный план: 3 одновременных соединения воркеров
* Все платные планы: 20 одновременных соединений воркеров
* Максимум приложений на соединение: 10

Ограничения соединений вашего плана можно найти на [странице биллинга](https://app.inngest.com/billing).

## Минимальные требования

### Язык
- **TypeScript**: SDK `3.34.1` или выше.
- **Go**: SDK `0.11.2` или выше.
- **Python**: SDK `0.5.0` или выше.
  - Установите SDK командой `pip install inngest[connect]`, так как требуются дополнительные зависимости.
  - Также рекомендуются следующие версии:
    - `protobuf>=5.29.4,<6.0.0`
    - `psutil>=6.0.0,<7.0.0`
    - `websockets>=15.0.0,<16.0.0`

### Среда выполнения

Необходимо использовать долгоработающий сервер (Render, Fly.io, Kubernetes и т.д.). Бессерверные среды выполнения (AWS Lambda, Vercel и т.д.) не поддерживаются.

При использовании TypeScript среда выполнения должна поддерживать встроенный WebSocket (Node `22.4.0` или выше, Deno `1.4` или выше, Bun `1.1` или выше).



## Начало работы

Использовать `connect` с вашим приложением очень просто. Метод «connect» каждого SDK требует только список функций, доступных для выполнения. (Примечание: поддержка Python находится в бета-версии; [проголосуйте за неё в нашем роадмапе](https://roadmap.inngest.com/roadmap?id=2bac8d74-288f-47c7-8afc-3fd1a0e94654))

Ниже приведён пример полностью рабочего приложения в одном файле, подключающегося к Inngest.

<CodeGroup>
```ts
import { Inngest } from "inngest";
import { connect } from "inngest/connect";

const inngest = new Inngest({
  id: "my-app",
});

const handleSignupFunction = inngest.createFunction(
  { id: "handle-signup", triggers: [{ event: "user.created" }] },
  async ({ event, step }) => {
    console.log("Function called", event);
  },
);

(async () => {
  const connection = await connect({
    apps: [{ client: inngest, functions: [handleSignupFunction] }],
  });

  console.log("Worker: connected", connection);
})();
```
```go
type UserCreatedEvent struct {
	Name string
	Data struct {
		UserID string `json:"user_id"`
	}
}

func main() {
	ctx := context.Background()

	client, err := inngestgo.NewClient(inngestgo.ClientOpts{
		AppID:      "my-app",
		Logger:     logger.StdlibLogger(ctx),
		AppVersion: nil, // Необязательно, по умолчанию — git commit SHA
	})
	if err != nil {
		panic(err)
	}

	_, err = inngestgo.CreateFunction(
		client,
		inngestgo.FunctionOpts{ID: "handle-signup", Name: "Handle signup"},
		inngestgo.EventTrigger("user.created", nil),
		func(ctx context.Context, input inngestgo.Input[UserCreatedEvent]) (any, error) {
			fmt.Println("Function called")
			return map[string]any{"success": true}, nil
		},
	)
	if err != nil {
		panic(err)
	}

	fmt.Println("Worker: connecting")

	conn, err := inngestgo.Connect(ctx, inngestgo.ConnectOpts{
		InstanceID: inngestgo.Ptr("example-worker"),
		Apps:       []inngestgo.Client{client},
	})
	if err != nil {
		fmt.Printf("ERROR: %#v\n", err)
		os.Exit(1)
	}

  defer func(conn connect.WorkerConnection) {
		<-ctx.Done()
		err := conn.Close()
		if err != nil {
			fmt.Printf("could not close connection: %s\n", err)
		}
	}(conn)
}
```
```python

import inngest

client = inngest.Inngest(app_id="my-app")

@client.create_function(

functions = [fn_1]

asyncio.run(
    connect(
        apps=[(client, functions)],
    ).start()
```

## Как это работает?

API `connect` устанавливает постоянное WebSocket-соединение с Inngest. Каждое соединение может одновременно обрабатывать выполнение нескольких функций и шагов. Каждое приложение может создавать несколько соединений с Inngest, обеспечивая горизонтальное масштабирование. Кроме того, connect поддерживает следующие возможности:

- **Автоматическое переподключение** — Соединение автоматически восстанавливается при разрыве.
- **Корректное завершение** — Соединение корректно закрывается при получении приложением сигнала завершения (`SIGTERM`). После закрытия соединения новые шаги не принимаются, а текущие шаги завершаются.
- **Максимальный параллелизм на уровне воркера (скоро)** — Каждый воркер может настроить максимальное количество одновременно обрабатываемых шагов. Это позволяет Inngest распределять нагрузку между воркерами и не перегружать один воркер.

## Локальная разработка

При локальной разработке установите переменную окружения `INNGEST_DEV=1` для включения режима локальной разработки. Это заставит SDK подключаться к [dev-серверу Inngest](/docs/dev-server). При запуске процесса воркера он автоматически подключится к dev-серверу и синхронизирует конфигурации ваших функций.

В режиме локальной разработки ключи подписи и события не требуются.

## Развёртывание в production

<Steps>
  <Step title="Установите ключи подписи и события">
    Для безопасного подключения приложения к Inngest необходимо задать переменные окружения `INNGEST_SIGNING_KEY` и `INNGEST_EVENT_KEY`.

    Эти ключи можно найти в панели управления Inngest. Подробнее о [ключах событий](/docs/events/creating-an-event-key) и [ключах подписи](/docs/platform/signing-keys).
  </Step>
  <Step title="Установите версию приложения">
    `appVersion` используется для идентификации версии вашего приложения, подключённой к Inngest. Это позволяет Inngest поддерживать скользящие развёртывания, при которых к Inngest могут быть подключены несколько версий приложения.

    При подключении новой версии приложения конфигурации функций синхронизируются с Inngest. После подключения новой версии Inngest обновляет конфигурацию функций в вашей среде и начинает направлять новые запуски функций на последнюю версию.

    Вы можете установить `appVersion` в любое значение, но рекомендуется использовать что-то, что автоматически изменяется при каждом развёртывании, например git commit sha или тег Docker-образа.

    <CodeGroup>
    ```ts {{ title: "Любая платформа" }}
    // Можно установить версию приложения в любую переменную окружения,
    // например номер сборки ('v2025.02.12.01'), git commit sha ('f5a40ff')
    // или произвольное значение ('my-app-v1').
    const inngest = new Inngest({
      id: 'my-app',
      appVersion: process.env.MY_APP_VERSION, // Используйте любую переменную окружения
    })
    ```
    ```ts {{ title: "GitHub Actions" }}
    // При использовании Github Actions для сборки приложения можно установить
    // версию приложения в переменную окружения GITHUB_SHA во время сборки
    // или передать её в Docker-образ.
    const inngest = new Inngest({
      id: 'my-app',
      appVersion: process.env.GITHUB_SHA,
    })
    ```
    ```ts {{ title: "Render" }}
    // Render включает переменную окружения RENDER_GIT_COMMIT во время сборки и выполнения.
    // https://render.com/docs/environment-variables
    const inngest = new Inngest({
      id: 'my-app',
      appVersion: process.env.RENDER_GIT_COMMIT,
    })
    ```
    ```ts {{ title: "Fly.io" }}
    // Fly включает переменную окружения версии машины во время выполнения.
    // https://fly.io/docs/machines/runtime-environment/
    const inngest = new Inngest({
      id: 'my-app',
      appVersion: process.env.FLY_MACHINE_VERSION,
    })
    ```
    </CodeGroup>
  </Step>
  <Step title="Установите ID экземпляра (рекомендуется)">
    `instanceId` используется для идентификации экземпляра воркера вашего приложения, подключённого к Inngest. Это позволяет Inngest поддерживать несколько экземпляров (воркеров) приложения, подключённых к Inngest.

    По умолчанию Inngest пытается использовать имя хоста воркера в качестве ID экземпляра. При запуске приложения в контейнерной среде можно задать `instanceId` равным ID контейнера.

    <CodeGroup>
    ```ts {{ title: "Любая платформа" }}
    // Установите ID экземпляра в любую переменную окружения, уникальную для воркера
    await connect({
      apps: [...],
      instanceId: process.env.MY_CONTAINER_ID,
    })
    ```
    ```ts {{ title: "Kubernetes + Docker" }}
    // instanceId по умолчанию использует переменную окружения HOSTNAME.
    // Kubernetes и Docker по умолчанию устанавливают HOSTNAME равным имени пода,
    // поэтому значение задаётся автоматически.
    await connect({
      apps: [...],
      // Это происходит под капотом, если instanceId не задан
      // instanceId: process.env.HOSTNAME,
    })
    ```
    ```ts {{ title: "Render" }}
    // Render включает переменную окружения RENDER_INSTANCE_ID во время выполнения.
    // https://render.com/docs/environment-variables
    await connect({
      apps: [...],
      instanceId: process.env.RENDER_INSTANCE_ID,
    })
    ```
    ```ts {{ title: "Fly.io" }}
    // Fly включает переменную окружения FLY_MACHINE_ID во время выполнения.
    // https://fly.io/docs/machines/runtime-environment/
    await connect({
      apps: [...],
      instanceId: process.env.FLY_MACHINE_ID,
    })
    ```
    </CodeGroup>
  </Step>
  <Step title="Установите максимальный параллелизм (рекомендуется)">
    Параметр `maxWorkerConcurrency` используется для ограничения количества шагов, одновременно выполняемых экземпляром воркера.
    Это позволяет контролировать использование ресурсов и предотвращать перегрузку одного воркера слишком большим числом одновременных запросов.

    <CodeGroup>
    ```ts
    await connect({
      apps: [...],
      maxWorkerConcurrency: 10,
    })
    ```

## Жизненный цикл

Поскольку connect-воркер является долгоработающим процессом, важно понимать его жизненный цикл и его связь с развёртыванием новых версий приложения. Ниже представлен обзор жизненного цикла connect-воркера и точки, в которых можно перехватить управление для корректного завершения работы и других событий.

<Steps>
  <Step>
    `CONNECTING` — Воркер устанавливает соединение с Inngest. Начинается при вызове `connect()`.

    Сначала воркер отправляет HTTP-запрос в Inngest API для получения информации о соединении. Ответ содержит URL WebSocket-шлюза. Затем воркер подключается к WebSocket-шлюзу.
  </Step>
  <Step>
    `ACTIVE` — Воркер подключён к Inngest и готов выполнять функции.

    * Синхронизируется новая `appVersion`, включая актуальные конфигурации функций.
    * Воркер начинает отправлять и получать «heartbeat»-сообщения Inngest для проверки активности соединения.
    * Воркер автоматически переподключается при потере соединения.

    ```ts {{ title: "TypeScript" }}
    // Promise от connect разрешится, когда соединение станет ACTIVE
    const connection = await connect({
      apps: [...],
    })
    console.log(`The worker connection is: ${connection.state}`)
    // The worker connection is: ACTIVE
    ```
  </Step>
  <Step>
    `RECONNECTING` — Воркер переподключается к Inngest после потери соединения.

    При потере WebSocket-соединения воркер автоматически отправит все выполняющиеся шаги через HTTP API.

    По умолчанию воркер будет пытаться переподключиться к Inngest бесконечное количество раз.
  </Step>
  <Step>
    `CLOSING` — Воркер начинает процесс завершения работы.

    * После перехода в это состояние новые шаги не принимаются.
    * Текущие шаги завершаются. После закрытия WebSocket-соединения воркер отправит все выполняющиеся шаги через HTTP API.

    По умолчанию SDK прослушивает сигналы `SIGTERM` и `SIGINT` и начинает процесс завершения. Это поведение можно настроить в каждом SDK:

    <CodeGroup>
    ```ts
    // Можно явно указать, какие сигналы SDK должен прослушивать,
    // передав массив сигналов в handleShutdownSignals:
    const connection = await connect({
      apps: [...],
      // Например, слушать только SIGTERM или передать пустой массив, чтобы не слушать сигналы
      handleShutdownSignals: ['SIGTERM'],
    })
    ```
    ```go
    // Go SDK должен получить объект Context, который будет уведомлён
    // при получении нужных сигналов. Используйте signal.NotifyContext:
    ctx, cancel := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
    defer cancel()

    // Далее в функции — передайте контекст в connect:
    ws, err := inngestgo.Connect(ctx, inngestgo.ConnectOpts{
        InstanceID: inngestgo.Ptr("example-worker"),
        Apps:       []inngestgo.Client{client},
    })
    ```
    </CodeGroup>

    Соединение можно закрыть вручную с помощью метода `close` объекта соединения:

    ```ts
    await connection.close()
    // Соединение закрыто
    ```
  </Step>
  <Step>
    `CLOSED` — WebSocket-соединение воркера закрыто.

    К этому моменту все выполняющиеся шаги будут отправлены через HTTP API по мере закрытия WebSocket-соединения, что гарантирует отсутствие потерь выполняющихся шагов.

    ```ts {{ title: "TypeScript" }}
    // Promise `closed` разрешится, когда соединение перейдёт в состояние "CLOSED"
    await connection.closed
    // Соединение закрыто
    ```
  </Step>
</Steps>

<Info>
  **WebSocket-соединение и HTTP-резерв** — Пока WebSocket-соединение открыто, воркер получает и отправляет все результаты шагов через него. При закрытии соединения воркер переключается на HTTP API для отправки оставшихся результатов шагов.
</Info>

[//]: <> (TODO: Создать диаграмму для объяснения синхронизации)

Inngest отслеживает версию, на которой работают ваши воркеры. Это внутреннее представление изменяется при обновлении конфигурации функции, предоставлении нового идентификатора версии приложения в конфигурации клиента или изменении версии SDK либо языка.

При развёртывании новой версии приложения первый воркер, подключившийся к Inngest, автоматически синхронизирует приложение. При этом конфигурации функций обновляются до желаемого состояния, заданного в коде.

`connect` поддерживает скользящие релизы: во время развёртывания приложения Inngest будет запускать функции на всех подключённых воркерах независимо от версии, если они способны обработать запрос для данной функции. Это предотвращает концентрацию трафика на одном экземпляре во время развёртывания и проблему «стада громов».

После завершения работы всех старых воркеров после развёртывания можно откатиться к старой версии, вернув старый воркер. Аналогично процессу развёртывания, это обновит конфигурацию функций до предыдущего состояния и позволит постепенно перенаправлять трафик на старую версию, добавляя старые воркеры и останавливая воркеры с новой версией.

## Проверки работоспособности

При запуске приложения в контейнерной среде рекомендуется использовать проверку работоспособности, чтобы убедиться, что приложение запущено и готово принимать соединения. Это ключевой элемент для корректного скользящего развёртывания новых версий. При использовании Kubernetes рекомендуется применять `readinessProbe` для проверки готовности приложения принимать соединения.

Простейший способ реализовать проверку работоспособности — создать HTTP-эндпоинт, прослушивающий запросы проверки. Поскольку connect является исходящим WebSocket-соединением, необходимо создать небольшой HTTP-сервер, который прослушивает запросы проверки и возвращает код 200, когда соединение с Inngest активно.

Ниже приведён пример использования `connect` с базовым HTTP-сервером Node.js для обработки запросов проверки работоспособности и возврата кода 200 при активном соединении с Inngest.

<CodeGroup>
```ts {{ title: "Node.js" }}
import { createServer } from "http";
import { connect, ConnectionState } from "inngest/connect";
import { inngest, functions } from "./src/inngest";

(async () => {
  const connection = await connect({
    apps: [{ client: inngest, functions }],
  });

  console.log("Worker: connected", connection);

  // Базовый веб-сервер, прослушивающий только эндпоинт /ready
  // и возвращающий 200, когда соединение с Inngest активно.
  const httpServer = createServer((req, res) => {
    if (req.url === "/ready") {
      if (connection.state === ConnectionState.ACTIVE) {
        res.writeHead(200, { "Content-Type": "text/plain" });
        res.end("OK");
      } else {
        res.writeHead(500, { "Content-Type": "text/plain" });
        res.end("NOT OK");
      }
      return;
    }
    res.writeHead(404, { "Content-Type": "text/plain" });
    res.end("NOT FOUND");
  });

  // Запускаем сервер на выбранном порту
  httpServer.listen(8080, () => {
    console.log("Worker: HTTP server listening on port 8080");
  });

  // Когда соединение Inngest корректно закроется,
  // этот промис разрешится и приложение завершится.
  await connection.closed;
  console.log("Worker: Shut down");

  // Останавливаем HTTP-сервер
  httpServer.close();
})();
```
```ts  {{ title: "Bun (JavaScript)" }}
import { connect, ConnectionState } from "inngest/connect";
import { inngest, functions } from "./src/inngest";

const connection = await connect({
  apps: [{ client: inngest, functions }],
});

console.log("Worker: connected", connection);

// Запускаем базовый веб-сервер, прослушивающий только эндпоинт /ready
// и возвращающий 200, когда соединение с Inngest активно.
const server = Bun.serve({
  port: 8080,
  routes: {
    "/ready": async () => {
      return connection.state === ConnectionState.ACTIVE
        ? new Response("OK")
        : new Response("Not Ready", { status: 500 });
    },
  },
  fetch(req) {
    return new Response("Not Found", { status: 404 });
  },
});

console.log("Worker: HTTP server listening on port 8080");

// Когда соединение Inngest корректно закроется,
// этот промис разрешится и приложение завершится.
await connection.closed;
console.log("Worker: Shut down");

// Останавливаем HTTP-сервер
await server.stop();
```

## Параллелизм воркеров

{/* TODO: Указать минимально поддерживаемую версию для этой функции */}

Параллелизм воркеров поддерживается в следующих версиях SDK:
- **TypeScript**: SDK `3.45.1` или выше.
- **Go**: SDK `0.14.3` или выше.
- **Python**: SDK `0.5.12` или выше.

Каждый воркер может настроить максимальное количество одновременно выполняемых шагов с помощью параметра `maxWorkerConcurrency`.
Это позволяет контролировать использование ресурсов и предотвращать перегрузку одного воркера слишком большим числом одновременных запросов.

По умолчанию ограничения на одновременное выполнение шагов нет. Воркер будет принимать столько шагов, сколько отправит Inngest.

При установке `maxWorkerConcurrency` Inngest будет распределять нагрузку между воркерами в зависимости от их доступной ёмкости.
Воркеры с доступной ёмкостью будут получать больше работы, тогда как воркеры, достигшие лимита, не будут получать дополнительные шаги до освобождения ёмкости.

<Info>
  Когда воркер достигает лимита `maxWorkerConcurrency`, в панели управления Inngest может появиться ошибка **«All workers are at capacity»** («Все воркеры заняты»). Это ожидаемое поведение — воркер отклоняет шаг, а Inngest автоматически повторно ставит его в очередь, повторяя попытки до появления воркера со свободной ёмкостью. Эти ошибки ёмкости **не** учитываются в настройках [повторных попыток](/docs/functions/retries) вашей функции.
</Info>

Inngest использует `instanceId` для отслеживания `maxWorkerConcurrency` каждого воркера.
Это означает, что при наличии нескольких воркеров с одинаковым `instanceId` они будут делить одинаковый лимит `maxWorkerConcurrency`.

Inngest учитывает `maxWorkerConcurrency` из последнего запроса на соединение.
Это значит, что при обновлении `maxWorkerConcurrency` после подключения воркера и создании нового соединения лимит воркера будет обновлён до нового значения.

**Преимущества:**
- Предотвращение исчерпания ресурсов воркера (CPU, память, соединения)
- Предсказуемое распределение ресурсов по воркерам
- Помощь в горизонтальном масштабировании как однородных, так и разнородных воркеров

{/* TODO: Раскомментировать при появлении интеллектуального балансирования нагрузки
- Позволяет Inngest интеллектуально распределять нагрузку по парку воркеров
- Помогает с горизонтальным масштабированием за счёт равномерного распределения
*/}

<CodeGroup>
```ts
await connect({
  apps: [{ client: inngest, functions }],
  instanceId: "example-worker",
  maxWorkerConcurrency: 10, // Макс. 10 одновременных шагов на этом воркере
})
```
```go
conn, err := inngestgo.Connect(ctx, inngestgo.ConnectOpts{
  InstanceID:           inngestgo.Ptr("example-worker"),
  MaxWorkerConcurrency: inngestgo.Ptr(int64(10)), // Макс. 10 одновременных шагов
  Apps:                 []inngestgo.Client{client},
})
```
```python
asyncio.run(
    connect(
        apps=[(client, functions)],
        instance_id="example-worker",
        max_worker_concurrency=10,  # Макс. 10 одновременных шагов
    ).start()
)
```
</CodeGroup>

**Переменная окружения:**

Максимальный параллелизм воркера также можно задать через переменную окружения `INNGEST_CONNECT_MAX_WORKER_CONCURRENCY`.
Это удобно для настройки параллелизма без изменения кода.

```bash
INNGEST_CONNECT_MAX_WORKER_CONCURRENCY=100
```

Если заданы и параметр, и переменная окружения, параметр имеет приоритет.

**Поведение по умолчанию:**

Если `maxWorkerConcurrency` не задан (или задан равным `0`), ограничения на одновременное выполнение шагов нет.
Inngest будет отправлять столько шагов, сколько допускает лимит параллелизма вашего аккаунта.

### Параллелизм на уровне соединения

Для каждого соединения одного воркера можно задать разное значение `maxWorkerConcurrency`, указав уникальный `instanceId` для каждого соединения.
Это позволит иметь разные лимиты `maxWorkerConcurrency` для разных соединений.

Из-за разных значений `instanceId` Inngest будет считать каждое соединение отдельным воркером и не будет делить между ними одинаковый лимит `maxWorkerConcurrency`.

<CodeGroup>
```ts
// Соединение 1 с отдельным лимитом параллелизма
await connect({
  apps: [{ client: inngest, functions }],
  instanceId: "worker-1-conn-1",
  maxWorkerConcurrency: 50,
})

// Соединение 2 с отдельным лимитом параллелизма
await connect({
  apps: [{ client: inngest, functions }],
  instanceId: "worker-1-conn-2",
  maxWorkerConcurrency: 20,
})
```
```go
// Соединение 1 с отдельным лимитом параллелизма
conn1, err := inngestgo.Connect(ctx, inngestgo.ConnectOpts{
  InstanceID:           inngestgo.Ptr("worker-1-conn-1"),
  MaxWorkerConcurrency: inngestgo.Ptr(int64(50)),
  Apps:                 []inngestgo.Client{client},
})

// Соединение 2 с отдельным лимитом параллелизма
conn2, err := inngestgo.Connect(ctx, inngestgo.ConnectOpts{
  InstanceID:           inngestgo.Ptr("worker-1-conn-2"),
  MaxWorkerConcurrency: inngestgo.Ptr(int64(20)),
  Apps:                 []inngestgo.Client{client},
})
```
```python
# Соединение 1 с отдельным лимитом параллелизма
asyncio.run(
    connect(
        apps=[(client, functions)],
        instance_id="worker-1-conn-1",
        max_worker_concurrency=50,
    ).start()
)

# Соединение 2 с отдельным лимитом параллелизма
asyncio.run(
    connect(
        apps=[(client, functions)],
        instance_id="worker-1-conn-2",
        max_worker_concurrency=20,
    ).start()
)
```
</CodeGroup>

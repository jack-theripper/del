## Как запускать cron-задачи только в продакшене?

Существует несколько способов добиться этого:
1. Условный рендеринг в зависимости от окружения.
```javascript
process.env.NODE_ENV === "production" ? { cron: "* * *" } : { event: "dev/manualXYZ" }
```
💡 Если в других окружениях вместо cron вы укажете событие, вы все равно сможете запускать свои функции вручную при необходимости.

2. [Отключите ветки окружения](/docs/platform/environments#disabling-branch-environments-in-vercel).

## Как остановить выполнение функций?
Лучший способ гарантировать, что устаревшая функция не будет запущена — развернуть приложение, не включая ее в ваш [обработчик serve](/docs/reference/serve). Вы можете временно добиться того же результата, заархивировав функцию в нашей панели управления, но учтите, что новое развертывание разархивирует функцию.

## Что такое шаг «Finalization» в моем трейсе?

Шаг «finalization» (финализация) в трейсе запуска представляет собой выполнение кода между последним шагом вашей функции и концом обработчика функции.

```ts {{ x: "10"}}
const myFunction = inngest.createFunction(
  {
    id: "handle-import",
    triggers: { event: "integration.connected" }
  },
  async ({ event, step }) => {
    const res = await step.run("import-data", async () => {
      // ...
    });
    // -- Начало финализации ⬇️ --
    const newRows = res.rows.filter((row) => row.created === true)
    return { message: `Imported ${newRows.length} rows` }
    // -- Конец финализации ⬆️ --
  },
)
```

## Почему я получаю ошибки «Event key not found» в ветках окружения?
Ветки окружения [автоматически архивируются](/docs/platform/environments#archiving-branch-environments) через 3 дня после последнего развертывания.
Вы можете отключить функцию автоматического архивирования для каждого активного окружения в нашей [панели управления](https://app.inngest.com/env).


## Как указать несколько путей serve для одного приложения Vercel в панели управления?
Вы можете передать несколько путей, добавив информацию о них для каждого проекта Vercel в [настройках интеграции Vercel](https://app.inngest.com/settings/integrations/vercel).

## Какой рекомендуемый способ удаления конфиденциальных данных из выходных данных шагов?
Вместо этого мы рекомендуем использовать [E2E шифрование](/docs/reference/middleware/examples#e2-e-encryption), так как это более безопасно, и открытые данные никогда не покидают ваши серверы.

## Почему я получаю ошибку `FUNCTION_INVOCATION_TIMEOUT`?
Это ошибка Vercel, которая означает, что время ожидания вашей функции истекло в инфраструктуре Vercel до того, как она смогла ответить Inngest. Дополнительную информацию можно найти в [документации Vercel](https://vercel.com/docs/functions/serverless-functions/runtimes#max-duration).

Если вы не можете достаточно увеличить таймаут в Vercel, вам может помочь наша [функция стриминга](/docs/streaming).

## Эндпоинт serve моего приложения требует аутентификации. Что мне делать?
Ваш [эндпоинт serve](/docs/learn/serving-inngest-functions) должен быть доступен нашим серверам, чтобы мы могли запускать ваши функции. По этой причине мы рекомендуем отключить аутентификацию для эндпоинта serve.

Наши серверы безопасно взаимодействуют с эндпоинтом serve вашего приложения, используя ваш [ключ подписи (signing key)](/docs/learn/serving-inngest-functions#signing-key).

### Vercel
По умолчанию Vercel включает [Deployment Protection](https://vercel.com/docs/security/deployment-protection) как для предварительных просмотров, так и для созданных продакшн-URL. Это означает, что эндпоинт serve вашего приложения будет недоступен для наших серверов, если вы не [отключите Deployment Protection](https://vercel.com/docs/security/deployment-protection#configuring-deployment-protection) или, если вы используете тариф Vercel Pro, не [настроите обход защиты](/docs/deploy/vercel#bypassing-deployment-protection).

## Почему я получаю ошибку `killed` при запуске Dev Server?
Бинарный файл Inngest CLI может быть поврежден, особенно во время обновлений при загрузке. Симптомы также могут включать отсутствие вывода от CLI или ошибку `Segmentation fault`.

Очистите кэш npx, запустив `rm -rf ~/.npm/_npx`, или кэш того пакетного менеджера, который вы используете для запуска Dev Server (например, `pnpm prune`, `yarn cache clean`).

Если ошибка сохраняется, пожалуйста, свяжитесь с нами в [нашем Discord](https://www.inngest.com/discord).

## Почему я получаю ошибку `NON_DETERMINISTIC_FUNCTION`?
Это ошибка, присутствующая в версии v2.x.x TypeScript SDK, которая может возникнуть, когда развертывание изменяет функцию в середине выполнения.

Если вы видите эту ошибку, мы рекомендуем вам обновиться до версии v3.x.x TypeScript SDK, которая корректно восстанавливается и продолжает работу в таких обстоятельствах.

Для получения дополнительной информации см. руководство по миграции [Обновление с v2 до v3](/docs/sdk/migration).

## Почему я получаю ошибку `Illegal invocation`?
При выполнении запросов к серверу Inngest TypeScript SDK использует [`fetch`](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API). Реализация этого API различается в зависимости от среды выполнения, версии и окружения. SDK старается учитывать эти различия внутренне, но иногда предоставление кастомной функции `fetch` необходимо или желательно.

Эта ошибка обычно указывает на то, что кастомная функция `fetch` была передана в вызов `new Inngest()` или `serve()`, но без сохранения ее [привязки (binding)](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_objects/Function/bind). Это распространенная особенность JavaScript, когда методы теряют свою привязку при передаче в объект.

Чтобы решить эту проблему, убедитесь, что вы повторно привязываете функцию `fetch` при передаче. Обычно она привязывается к `globalThis`, хотя в зависимости от вашей среды выполнения/версии/окружения это может варьироваться.

```ts
new Inngest({
  fetch: fetch.bind(globalThis),
});
```

## Почему я получаю ошибку `Invalid signature` в моих запусках?
Недавние изменения в среде выполнения Vercel выявили проблему проверки подписи в старых версиях SDK. Обновление до TypeScript SDK `3.50.0` (или выше) решает эту проблему и восстанавливает нормальное поведение проверки запусков.

## Почему сервер разработки опрашивает несуществующие эндпоинты?
Сервер разработки автоматически обнаруживает и подключается к приложениям, работающим на стандартных портах и эндпоинтах. Эти эндпоинты включают `/api/inngest`, `/x/inngest`, `/.netlify/functions/inngest`, `/.redwood/functions/inngest`.

Вы можете отключить автоматическое обнаружение, передав флаг `--no-discovery` команде `dev`:

```sh
npx --ignore-scripts=false inngest-cli@latest dev --no-discovery
```

Узнайте больше об этом в документации [dev server](/docs/dev-server#auto-discovery).

## Почему Dev Server ничего не делает при запуске?

Если команда Dev Server (`npx inngest-cli dev`) ничего не делает, это может быть связано с тем, что вы отключили скрипты установки npm. Попробуйте очистить кэш npm и повторно запустить команду Dev Server с флагом `--ignore-scripts=false`:

```sh
$ rm -rf ~/.npm/_npx/*
$ npx --ignore-scripts=false inngest-cli dev
```

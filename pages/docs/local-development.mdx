# Локальная разработка

Инструменты Inngest позволяют легко разрабатывать функции локально с любым фреймворком с помощью Inngest Dev Server.

Inngest Dev Server — это полнофункциональная [open-source](https://github.com/inngest/inngest) локальная версия [платформы Inngest](/docs/platform/deployment), обеспечивающая плавный переход от локальной разработки к preview-, staging- и production-окружениям.

![Демонстрация Dev Server](/assets/docs/local-development/dev-server-demo-2025-01-15.gif)

## Начало работы

Вы можете запустить dev server одной командой. Dev server попытается найти эндпоинт Inngest `serve` API, сканируя порты и эндпоинты, которые обычно используются для этой цели (смотрите раздел «[Авто-обнаружение](#auto-discovery)»). Либо вы можете указать URL эндпоинта `serve` вручную:

<CodeGroup>
```shell {{ title: "npx (npm)" }}
npx --ignore-scripts=false inngest-cli@latest dev
# Вы можете указать URL вашего эндпоинта `serve` API для разработки
npx --ignore-scripts=false inngest-cli@latest dev -u http://localhost:3000/api/inngest
```
```shell {{ title: "Docker" }}
docker run -p 8288:8288 -p 8289:8289 inngest/inngest \
  inngest dev -u http://host.docker.internal:3000/api/inngest
```
</CodeGroup>

Теперь вы можете открыть браузерный интерфейс dev server по адресу [`http://localhost:8288`](http://localhost:8288).

## Подключение приложений к Dev Server

Существует два способа подключить приложения к Dev Server:

1. **Автоматически**: Dev Server попытается «автоматически обнаружить» приложения, работающие на распространённых портах и эндпоинтах (смотрите раздел «[Авто-обнаружение](#auto-discovery)»).
2. **Вручную**: Вы можете явно добавить URL приложения в Dev Server одним из следующих способов:
    - С помощью параметра `-u` CLI (например, `npx --ignore-scripts=false inngest-cli@latest dev -u http://localhost:3000/api/inngest`)
    - Добавив URL на странице Apps в Dev Server. Вы можете редактировать URL или удалить вручную добавленное приложение в любое время
    - Используя файл конфигурации `inngest.json` (или аналогичный) (смотрите раздел «[Файл конфигурации](#configuration-file)»)

![Демонстрация ручной синхронизации приложения в Dev Server](/assets/docs/local-development/dev-server-apps-demo-2025-01-15.gif)

<Tip>
  Dev server выполняет «авто-обнаружение», сканируя популярные порты и эндпоинты, такие как `/api/inngest` и `/.netlify/functions/inngest`. **Если вы хотите отключить авто-обнаружение, передайте флаг `--no-discovery` команде `dev`**. Подробнее об этом [ниже](#auto-discovery).
</Tip>

### Отправка событий в Dev Server

При локальном тестировании события можно отправлять в dev server разными способами:

1. С помощью Inngest SDK
2. С помощью кнопки «Test Event» в интерфейсе Dev Server
3. Через HTTP-запрос (например, curl)

#### С помощью Inngest SDK

При локальном использовании Inngest SDK он пытается определить, запущен ли dev server на вашей машине. Если да — событие будет отправлено туда.

<CodeGroup>
```ts {{ title: "Node.js" }}
import { Inngest } from "inngest";

const inngest = new Inngest({ id: "my_app" });
await inngest.send({
  name: "user.avatar.uploaded",
  data: { url: "https://a-bucket.s3.us-west-2.amazonaws.com/..." },
});
```

```python {{ title: "Python" }}
from inngest import Inngest

inngest_client = inngest.Inngest(app_id="my_app")
await inngest_client.send(
  name="user.avatar.uploaded",
  data={"url": "https://a-bucket.s3.us-west-2.amazonaws.com/..."},
)
```

```go {{ title: "Go" }}
!snippet:path=snippets/go/docs/examples/devserver/main.go
```
</CodeGroup>
**Примечание** — В процессе локальной разработки для переменной окружения [`INNGEST_EVENT_KEY`](/docs/sdk/environment-variables#inngest-event-key?ref=local-development) можно использовать любое значение-заглушку. Dev server не валидирует ключи локально.

#### С помощью кнопки «Test Event»

Интерфейс dev server также имеет кнопку «Test Event» в правом верхнем углу, которая позволяет ввести любую JSON-нагрузку события и отправить её вручную. Это удобно для тестирования различных вариантов полезных нагрузок с вашими функциями.

#### Через HTTP-запрос

Все события отправляются в Inngest через простое HTTP API с JSON-телом. Вот пример curl-запроса к эндпоинту `/e/<EVENT_KEY>` локального dev server, запущенного на порту `8288` по умолчанию, с заглушкой ключа события `123`:

```shell
curl -X POST -v "http://localhost:8288/e/123" \
  -d '{
    "name": "user.avatar.uploaded",
    "data": { "url": "https://a-bucket.s3.us-west-2.amazonaws.com/..." }
  }'
```

<Callout>
  💡 Поскольку события можно отправлять через HTTP, это означает, что вы можете отправлять события с использованием любого языка программирования или из любимых инструментов тестирования, например Postman.
</Callout>

## Файл конфигурации

При использовании множества параметров конфигурации или указании нескольких флагов `-u` для проекта вы можете настроить CLI через файл конфигурации `inngest.json`. Команда `dev` начинает поиск с текущего каталога и поднимается по дереву директорий, пока не найдёт файл. Также поддерживаются форматы и расширения `yaml`, `yml`, `toml` и `properties`. Все параметры можно посмотреть с помощью `dev --help`. Пример файла с двумя URL приложений и параметром `no-discovery`:

<CodeGroup>
```json {{ title: "inngest.json" }}
{
  "sdk-url": [
    "http://localhost:3000/api/inngest",
    "http://localhost:3030/api/inngest"
  ],
  "no-discovery": true
}
```
```yaml {{ title: "inngest.yaml" }}
sdk-url:
  - "http://localhost:3000/api/inngest"
  - "http://localhost:3030/api/inngest"
no-discovery: true
```
</CodeGroup>


## Отладочный эндпоинт Inngest SDK

[Эндпоинт `serve` API SDK](/docs/learn/serving-inngest-functions) возвращает диагностическую информацию о конфигурации вашего сервера при GET-запросе. Это можно сделать через команду `curl` или открыв URL в браузере.

Пример curl-запроса к приложению Inngest, запущенному на `http://localhost:3000/api/inngest`:

```sh
$ curl -s http://localhost:3000/api/inngest | jq
{
  "message": "Inngest endpoint configured correctly.",
  "hasEventKey": false,
  "hasSigningKey": false,
  "functionsFound": 1
}
```

## Авто-обнаружение

Dev server автоматически обнаруживает и подключается к приложениям, работающим на распространённых портах и эндпоинтах. Вы можете отключить авто-обнаружение, передав флаг `--no-discovery` команде `dev`:

```sh
npx --ignore-scripts=false inngest-cli@latest dev --no-discovery -u http://localhost:3000/api/inngest
```

## Разработка с Docker

Inngest предоставляет Docker-образ для запуска Inngest Dev Server в контейнере. Это удобно при локальном запуске Inngest или в CI/CD-окружении.

### Docker-образ

Образ [`inngest/inngest`](https://hub.docker.com/r/inngest/inngest) доступен на Docker Hub. Образ регулярно обновляется, поэтому мы рекомендуем загружать последнюю версию. Последний релиз можно найти в [нашем репозитории Github](https://github.com/inngest/inngest/releases).

```bash
docker pull inngest/inngest
```

### Отдельный Docker-контейнер

Docker удобен для запуска Inngest Dev Server в отдельном контейнере. Это полезно, если вы не хотите использовать метод `npx --ignore-scripts=false inngest-cli@latest` для запуска Dev Server.

Для запуска контейнера Inngest вам нужно:

1. Открыть порты Dev Server (`8288`, `8289`).
2. Использовать команду `inngest dev` с флагом `-u`, чтобы указать URL, по которому Inngest сможет найти ваше приложение.

В этом примере приложение работает на хост-машине на порту `3000`. Мы используем хостнейм `host.docker.internal` для подключения к хост-машине изнутри Docker-контейнера. Для удобства чтения команда разбита на несколько строк.

```bash
docker run -p 8288:8288 -p 8289:8289 \
  inngest/inngest \
  inngest dev -u http://host.docker.internal:3000/api/inngest
```

После этого вы сможете получить доступ к Inngest Dev Server на хост-машине по адресу `http://localhost:8288` или любому другому сконфигурированному вами хостнейму. При использовании другой настройки Docker-сети может потребоваться скорректировать хостнейм приложения.

<Callout>
  Если вы решите запустить Dev Server на другом порту, вам нужно будет установить переменную окружения `INNGEST_BASE_URL` в вашем приложении, указав правильный порт. По умолчанию это значение равно `http://localhost:8288`.
</Callout>

### Docker Compose

Если вы используете [Docker Compose](https://docs.docker.com/compose/) для локального запуска сервисов, вы можете легко добавить Inngest в своё локальное окружение. Пример файла `docker-compose.yml` с включённым Inngest:

```yaml {{ filename: "docker-compose.yaml" }}
services:
  app:
    build: ./app
    environment:
      - INNGEST_DEV=1
      - INNGEST_BASE_URL=http://inngest:8288
    ports:
      - '3000:3000'
  inngest:
    image: inngest/inngest:v1.17.2 # Обновите до последней версии
    command: 'inngest dev -u http://app:3000/api/inngest'
    ports:
      - '8288:8288'
      - '8289:8289' # Используется для connect()
```

В этом примере есть два сервиса: `app` и `inngest`. Сервис `app` — это ваше приложение, а сервис `inngest` — Inngest Dev Server. Несколько ключевых настроек:

* Переменная окружения `INNGEST_DEV=1` сообщает Inngest SDK, что нужно подключиться к Dev Server*.
* Переменная окружения `INNGEST_BASE_URL=http://inngest:8288` сообщает Inngest SDK, где запущен Dev Server. В нашем примере сервис `inngest` работает на порту `8288` (порт Dev Server по умолчанию).
* Команда `command: 'inngest dev -u http://app:3000/api/inngest'` указывает Dev Server, где найти ваше приложение в Docker-сети. В этом примере сервис `app` работает на порту `3000`.
* Конфигурация `ports` открывает Dev Server на порту `8288`, чтобы вы могли просматривать его в браузере на хост-машине. Также открывается порт `8289`, который используется для [`connect()`](/docs/setup/connect).

\* — TypeScript SDK по умолчанию работает в облачном режиме. Установка `INNGEST_DEV=1` явно включает dev-режим, который подключается к Dev Server и отключает верификацию подписи.

## Процесс разработки

Разработка с Inngest выглядит следующим образом:

1. Настройте Inngest SDK в вашем приложении
2. [Подключите Inngest Dev Server к вашему локальному приложению](#connecting-apps-to-the-dev-server)
3. Разрабатывайте ваши Inngest-функции с помощью [Steps](/docs/learn/inngest-steps), [Flow Control](/docs/guides/flow-control) и [других возможностей](/docs/learn/inngest-functions)
5. _(Необязательно) — Настройте Preview-окружения с помощью [интеграции с Vercel](/docs/deploy/vercel)_

6. [Создайте приложение Inngest](/docs/apps) на платформе Inngest и [настройте его Event и Signing Keys в облаке](/docs/platform/deployment).
7. Используйте платформу Inngest для управления и мониторинга событий и запусков функций

## Часто задаваемые вопросы

<CardGroup cols={1}>
  <Card title="Можно ли запускать Inngest Dev Server в продакшене?" icon={<RiQuestionLine />}>
    Inngest Dev Server не предназначен для запуска в продакшене, но вы можете запускать его где угодно, включая тестовые окружения или CI/CD-пайплайны.

  <Card title="Как тестировать вебхуки локально?" icon={<RiQuestionLine />}>

    Внешние вебхуки от Stripe и Clerk должны проходить через туннельное решение (например, [ngrok](https://ngrok.com/) или [localtunnel](https://theboroer.github.io/localtunnel-www/)) для доступа к Dev Server.
  </Card>
  <Card title="Поддерживаются ли Cron-задачи локально?" icon={<RiQuestionLine />}>
    Да. Вы также можете запустить функцию в любое время, используя кнопку «Invoke» в списке функций Dev Server.
  </Card>
</CardGroup>

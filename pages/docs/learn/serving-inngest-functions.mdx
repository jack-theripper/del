# Настройка вашего приложения Inngest

С помощью Inngest вы определяете функции или рабочие процессы с использованием SDK и развертываете их на любой платформе или облачном провайдере, включая серверлесс-среды и контейнеры.

Для того чтобы Inngest мог удаленно выполнять ваши функции, вам необходимо настроить соединение между вашим приложением и Inngest. Это можно сделать одним из двух способов:

## Обслуживание функций Inngest

<LanguageSelector>

<LanguageSection show="typescript">

Inngest предоставляет обработчик `serve()`, который добавляет API-эндпоинт в ваш роутер. Вы открываете свои функции для Inngest через этот HTTP-эндпоинт. Чтобы упростить автоматическое развертывание, **эндпоинт должен быть определен по пути `/api/inngest`** (хотя вы можете [изменить путь к API](/docs/reference/serve#serve-client-functions-options)).

```ts {{ title: "./api/inngest.ts" }}
// Все обработчики serve имеют одинаковые аргументы:
serve({
  client: inngest, // клиент, созданный с помощью new Inngest()
  functions: [fnA, fnB], // массив функций Inngest для обслуживания, созданных с помощью inngest.createFunction()
  /* Дополнительная опциональная конфигурация */
});
```

<CodeGroup>

```ts
import { serve } from "inngest/astro";
import { functions, inngest } from "../../inngest";

export const { GET, POST, PUT } = serve({

  functions,
});
```

### Фреймворк: AWS Lambda

Мы рекомендуем использовать [Lambda function URLs](https://docs.aws.amazon.com/lambda/latest/dg/lambda-urls.html) для запуска ваших функций, так как они не требуют дополнительной настройки или затрат.

В качестве альтернативы вы можете использовать API Gateway для маршрутизации запросов к вашей Lambda. Обработчик поддерживает [API Gateway V1](https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-develop-integrations-lambda.html) и [API Gateway V2](https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-develop-integrations-lambda.html). Если вы используете API Gateway за прокси-сервером или имеете другую специфическую конфигурацию, вам может потребоваться указать параметры `serveOrigin` и `servePath` при вызове `serve()`, чтобы Inngest знал URL, по которому вы обслуживаете свои функции. См. [Настройка пути API](/docs/reference/serve#serve-client-functions-options) для подробностей.

<CodeGroup>
```ts
import { serve } from "inngest/lambda";
import { inngest } from "./client";
import fnA from "./fnA"; // Your own function

export const handler = serve({
  client: inngest,
  functions: [fnA],
});
```
</CodeGroup>

### Bun.serve()

Вы можете использовать обработчик `inngest/bun` с `Bun.serve()` для легковесного сервера Inngest:

```ts {{ title: "index.ts" }}
import { serve } from "inngest/bun";
import { functions, inngest } from "./inngest";

Bun.serve({
  port: 3000,
  routes: {
    // ...other routes...
    "/api/inngest": serve({ client: inngest, functions }),
  },
});
```

См. [пример для Bun](https://github.com/inngest/inngest-js/tree/main/examples/bun) для получения дополнительной информации.

</CodeGroup>

#### Локальная разработка с Wrangler

При локальной разработке с использованием Wrangler и флага `--remote` ваш код развертывается и запускается удаленно. Чтобы использовать это с локальным сервером Inngest Dev Server, вы должны использовать такой инструмент, как [ngrok](https://ngrok.com/) или [localtunnel](https://theboroer.github.io/localtunnel-www/), чтобы обеспечить доступ к Dev Server из интернета.

```sh
ngrok http 8288
```

```toml {{ title: "wrangler.toml" }}
[vars]
# URL вашего туннеля. Это позволяет "облачному" воркеру получить доступ к локальному Dev Server
INNGEST_DEV = "https://YOUR_TUNNEL_URL.ngrok.app"
# Это может потребоваться:
# URL вашего локального сервера. Это позволяет Dev Server получить доступ к приложению по этому локальному URL
# Вам может потребоваться изменить этот URL, если ваше локальное приложение работает на другом порту.
# Без этого "облачный" воркер может попытаться перенаправить Inngest на неправильный URL.
INNGEST_SERVE_ORIGIN = "http://localhost:8787"
```

См. пример этого в [примере фреймворка Hono на GitHub](https://github.com/inngest/inngest-js/tree/main/examples/framework-hono).

### Фреймворк: DigitalOcean Functions

Функция обслуживания DigitalOcean позволяет развертывать Inngest в серверлесс-функциях DigitalOcean.
Поскольку DigitalOcean не предоставляет URL запроса в аргументах функции, вы **должны** включить
URL и путь функции при настройке обработчика:

<CodeGroup>
```ts
import { serve } from "inngest/digitalocean";
import { inngest } from "./src/inngest/client";
import fnA from "./src/inngest/fnA"; // Your own function

const main = serve({
  client: inngest,
  functions: [fnA],
  // Ваше имя хоста digitalocean.  Это необходимо, иначе ваши функции не будут работать.
  serveOrigin: "https://faas-sfo3-your-url.doserverless.co",
  // И ваш путь выполнения также обязателен.
  servePath: "/api/v1/web/fn-your-uuid/inngest",
});

// ВАЖНО: Делает функцию доступной как модуль в проекте.
// Это необходимо для любых функций, использующих внешние зависимости.
module.exports.main = main;
```
</CodeGroup>

Функции Inngest также могут быть развернуты на [App Platform или Droplets от DigitalOcean](/docs/deploy/digital-ocean).


### Фреймворк: ElysiaJS

Для [вариантов развертывания](https://elysiajs.com/patterns/deploy.html), Elysia может компилироваться в бинарный файл или в JavaScript, а также вы можете развернуться с помощью Docker или Railway.


```ts {{ title: "src/index.ts" }}
import { Elysia } from "elysia";
import { serve } from "inngest/bun";
import { functions, inngest } from "./inngest";

const handler = serve({
  client: inngest,
  functions,
});

const inngestHandler = new Elysia().all("/api/inngest", ({ request }) =>
  handler(request)
);

// зарегистрируйте обработчика в Elysia
const app = new Elysia()
.use(inngestHandler)

```

Функция `use` в Elysia ожидает один аргумент. Мы используем метод `all` для маршрута API Inngest, чтобы обрабатывать ожидаемые методы, а затем получаем запрос из объекта контекста, передаваемого в обработчики Elysia.

См. [пример для ElysiaJS](https://github.com/inngest/inngest-js/tree/main/examples/framework-elysiajs) для получения дополнительной информации.

### Фреймворк: Express

Вы можете обслуживать функции Inngest внутри вашего существующего приложения Express, развернутого у любого хостинг-провайдера, такого как Render, Fly, AWS, K8S и др.:

<CodeGroup>
```ts
import { serve } from "inngest/express";
import { inngest } from "./src/inngest/client";
import fnA from "./src/inngest/fnA"; // Your own function

// Важно: убедитесь, что вы добавили JSON-middleware для обработки входящих JSON-тел в POST-запросах.
app.use(express.json());
app.use(
  // Разместите middleware по нашему рекомендуемому пути `/api/inngest`.
  "/api/inngest",
  serve({ client: inngest, functions: [fnA] })
);
```
</CodeGroup>

Вы должны убедиться, что используете middleware `express.json()`, иначе ваши функции не будут выполнены. **Примечание** — вам может потребоваться установить параметр `limit` в `express.json()` выше значения по умолчанию `100kb`, чтобы поддерживать большие полезные нагрузки событий и состояния функций.

См. [пример для Express](https://github.com/inngest/inngest-js/tree/main/examples/framework-express) для получения дополнительной информации.

#### Стриминг

Express также может передавать ответы обратно в Inngest в потоковом режиме, что потенциально позволяет использовать гораздо более длительные тайм-ауты.

Чтобы включить это, добавьте параметр `streaming: true` в ваш обработчик serve:

<CodeGroup>
```ts
const handler = serve({
  client: inngest,
  functions: [...fns],
  streaming: true,
});
```
</CodeGroup>

Для получения дополнительной информации посетите страницу [Стриминг](/docs/streaming).


### Фреймворк: Fastify

Вы можете обслуживать функции Inngest внутри вашего существующего приложения Fastify.

Мы рекомендуем использовать экспортируемый плагин `inngestFastify`, хотя мы также предоставляем общую функцию `serve()`, если вы хотите вручную создать маршрут.

<CodeGroup>
```ts {{ title: "Plugin" }}
import Fastify from "fastify";
import { fastifyPlugin } from "inngest/fastify";
import { inngest, fnA } from "./inngest";

const fastify = Fastify();

fastify.register(fastifyPlugin, {
  client: inngest,
  functions: [fnA],
  options: {},
});

fastify.listen({ port: 3000 }, function (err, address) {
  if (err) {
    fastify.log.error(err);
    process.exit(1);
  }
});
```
```ts {{ title: "Custom route" }}
import Fastify from "fastify";
import { serve } from "inngest/fastify";
import { fnA, inngest } from "./inngest";

const fastify = Fastify();

fastify.route({
  method: ["GET", "POST", "PUT"],
  handler: serve({ client: inngest, functions: [fnA] }),
  url: "/api/inngest",
});

fastify.listen({ port: 3000 }, function (err, address) {
  if (err) {
    fastify.log.error(err);
    process.exit(1);
  }
});
```
</CodeGroup>

См. [пример для Fastify](https://github.com/inngest/inngest-js/tree/main/examples/framework-fastify) для получения дополнительной информации.

### Framework: Google Cloud Run Functions

Google's [Functions Framework](https://github.com/GoogleCloudPlatform/functions-framework-nodejs) has an Express-compatible API which enables you to use the Express serve handler to deploy your Inngest functions to Google Cloud Run. This is an example of a function:

<CodeGroup>
```ts
import * as ff from "@google-cloud/functions-framework";
import { serve } from "inngest/express";
import { inngest } from "./src/inngest/client";
import fnA from "./src/inngest/fnA"; // Your own function

ff.http(
  "inngest",
  serve({
    client: inngest,
    functions: [fnA],
    servePath: "/",
  })
);
```
</CodeGroup>

You can run this locally with `npx @google-cloud/functions-framework --target=inngest` which will serve your Inngest functions on port `8080`.

See the [Google Cloud Functions example](https://github.com/inngest/inngest-js/tree/main/examples/framework-google-functions-framework) for more information.

<Callout>
1st generation Cloud Run Functions are not officially supported. Using one may result in a signature verification error.
</Callout>

### Framework: Firebase Cloud Functions

Based on the Google Cloud Function architecture, the Firebase Cloud Functions provide a different API to serve functions using `onRequest`:

```typescript
import { onRequest } from "firebase-functions/v2/https";

import { serve } from "inngest/express";
import { inngest as inngestClient } from "./inngest/client";

export const inngest = onRequest(
  serve({
    client: inngestClient,
    functions: [/* ...functions... */],
  })
);
```

<Callout>

Firebase Cloud Functions require configuring `INNGEST_SERVE_PATH` with the custom function path.

For example, for a project named `inngest-firebase-functions` deployed on the `us-central1` region, the `INNGEST_SERVE_PATH` value will be as follows:

```
/inngest-firebase-functions/us-central1/inngest/
```

</Callout>

To serve your Firebase Cloud Function locally, use the following command:

```bash
firebase emulators:start
```

Please note that you'll need to start your Inngest Local Dev Server with the `-u` flag to match our Firebase Cloud Function's custom path  as follows:

```bash
npx --ignore-scripts=false inngest-cli@latest dev -u http://127.0.0.1:5001/inngest-firebase-functions/us-central1/inngest
```

_The above command example features a project named `inngest-firebase-functions` deployed on the `us-central1` region_.

### Framework: H3

Inngest supports [H3](https://github.com/unjs/h3) and frameworks built upon it. Here's a simple H3 server that hosts serves an Inngest function.

<CodeGroup filename="index.js">
```ts
import { createApp, eventHandler, toNodeListener } from "h3";
import { serve } from "inngest/h3";
import { createServer } from "node:http";
import { inngest } from "./inngest/client";
import fnA from "./inngest/fnA";

const app = createApp();
app.use(
  "/api/inngest",
  eventHandler(
    serve({
      client: inngest,
      functions: [fnA],
    })
  )
);

createServer(toNodeListener(app)).listen(process.env.PORT || 3000);
```
</CodeGroup>

See the [github.com/unjs/h3](https://github.com/unjs/h3) repository for more information about how to host an H3 endpoint.

### Framework: Koa

Add the following to your routing file:

<CodeGroup>
```ts
import { serve } from "inngest/koa";
import Koa from "koa";
import bodyParser from "koa-bodyparser";
import { functions, inngest } from "./inngest";

const app = new Koa();
app.use(bodyParser()); // make sure we're parsing incoming JSON

const handler = serve({
  client: inngest,
  functions,

app.use((ctx) => {
  if (ctx.request.path === "/api/inngest") {
    return handler(ctx);
  }
});
```
</CodeGroup>

### Framework: NestJS

Add the following to `./src/main.ts`:

<CodeGroup>
```ts
import { Logger } from '@nestjs/common';
import { NestFactory } from '@nestjs/core';
import { NestExpressApplication } from '@nestjs/platform-express';
import { serve } from 'inngest/express';

import { inngest } from '@modules/common/inngest/client';
import { getInngestFunctions } from '@modules/common/inngest/functions';

import { AppModule } from './app.module';
import { AppService } from './app.service';

async function bootstrap() {
  const app = await NestFactory.create<NestExpressApplication>(AppModule, {
    bodyParser: true,
  });

  // Setup inngest
  app.useBodyParser('json', { limit: '10mb' });

  // Inject Dependencies into inngest functions

  const logger = app.get(Logger);
  const appService = app.get(AppService);

  // Pass dependencies into this function
  const inngestFunctions = getInngestFunctions({
    appService,
    logger,
  });

  // Register inngest endpoint
  app.use(
    '/api/inngest',
    serve({
      client: inngest,
      functions: inngestFunctions,
    }),
  );

  // Start listening for http requests
  await app.listen(3000);
}

bootstrap();
```
</CodeGroup>

See the [NestJS example](https://github.com/inngest/inngest-js/tree/main/examples/framework-nestjs) for more information.


### Framework: Next.js

Inngest has first class support for Next.js API routes, allowing you to easily create the Inngest API. Both the App Router and the Pages Router are supported. For the App Router, Inngest requires `GET`, `POST`, and `PUT` methods.

<CodeGroup>
```typescript {{ title: "App Router" }}
// src/app/api/inngest/route.ts
import { serve } from "inngest/next";
import { inngest } from "../../../inngest/client";
import fnA from "../../../inngest/fnA"; // Your own functions

export const { GET, POST, PUT } = serve({
  client: inngest,
  functions: [fnA],
});
```
```typescript {{ title: "Pages Router" }}
// pages/api/inngest.ts
import { serve } from "inngest/next";
import { inngest } from "../../inngest/client";
```

#### Streaming

Next.js Functions hosted on [Vercel](/docs/deploy/vercel) with Fluid compute can stream responses back to Inngest which can help you reach the maximum duration of 800s (13m20s) provided you are on a paid Vercel plan.

To enable this, add the `streaming: true` option to your serve handler:

**Next.js 13+ on Fluid compute**
<CodeGroup>
```ts
export const { GET, POST, PUT } = serve({
  client: inngest,
  functions: [...fns],
  streaming: true,
});
```
</CodeGroup>


<details>
  <summary>**Edge runtime**</summary>
  <div>
  If you are not using Vercel Fluid compute, you can also stream responses to Inngest by running on their [edge runtime](https://vercel.com/docs/functions/runtimes/edge).

  To enable this, set your runtime to `"edge"` and add the `streaming: true` option to your serve handler:

  **Next.js 13+**
  <CodeGroup>
  ```ts
  export const runtime = "edge";

  export const { GET, POST, PUT } = serve({
    client: inngest,
    functions: [...fns],
    streaming: true,
  });
  ```
  </CodeGroup>
  </div>
</details>

<details>
  <summary>**Older versions (Next.js 12)**</summary>
  <CodeGroup>
    ```ts
    export const config = {
      runtime: "edge",
    };

    const handler = serve({
      client: inngest,
      functions: [...fns],
      streaming: true,
    });
    ```
  </CodeGroup>
</details>

For more information, check out the [Streaming](/docs/streaming) page.

### Кастомные фреймворки

Если фреймворк, который использует ваше приложение, не входит в приведенный выше список официально поддерживаемых, вы можете создать собственный обработчик `serve`.

Чтобы создать свой собственный обработчик, изучите [пример обработчика](https://github.com/inngest/inngest-js/blob/main/packages/inngest/src/test/functions/handler.ts) в открытом репозитории нашего SDK, чтобы понять, как он работает. Вот пример создания и использования кастомного обработчика:

```ts
import { Inngest, InngestCommHandler, type ServeHandlerOptions } from "inngest";

const serve = (options: ServeHandlerOptions) => {
  const handler = new InngestCommHandler({
    frameworkName: "edge",
    fetch: fetch.bind(globalThis),
    ...options,
    handler: (req: Request) => {
      return {
        body: () => req.json(),
        headers: (key) => req.headers.get(key),
        method: () => req.method,
        url: () => new URL(req.url, `https://${req.headers.get("host") || ""}`),
        transformResponse: ({ body, status, headers }) => {
          return new Response(body, { status, headers });
        },
      };
    },
  });

  return handler.createHandler();
};

const inngest = new Inngest({ id: "example-edge-app" });

const fn = inngest.createFunction(
  { id: "hello-world", triggers: { event: "test/hello.world" } },
  () => "Hello, World!"
);

export default serve({ client: inngest, functions: [fn] });
```

### Ключ подписи

Вам потребуется назначить ваш [ключ подписи](/docs/platform/signing-keys) переменной окружения [`INNGEST_SIGNING_KEY`](/docs/sdk/environment-variables#inngest-signing-key) у вашего хостинг-провайдера или в локальном файле `.env`, что позволит SDK безопасно взаимодействовать с Inngest. Если вы не можете использовать переменную окружения, вы можете передать ключ подписи напрямую в конструктор клиента Inngest через параметр `signingKey`. [Прочтите справочник для получения дополнительной информации](/docs/sdk/reference/serve#reference).

### Другая конфигурация

При использовании `serve` разрешайте запросы размером до 4 МБ. Это максимальный размер запроса, который Inngest будет отправлять в ваше приложение. Настройка максимального размера запроса зависит от фреймворка, поэтому обратитесь к документации вашего фреймворка за подробностями.

# Безопасность

Безопасность является первоочередным соображением при переводе систем в промышленную эксплуатацию. В этом разделе мы разберем, как Inngest обеспечивает безопасность, включая защиту эндпоинтов, шифрование, стандартные практики и добавление аутентификации SAML в ваш аккаунт.

## Соответствие стандартам, аудиты и отчеты

Inngest [соответствует стандарту SOC 2 Type II](/blog/soc2-compliant?ref=docs-security). Наша компания и платформа регулярно проходят аудит на соответствие стандартам SOC 2. Это гарантирует наличие необходимых мер контроля для защиты данных наших клиентов и обеспечения безопасности и конфиденциальности их информации. Наша платформа и SDK периодически проходят независимую оценку безопасности, включая тестирование на проникновение и имитацию атак (red-team).

Для получения дополнительной информации о наших практиках безопасности или для запроса копии отчета SOC 2, пожалуйста, [свяжитесь с нашей командой](/contact?ref=docs-security).

## Ключи подписи и безопасность SDK

Во-первых, важно понимать, что в промышленной среде вся связь между Inngest и вашими серверами зашифрована через TLS. SDK также активно противодействует атакам, используя [ключ подписи (signing key)](/docs/platform/signing-keys) — секретный предварительно распределенный ключ, уникальный для каждой среды. По умолчанию ключ подписи обеспечивает следующее:
- **Аутентификация**: запросы к вашему эндпоинту аутентифицируются, что гарантирует их отправку именно из Inngest. SDK Inngest отклоняют все запросы, которые не аутентифицированы ключом подписи.
- **Предотвращение атак повторного воспроизведения**: запросы подписываются со встроенной меткой времени, и старые запросы отклоняются, даже если они правильно аутентифицированы.

**Важно держать ключ подписи в секрете.** Если ваш ключ подписи будет раскрыт, это поставит под угрозу безопасность ваших эндпоинтов. Обратите внимание, что можно [ротировать ключи подписи](/docs/platform/signing-keys#rotation) без простоя системы.

### Регистрация функций + рукопожатие (handshake)

Функции определяются и обслуживаются на вашей собственной инфраструктуре с использованием одного из наших SDK. Чтобы запускать функции, их необходимо [синхронизировать](/docs/apps/cloud) или зарегистрировать в вашем аккаунте Inngest. Это требуется для того, чтобы Inngest знал, какие функции обслуживает ваше приложение, и какова конфигурация каждой функции. Синхронизация функций выполняется через безопасное рукопожатие (handshake). Вот как это работает:
1. После того как эндпоинт вашего SDK станет доступен, PUT-запрос к нему инициирует безопасное рукопожатие с серверами Inngest.
2. SDK отправляет конфигурацию функций в API Inngest, используя ключ подписи в качестве bearer-токена.
3. SDK идемпотентно обновляет ваши приложения и функции. Если изменений нет, ничего не происходит.

Этот процесс необходим по нескольким причинам, во многом из-за того, что серверлесс-среды являются «наименьшим общим знаменателем». У серверлесс-сред нет стандартного процесса загрузки/инициализации, а значит, они не могут сами инициировать синхронизацию. Во-вторых, серверлесс-платформы, такие как AWS Lambda и Vercel, создают уникальные URL для каждой развернутой функции, которые невозможно узнать заранее. Входящий PUT-запрос позволяет SDK определить URL запроса, который затем используется для всех вызовов функций.

Обратите внимание, что поскольку SDK отправляет только HTTP-запросы к `api.inngest.com` для завершения синхронизации, он никогда не передает ключ подписи клиентам, пытающимся выполнить регистрацию, сохраняя ваш ключ в безопасности.

## Сквозное шифрование (End to end encryption)

Inngest запускает функции автоматически на основе данных событий, которые вы отправляете. Кроме того, Inngest выполняет шаги транзакционно и сохраняет результат каждого `step.run` в состоянии функции. Это состояние может содержать регулируемые конфиденциальные данные.

**Если вы обрабатываете конфиденциальные данные, мы** **настоятельно рекомендуем (а иногда и требуем) включить сквозное шифрование в наших SDK**. [Сквозное шифрование — это промежуточное ПО (middleware)](/docs/features/middleware/encryption-middleware), которое перехватывает запросы, ответы и логику SDK на ваших собственных серверах. При сквозном шифровании данные шифруются на ваших серверах ключом, к которому есть доступ только у вас. Применяется следующее:
- Все данные в `event.data.encrypted` шифруются _до того_, как они покинут ваши серверы. Inngest никогда не сможет прочитать данные в этом объекте.
- Все результаты шагов и результаты функций шифруются _до того_, как они покинут ваши серверы. Inngest получает только зашифрованные значения и никогда не сможет прочитать эти данные. Состояние функции отправляется в SDK полностью зашифрованным. SDK расшифровывают данные на ваших серверах и затем продолжают работу в обычном режиме.

При включении этой функции даже в случае непредвиденных инцидентов ваши данные остаются зашифрованными и защищенными. Это значительно повышает уровень безопасности при работе с конфиденциальными данными.

## SAML

Корпоративные пользователи (Enterprise) могут включить аутентификацию SAML для доступа к своему аккаунту. Чтобы включить SAML, необходимо:
1. Связаться с вашим аккаунт-менеджером и запросить интеграцию SAML.
2. После этого мы запросим конфигурацию, относящуюся к вашему провайдеру SAML. Она различается в зависимости от провайдера и может включать:
   1. URL метаданных; URL для SSO; IdP entity ID; сертификат IdP x.509 и так далее.
3. Ваш аккаунт-менеджер отправит вам ACS и URL метаданных, используемые для настройки вашего аккаунта.
4. Менеджер поможет вам правильно сопоставить атрибуты, чтобы обеспечить полноценную работу входа в систему.

Важно отметить, что после включения SAML пользователи **должны** входить через SAML. Если вы не на корпоративном тарифе, [свяжитесь с нами здесь](/contact?ref=docs-security), и мы поможем вам с настройкой. Для компаний с числом пользователей менее 200 дополнительная плата за SAML-аутентификацию не взимается.

## IP-адреса

Для обеспечения безопасности и настройки сети вам может потребоваться знать IP-адреса, которые Inngest использует для исходящих запросов к вашим функциям и вебхукам. Эти IP-адреса используются инфраструктурой Inngest для выполнения аутентифицированных запросов к вашим эндпоинтам.

Актуальный список IP-адресов можно найти по ссылкам:
- [IPv4 адреса](https://www.inngest.com/ips-v4)
- [IPv6 адреса](https://www.inngest.com/ips-v6)

Эти диапазоны IP-адресов используются для всех вызовов функций Inngest и доставки вебхуков. Если вам нужно добавить эти IP в белый список вашего брандмауэра или групп безопасности, пожалуйста, используйте полные диапазоны, указанные на этих страницах.

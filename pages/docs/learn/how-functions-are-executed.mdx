# Как выполняются функции Inngest: Durable Execution

Одной из основных особенностей Inngest является Durable Execution (Надежное выполнение). Durable Execution позволяет вашим функциям быть отказоустойчивыми и устойчивыми к сбоям. Конечным результатом является то, что ваш код и, следовательно, все ваше приложение становятся более надежными.

На этой странице рассматривается, что такое Durable Execution, как оно работает и как оно применяется в функциях Inngest.

## Что такое Durable Execution?

Durable Execution — это отказоустойчивый подход к выполнению кода, который достигается за счет изящной обработки сбоев и прерываний с помощью автоматических повторов и сохранения состояния. Это означает, что ваш код может продолжать работу даже при возникновении таких проблем, как сетевые сбои, тайм-ауты, сбои инфраструктуры и другие временные ошибки.

Ключевые аспекты Durable Execution включают:

* **Сохранение состояния** — состояние функции сохраняется вне контекста выполнения функции. Это позволяет возобновить выполнение функции с момента сбоя на той же _или_ другой инфраструктуре.
* **Отказоустойчивость** — ошибки или исключения перехватываются уровнем выполнения и автоматически повторяются. Поведение при повторах можно настроить: задать допустимое количество попыток и способы обработки различных типов ошибок.

На практике Durable Execution реализуется в форме "надежных функций" (durable functions), которые иногда также называют "надежными рабочими процессами" (durable workflows). Надежные функции могут выдавать ошибки или исключения и автоматически повторять попытки, возобновляя выполнение с момента сбоя. Они спроектированы как длительно работающие и сохраняющие состояние, что означает их способность сохранять данные между вызовами функций и повторными попытками.

## Как работают функции Inngest

Функции Inngest являются надежными (durable): они выбрасывают ошибки или исключения, автоматически повторяют попытки с момента сбоя, могут сохранять состояние и выполняться в течение длительного времени.

Функции Inngest используют "**Шаги**" (Steps) для определения потока выполнения функции. Каждый шаг:

* Является единицей работы, которая может быть запущена и перезапущена независимо.
* Перехватывает любую ошибку или исключение, возникшее внутри него.
* Не будет выполнен повторно, если он уже был успешно выполнен.
* Возвращает состояние (_данные_), которое может быть использовано последующими шагами.
* Может выполняться параллельно или последовательно, в зависимости от конфигурации функции.

Сложные функции могут состоять из множества шагов. Это позволяет разбить длительную функцию на более мелкие и управляемые единицы работы. Поскольку каждый шаг повторяется независимо, выполнение функции может быть возобновлено с момента сбоя, что позволяет избежать ненужного повторного выполнения уже проделанной работы.

Для сравнения, некоторые другие системы Durable Execution модифицируют среду выполнения для сохранения состояния или перехвата ошибок. SDK Inngest написаны с использованием стандартных примитивов языка, что позволяет функциям Inngest работать в любой среде или рантайме — включая серверлесс-среды — без модификаций.

### Как выполняются шаги

Функции Inngest определяются как серия шагов, формирующих поток выполнения. Каждый шаг имеет уникальный ID и функцию, описывающую выполняемую работу. Возвращаемые данные могут использоваться последующими шагами.

Функции Inngest выполняются инкрементально, _шаг за шагом_. По мере выполнения функции результаты каждого шага возвращаются в Inngest и сохраняются в управляемом хранилище состояний функций. Успешно выполненные шаги [_мемоизируются_](https://ru.wikipedia.org/wiki/Мемоизация). Затем функция возобновляется, пропуская уже завершенные шаги, а SDK внедряет данные, возвращенные предыдущим шагом, в функцию.

Каждый шаг в вашей функции выполняется как **отдельный HTTP-запрос**. Любая недетерминированная логика (например, вызовы БД или API) должна быть помещена внутри вызова `step.run()`, чтобы гарантировать её эффективное и правильное выполнение в рамках данной модели.

Давайте рассмотрим пример функции и разберем, как она выполняется:

```typescript
const fn = inngest.createFunction(
  { id: "import-contacts", triggers: { event: "contacts/csv.uploaded" } },
  // Обработчик функции:
  async ({ event, step }) => {
    const rows = await step.run("parse-csv", async () => {
      return await parseCsv(event.data.fileURI);
    });

    const normalizedRows = await step.run("normalize-raw-csv", async () => {
      const normalizedColumnMapping = getNormalizedColumnNames();
      return normalizeRows(rows, normalizedColumnMapping);
    });

    const results = await step.run("input-contacts", async () => {
      return await importContacts(normalizedRows);
    });

    return { results };
  }
);
```

### Первое выполнение

1. При первом вызове функции вызывается _обработчик функции_ только с переданными данными `event`.
2. Когда обнаруживается первый шаг, запускается шаг `"parse-csv"`. Так как шаг ранее не выполнялся, код шага (callback-функция) запускается, и результат фиксируется.
3. Функция не продолжает выполнение дальше этого шага. Каждый SDK использует свой метод для прерывания выполнения функции перед запуском любого другого кода в вашем обработчике.
4. Внутренне ID шага (`"parse-csv"`) хешируется как идентификатор состояния для использования в будущих выполнениях. Кроме того, в результат включается индекс шага (`0` в данном случае).
5. Результат отправляется обратно в Inngest и сохраняется в хранилище состояний функций.

### Вторичные выполнения — Мемоизация шагов

Каждый из последующих шагов использует состояние предыдущих выполнений и мемоизацию. Вот как это работает:

6. Функция запускается снова, на этот раз с данными `event` и состоянием предыдущего выполнения в формате JSON.
7. Обнаруживается следующий шаг (`"parse-csv"`).
8. Предыдущий результат находится в состоянии прошлых выполнений. Внутренне SDK использует хеш имени шага для поиска результата в данных состояния.
9. Код шага не выполняется, вместо этого SDK внедряет результат в возвращаемое значение `step.run` (в данном примере данные будут возвращены как `rows`).
10. Функция продолжает выполнение до обнаружения следующего шага (`"normalize-raw-csv"`).
11. Код шага выполняется, и результат возвращается в Inngest (по той же схеме, что и в шагах 2-5 выше).

### Обработка ошибок

Некоторые шаги могут выбрасывать ошибки или исключения во время выполнения. Вот как работает обработка ошибок при выполнении функции:

12. Если во время выполнения шага возникает ошибка (например, в `"input-contacts"`), выполнение функции прерывается, и ошибка перехватывается SDK.
13. Ошибка сериализуется и возвращается в Inngest. Количество попыток логируется, а ошибка сохраняется в хранилище состояний функций.
14. В зависимости от настроенного количества попыток для функции, может быть инициирован повтор (см.: [Обработка ошибок](/docs/guides/error-handling)):
    * Если количество попыток _не_ исчерпано, функция запускается снова с момента сбоя со всеми состояниями предыдущих шагов. Шаг выполняется повторно по тому же процессу, что описан выше (см. шаги 6-11).
    * Если количество попыток _исчерпано_, функция запускается снова с выброшенной ошибкой. Функция может перехватить и обработать ошибку по желанию (см.: [Обработка неудачного шага](/docs/guides/error-handling#handling-a-failing-step)).

## Заключение

Функции Inngest используют шаги и мемоизацию для инкрементального и надежного выполнения. Этот подход гарантирует отказоустойчивость функций и их устойчивость к сбоям. Разделяя функции на шаги, Inngest позволяет повторять и возобновлять их выполнение с момента сбоя. Такой подход делает ваш код более надежным и позволяет изящно справляться с временными ошибками.

# Версионирование и эволюция функций

Длительно работающие функции неизбежно меняются со временем. Inngest позволяет разработчикам развертывать изменения в функциях без явных маркеров версий или сложной логики миграции. В этом руководстве объясняется, как Inngest обрабатывает версионирование и какие стратегии можно использовать для безопасного развития функций.

## Как Inngest обрабатывает версионирование

Inngest использует **мемоизацию на основе шагов** и модель детерминированного выполнения. Это единообразно для всех SDK. В отличие от систем, требующих явных аннотаций версий в коде, Inngest отслеживает состояние функции через идентификаторы шагов, что позволяет изменять функции прямо во время их выполнения.

### Мемоизация на основе шагов

Каждый [шаг (step)](/docs/learn/inngest-steps) в функции имеет уникальный строковый идентификатор. При выполнении функции SDK:

1. Хеширует идентификатор шага вместе со счетчиком (что позволяет использовать шаги внутри циклов).
2. Проверяет, существует ли этот хеш в сохраненном состоянии функции.
3. Если найден, возвращает мемоизированный результат без повторного выполнения.
4. Если не найден, выполняет шаг и сохраняет результат.

Это означает, что завершенные шаги никогда не выполняются повторно, даже после новых развертываний. SDK определяет, что нужно запустить, на основе идентификаторов шагов в вашем коде, а не номеров версий.

Более подробную информацию об этой модели выполнения см. в разделе [Как выполняются функции Inngest](/docs/learn/how-functions-are-executed).

### Детерминизм по умолчанию

SDK по умолчанию обрабатывает изменения **бесшовно**:

- **Новые шаги выполняются при обнаружении** — если вы добавите шаг в функцию, выполняющиеся запуски (in-progress runs) выполнят его, когда столкнутся с ним.
- **Предупреждения вместо сбоев** — если порядок выполнения шагов меняется, SDK записывает предупреждение в лог вместо того, чтобы прервать функцию ошибкой.

Этот подход позволяет расширять и улучшать функции со временем, не беспокоясь о сбоях уже запущенных процессов.

## Развитие функций со временем

Существует несколько стратегий развития функций в зависимости от типа вносимых изменений.

### Добавление новых шагов

Добавление новых шагов в функцию обычно безопасно. Новые шаги будут выполнены, когда выполняющиеся запуски обнаружат их. Это полезно для добавления логирования, аналитики, уведомлений или другой дополнительной функциональности.

<CodeGroup>
```ts {{ title: "До" }}
inngest.createFunction(
  { id: "user-signup", triggers: { event: "user/created" } },
  async ({ event, step }) => {
    await step.run("send-welcome-email", async () => {
      return sendWelcomeEmail(event.data.email);
    });

    await step.run("sync-to-crm", async () => {
      return crm.contacts.create(event.data);
    });

    await step.run("schedule-followup", async () => {
      return scheduleFollowupEmail(event.data.userId, "3 days");
    });
  }
);
```
</CodeGroup>

<CodeGroup>
```ts {{ title: "После: Добавлена аналитика" }}
inngest.createFunction(
  { id: "user-signup", triggers: { event: "user/created" } },
  async ({ event, step }) => {
    await step.run("send-welcome-email", async () => {
      return sendWelcomeEmail(event.data.email);
    });

    // Новый шаг - выполнится при обнаружении, даже если последующие шаги уже завершены
    await step.run("track-signup-analytics", async () => {
      return analytics.track("user_signup_complete", {
        userId: event.data.userId,
      });
    });

    await step.run("sync-to-crm", async () => {
      return crm.contacts.create(event.data);
    });

    await step.run("schedule-followup", async () => {
      return scheduleFollowupEmail(event.data.userId, "3 days");
    });
  }
);
```
</CodeGroup>

При развертывании этого изменения запущенная функция, которая уже завершила `send-welcome-email` и `sync-to-crm`:
1. Пропустит `send-welcome-email` (мемоизировано).
2. Выполнит `track-signup-analytics` (новый шаг).
3. Пропустит `sync-to-crm`, если он уже был завершен (мемоизировано).
4. Выполнит `schedule-followup` (новый шаг).

<Callout variant="warning">
  Новые шаги не должны зависеть от данных из шагов, которые еще не были выполнены. В примере выше `track-signup-analytics` использует только данные из инициирующего события, поэтому его безопасно добавлять в любой момент.
</Callout>

### Изменение существующих шагов

**Изменение логики шага с тем же ID** безопасно. Если вы измените код внутри шага, но сохраните прежний ID, выполняющиеся запуски, которые уже завершили этот шаг, будут использовать мемоизированный результат. Новые запуски выполнят обновленную логику.

**Изменение ID шага** принудительно вызывает повторное выполнение. Если вам нужно перезапустить шаг с новой логикой для уже запущенных процессов, измените ID шага:

<CodeGroup>
```ts {{ title: "До" }}
const score = await step.run("calculate-risk-score", async () => {
  return calculateRiskScore(user.profile);
});
```
</CodeGroup>

<CodeGroup>
```ts {{ title: "После: Изменен ID для принудительного перезапуска" }}
// Изменение ID означает, что запущенные функции выполнят расчет заново,
// даже если они уже запускали "calculate-risk-score"
const score = await step.run("calculate-risk-score-v2", async () => {
  return calculateRiskScoreWithNewModel(user.profile);
});
```
</CodeGroup>

<Callout variant="info">
  SDK выводит предупреждение при изменении порядка выполнения шагов. Это ожидаемое поведение при намеренном принудительном перезапуске.
</Callout>

### Удаление шагов

Когда вы удаляете шаг, запущенные процессы, которые уже завершили его, продолжат работу в обычном режиме. Мемоизированные данные для удаленного шага остаются в состоянии, но просто игнорируются.

### Изменение порядка шагов

Изменение порядка шагов вызывает предупреждение, так как порядок выполнения отличается от сохраненного состояния. SDK обрабатывает это корректно — мемоизированные шаги возвращают свои сохраненные результаты независимо от их позиции в коде, а новые шаги выполняются при их обнаружении.

## Крупные изменения логики

Для полной переработки, где новая логика несовместима с запущенными процессами, используйте **паттерн новой функции** с маршрутизацией на основе меток времени.

Этот паттерн использует две функции, подписанные на одно и то же событие, с выражениями [`if`](/docs/reference/functions/create#trigger) для маршрутизации событий по времени:

<CodeGroup>
```ts {{ title: "Оригинальная функция с фильтром по времени" }}
// Обработка событий ДО момента перехода
const CUTOVER_TS = 1704067200000; // 1 января 2024 00:00:00 UTC

export const processUploadV1 = inngest.createFunction(
  {
    id: "process-upload",
    triggers: {
      event: "file/uploaded",
      if: `event.ts < ${CUTOVER_TS}`,
    },
  },
  async ({ event, step }) => {
    // Оригинальная логика - продолжается для запущенных процессов
    await step.run("process-file", async () => {
      return legacyProcessor(event.data.fileId);
    });
  }
);
```
</CodeGroup>

<CodeGroup>
```ts {{ title: "Новая функция для событий после перехода" }}
// Обработка событий ПОСЛЕ момента перехода
const CUTOVER_TS = 1704067200000;

export const processUploadV2 = inngest.createFunction(
  {
    id: "process-upload-v2",
    triggers: {
      event: "file/uploaded",
      if: `event.ts >= ${CUTOVER_TS}`,
    },
  },
  async ({ event, step }) => {
    // Полностью переписанный рабочий процесс
    const metadata = await step.run("extract-metadata", async () => {
      return extractMetadata(event.data.fileId);
    });

    await step.run("validate-content", async () => {
      return validateContent(metadata);
    });

    await step.run("process-modern", async () => {
      return modernProcessor(event.data.fileId, metadata);
    });
  }
);
```
</CodeGroup>

Этот подход гарантирует:
- Запущенные процессы завершаются по старой логике.
- Новые события запускают обновленную функцию.
- Отсутствие потери данных или непредвиденных сбоев.

<Callout variant="info">
  Это создает новую функцию в вашей панели управления Inngest. Как только все запуски v1 завершатся, вы сможете удалить старую функцию.
</Callout>

### Использование версий событий

Вы также можете использовать поле версии `v` [формата события](/docs/features/events-triggers/event-format#event-payload-format) в сочетании с паттерном `if` выше:

```typescript
await inngest.send({
  name: "file/uploaded",
  data: {/* ... */},
  v: "2026-03-11" // Используйте версию для маршрутизации к нужной функции
})
```

```typescript
// v1
export const processUploadV1 = inngest.createFunction(
  {
    id: "process-upload",
    triggers: {
      event: "file/uploaded",
      if: `event.v != "2026-03-11"`, // или если вы уже используете версии, укажите это явно
    },
  },
  //...
});
// v2
export const processUploadV2 = inngest.createFunction(
  {
    id: "process-upload-v2",
    triggers: {
      event: "file/uploaded",
      if: `event.v == "2026-03-11"`,
    },
  },
  //...
});
```

## Лучшие практики

### Именование ID шагов

Выбирайте идентификаторы шагов, которые являются:

- **Описательными**: `"charge-customer-payment"`, а не `"step-1"`.
- **Стабильными**: избегайте ID, которые содержат значения, способные измениться.
- **Уникальными**: каждый шаг в функции должен иметь свой уникальный ID.

```ts
// Хорошо - описательно и стабильно
await step.run("send-order-confirmation", async () => { ... });

// Плохо - слишком общее, возможны конфликты
await step.run("send", async () => { ... });
```

### Тестирование изменений версий локально

Используйте [Inngest Dev Server](/docs/local-development) для проверки того, как изменения влияют на запущенные процессы:

1. Запустите функцию с `step.sleep()`, чтобы приостановить выполнение.
2. Измените код функции, пока она "спит".
3. Наблюдайте, как функция обрабатывает изменения после возобновления.

Это поможет вам понять влияние изменений перед развертыванием в промышленную эксплуатацию.

# Обработка идемпотентности

Обеспечение идемпотентности кода является основой создания надежных систем. В Inngest существует несколько способов гарантировать идемпотентность ваших функций.

## Что такое идемпотентность?

Идемпотентность, по определению, описывает операцию, которая может выполняться несколько раз без изменения результата после первоначального выполнения. В мире программного обеспечения это означает, что функция может быть выполнена многократно, но она всегда будет иметь тот же эффект, что и при однократном вызове. Примером этого является операция «upsert» (обновление или вставка).

## Как обрабатывать идемпотентность в Inngest

Основной целью всегда должно быть написание кода, который сам по себе является идемпотентным внутри вашей системы или функций Inngest, но в Inngest также есть функции, которые помогут вам обеспечить идемпотентность.

Поскольку функции Inngest запускаются событиями, существует два основных способа обеспечения идемпотентности:

* [на уровне события (_сторона производителя_)](#at-the-event-level-the-producer) и/или
* [на уровне функции (_сторона потребителя_)](#at-the-function-level-the-consumer)

## На уровне события (сторона производителя)

Каждое событие, полученное Inngest, запускает все функции с соответствующим триггером. Если событие отправлено дважды, Inngest запустит функцию дважды. Это поведение по умолчанию, так как Inngest не знает, является ли событие тем же самым или новым.

<Callout>
**Пример:** В интернет-магазине пользователь может добавить одну и ту же футболку в корзину дважды, потому что хочет купить две (_2 уникальных события_). Тот же пользователь может перейти к оплате всех товаров в корзине, но нажать кнопку «оплатить» дважды (_2 дублирующих события_).
</Callout>

Чтобы предотвратить повторную обработку события, вы можете установить уникальный `id` события при [его отправке](/docs/reference/events/send#inngest-send-event-payload-event-payload-promise). Этот `id` действует как ключ идемпотентности **в течение 24-часового периода**, и Inngest проверит, было ли это событие уже получено, прежде чем запускать функцию снова.


<CodeGroup>

```ts
const cartId = 'CGo5Q5ekAxilN92d27asEoDO';
await inngest.send({
  id: `checkout-completed-${cartId}`, // <-- Это ключ идемпотентности
  name: 'cart/checkout.completed',
  data: {
    email: 'taylor@example.com',
    cartId: cartId
  }
})
```


```go {{ title: "Go" }}
cart_id := "CGo5Q5ekAxilN92d27asEoDO"
inngest.Send(context.Background(), inngestgo.Event{
  ID: fmt.Sprintf("checkout-completed-%s", cart_id), // <-- Это ключ идемпотентности
  Name: "cart/checkout.completed",
  Data: map[string]any{"email": "taylor@example.com", "cart_id": cart_id},
})
```


```python {{ title: "Python" }}
cart_id = 'CGo5Q5ekAxilN92d27asEoDO'
await inngest.send({
  id: f'checkout-completed-{cart_id}', // <-- Это ключ идемпотентности
  name: 'cart/checkout.completed',
  data: {
    email: 'taylor@example.com',
    cart_id: cart_id
  }
})
```

</CodeGroup>

| Event ID | Метка времени | Функция |
| -------- | --------- | -------- |
| `checkout-completed-CGo5Q5ekAxilN92d27asEoDO` | 08:00:00.000 | ✅ Функции запущены |
| `checkout-completed-CGo5Q5ekAxilN92d27asEoDO` | 08:00:00.248 | ❌ Ничего не запущено |

Как видно из приведенного выше примера, установка `id` позволяет предотвратить дублирование выполнения на стороне производителя, где возникает событие.

Другие важные моменты:

* ID событий используются для предотвращения дублирования только в течение 24 часов. По истечении 24 часов событие будет рассматриваться как новое и снова активирует соответствующие функции.
* Inngest сохранит второе событие, и оно будет видно в истории событий, но оно _не_ запустит никаких функций.
* События, которые разветвляются (fan-out) на несколько функций, запустят каждую функцию в обычном режиме.

  **Подсказка** — Если вы используете [трансформации вебхуков](/docs/platform/webhooks#defining-a-transform-function) Inngest, вы можете установить `id` в трансформации, чтобы обеспечить идемпотентность события.

<Callout variant="warning">
  Идемпотентность событий игнорируется некоторыми функциями:
  - Debouncing (удаление дребезга)
  - Пакетная обработка событий (batching)
  - Приостановка функций. Пока функция приостановлена, идемпотентность событий игнорируется. Таким образом, если повтор (replay) создается после возобновления, он может иметь «пропущенные» запуски, которые игнорировали идемпотентность событий.
</Callout>

## На уровне функции (сторона потребителя)

Возможно, вы предпочтете обеспечивать идемпотентность на уровне функции или не сможете контролировать отправляемое событие (например, если оно поступает через вебхук). Параметр конфигурации [функции `idempotency`](/docs/reference/functions/create#inngest-create-function-configuration-trigger-handler-inngest-function) позволяет сделать это.

Ключ `idempotency` каждой функции определяется как [выражение CEL](/docs/guides/writing-expressions), которое вычисляется на основе данных полезной нагрузки события. Выражение используется для генерации уникального строкового ключа, который идемпотентно предотвращает повторное выполнение функции.

Каждое уникальное выражение активирует только одно выполнение функции **в течение 24-часового периода**. По истечении 24 часов новое событие, генерирующее то же самое уникальное выражение, снова запустит выполнение функции.

### Пример

Воспользуемся тем же примером интернет-магазина. У нас есть событие без установленного `id` ([см. выше](#at-the-event-level-the-producer)), но мы хотим гарантировать, что функция `send-checkout-email` будет запущена только один раз для каждого `cartId`, чтобы предотвратить отправку дубликатов писем.

<Row>
  <Col>

```json {{ title: "Полезная нагрузка события"}}

  "data": {
    "email": "blake@example.com",
    "cartId": "s6CIMNqIaxt503I1gVEICfwp"
  },
  "ts": 1703275661157
}
```

  </Col>
  <Col>

```ts {{ title: "Определение функции с ключом идемпотентности"}}
export const sendEmail = inngest.createFunction(
  {
    id: 'send-checkout-email',
    // Это ключ идемпотентности
    idempotency: 'event.data.cartId',
    // Вычисляется в: "s6CIMNqIaxt503I1gVEICfwp"
    // для данной полезной нагрузки события
    triggers: { event: 'cart/checkout.completed' },
  },
  async  ({ event, step }) => { /* ... */ }
})
```

### Написание выражений CEL

Хотя CEL позволяет делать многое, мы сосредоточимся на том, как использовать его для генерации уникального строкового ключа для идемпотентности. Основные моменты:

* Вы можете получить доступ к любым данным полезной нагрузки события, используя переменную `event` и точечную нотацию для вложенных свойств.
* Вы можете использовать оператор `+` для объединения строк.

Объединение двух или более свойств — хороший способ обеспечить необходимый уровень уникальности. Вот пара примеров:

* **Регистрация пользователя:** Вы хотите отправить приветственное письмо только один раз для каждого пользователя, поэтому вы устанавливаете `idempotency` в `event.data.userId` на случай, если ваш API отправит дубликаты событий.
* **Приглашение в команду организации:** Пользователь может быть частью нескольких организаций в вашем приложении. Вы хотите отправить письмо с приглашением в команду только один раз для каждой комбинации пользователя и организации, поэтому вы устанавливаете `idempotency` в `event.data.userId + "-" + event.data.organizationId`.

Для получения дополнительной информации о написании выражений CEL прочтите [наше руководство](/docs/guides/writing-expressions).

<Callout>
  💡 Если вы хотите контролировать выполнение функции в течение определенного периода времени, вам могут подойти другие инструменты:

  * [`rateLimit`](/docs/reference/functions/rate-limit) — Ограничение количества выполнений функций за период времени.
  * [`debounce`](/docs/reference/functions/debounce) — Задержка выполнения функции для дублирующихся событий в течение периода времени.
</Callout>

### Ключи идемпотентности и fan-out (разветвление)

{/* TODO - Это должно ссылаться на будущую итерацию руководства по fan-out: 1 событие, n функций */}
Одной из причин использования `idempotency` на уровне функции является наличие `event`, которое разветвляется на несколько функций. Рассмотрим следующий пример:

| Функция | Триггер события | Как часто |
| -------- | ------------- | --------- |
| Отслеживание запросов | `ai/generation.requested` | Каждый раз |
| Запуск генерации | `ai/generation.requested` | Один раз на запрос |

В этом случае вы захотите установить `idempotency` для функции «Запуск генерации», чтобы она запускалась один раз, например, для каждого уникального отправленного промпта. Это нужно, чтобы не запускать один и тот же промпт повторно и не тратить вычислительные ресурсы/кредиты. Однако вы по-прежнему можете захотеть отслеживать количество запросов, отправленных каждым пользователем, поэтому для функции «Отслеживание запросов» идемпотентность устанавливать не нужно. Код для обеих функций приведен ниже.

<details>
  <summary>**Просмотреть код функций**</summary>

  Обе функции используют один и тот же триггер события `ai/generation.requested`, который содержит `promptHash` и `userId` в полезной нагрузке события.

```ts {{ title: "Функция отслеживания запросов" }}
const trackRequests = inngest.createFunction(
  { id: 'track-requests', triggers: { event: 'ai/generation.requested' } },
  async ({ event, step }) => {
    // Отслеживание запроса
  }
)
```

```ts {{ title: "Функция запуска генерации" }}
const runGeneration = inngest.createFunction(
  {
    id: 'run-generation',
    // Учитывая, что полезная нагрузка события содержит хэш промпта,
    // эта функция запустится только один раз для уникального промпта на пользователя
    // каждые 24 часа:
    idempotency: `event.data.promptHash + "-" + event.data.userId`,
    triggers: { event: 'ai/generation.requested' },
  },
  async ({ event, step }) => {
    // Выполнение генерации
  }
)
```

</details>

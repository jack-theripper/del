# Ограничение частоты (Rate limiting)

Ограничение частоты — это _жесткий лимит_ на количество запусков функции, которые могут начаться в течение заданного периода времени. События, превышающие лимит, _пропускаются_ и не запускают функцию. Это предотвращает избыточные запуски функций за определенный период. Некоторые варианты использования:

* Защита системы от злоупотреблений.
* Снижение частоты функций синхронизации данных.
* Пропуск шумных или дублирующихся событий [вебхуков](/docs/platform/webhooks).

## Как настроить ограничение частоты

<CodeGroup>
```ts {{ title: "TypeScript" }}
export default inngest.createFunction(
  {
    id: "synchronize-data",
    rateLimit: {
      limit: 1,
      period: "4h",
      key: "event.data.company_id",
    },
    triggers: { event: "intercom/company.updated" },
  },
  async ({ event, step }) => {
    // Функция будет ограничена по частоте
    // Она запустится не чаще одного раза в 4 часа для данного company_id в полезной нагрузке события
  }
);
```

```go {{ title: "Go" }}
inngestgo.CreateFunction(
	client,
	inngestgo.FunctionOpts{
		ID: "synchronize-data",
		RateLimit: &inngestgo.ConfigRateLimit{
			Limit:  1,
			Period: 4 * time.Hour,
			Key:    inngestgo.StrPtr("event.data.company_id"),
		},
	},
	inngestgo.EventTrigger("intercom/company.updated", nil),
	func(ctx context.Context, input inngestgo.Input[map[string]any]) (any, error) {
		// Функция будет ограничена до 1 запуска за 4 часа для заданного company_id
		return nil, nil
	},
)
```

```py {{ title: "Python" }}
@inngest.create_function(
  id="synchronize-data",
  rate_limit=inngest.RateLimit(
    limit=1,
    period=datetime.timedelta(hours=4),
    key="event.data.company_id",
  ),
  trigger=inngest.Trigger(event="intercom/company.updated")
)
async def synchronize_data(ctx: inngest.Context):
    # Функция будет ограничена до 1 запуска за 4 часа для заданного company_id
```

</CodeGroup>

### Справочник по конфигурации

* `limit` — максимальное количество запусков функции в заданный период времени.
* `period` — период времени, на который устанавливается лимит. Период начинается, когда получено первое подходящее событие.
* `key` — необязательное [выражение](/docs/guides/writing-expressions), использующее данные события для применения лимита. Каждое уникальное значение `key` имеет свой собственный лимит, что позволяет ограничивать частоту запусков функций по конкретному ключу (например, ID пользователя).

<Callout>
  Любые события сверх вашего `limit` игнорируются. Это означает, что данный подход не подходит, если вам необходимо обработать абсолютно каждое событие, отправленное в Inngest. Рассмотрите вместо этого [дросселирование (throttling)](/docs/guides/throttling).
</Callout>

## Как это работает

Каждый раз, когда поступает событие, соответствующее триггеру вашей функции, перед выполнением функции оно сначала проверяется. Если настроен `rateLimit`, Inngest использует параметры `limit` и `period`, чтобы запускать не более заданного количества функций в течение этого периода.

Реализация ограничения частоты в Inngest использует [Generic Cell Rate Algorithm (GCRA)](https://en.wikipedia.org/wiki/Generic_cell_rate_algorithm). Упрощенно: Inngest использует `limit` и `period`, чтобы создать «окна» времени (buckets), в каждом из которых ваша функция может выполниться _один раз_.

```
limit / period = размер временного окна (bucket)
```

Например, при `limit: 10` и `period: '60m'` (60 минут) размер окна составит 6 минут. Любое событие, запускающее функцию, «заполняет» окно на этот интервал, и дополнительные события игнорируются, пока окно не обновится. Алгоритм (GCRA) сложнее, но базовая идея такова — `rateLimit` гарантирует, что вы выполните максимум `limit` запусков за указанный `period`.

<Callout>
  События, проигнорированные функцией, продолжат храниться в Inngest.
</Callout>

### Использование ключа (`key`)

При добавлении `key` отдельный лимит применяется к каждому уникальному значению выражения `key`. Например, если ваш `key` установлен в `event.data.customer_id`, для каждого клиента будет применяться свой собственный лимит. Это означает, что разные пользователи могут запускать одну и ту же функцию в одном и том же временном окне, но два запуска не произойдут для одного и того же `event.data.customer_id`. Подробнее — в [руководстве по написанию выражений](/docs/guides/writing-expressions).

**Примечание.** Чтобы предотвратить повторный запуск функции одним и тем же дублирующим событием чаще, чем раз в 24 часа, используйте [опцию `idempotency`](/docs/guides/handling-idempotency#at-the-function-level-the-consumer), что эквивалентно `rateLimit` с `key`, `limit: 1` и `period: '24hr'`.

## Ограничения

* Максимальное значение `period` — 24 часа.

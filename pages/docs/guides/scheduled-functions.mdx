# Cron-задачи (Запланированные функции)

Вы можете создавать запланированные задачи с помощью cron-расписаний непосредственно в Inngest. Cron-расписания Inngest также поддерживают часовые пояса, что позволяет запускать задачи в любом нужном вам часовом поясе.


<LanguageSelector>

<LanguageSection show="typescript">

Вы можете создавать запланированные функции, работающие в любом часовом поясе, используя [`createFunction()`](/docs/reference/functions/create) из SDK:

```ts
import { Inngest } from "inngest";

const inngest = new Inngest({ id: "signup-flow" });

// Эта функция еженедельного дайджеста будет запускаться в 12:00 в пятницу по парижскому времени
export const prepareWeeklyDigest = inngest.createFunction(
  { id: "prepare-weekly-digest", triggers: { cron: "TZ=Europe/Paris 0 12 * * 5" } },
  async ({ step }) => {
    // Загружаем всех пользователей из базы данных:
    const users = await step.run(
      "load-users",
      async () => await db.load("SELECT * FROM users")
    );

    // 💡 Поскольку мы хотим отправить еженедельный дайджест каждому пользователю,
    // перебор всех пользователей и отправка писем может занять много времени.

    // Вместо этого мы используем эту запланированную функцию для отправки события в Inngest
    // для каждого пользователя, а затем обрабатываем фактическую отправку письма
    // в отдельной функции, запускаемой этим событием.

    // ✨ Это называется паттерном «fan-out» ✨

    // 1️⃣ Сначала создаём объект события для каждого пользователя из запроса:
    const events = users.map((user) => {
      return {
        name: "app/send.weekly.digest",
        data: {
          user_id: user.id,
          email: user.email,
        },
      };
    });

    // 2️⃣ Теперь отправляем все события одним пакетом:
    await step.sendEvent("send-digest-events", events);

    // Эта функция может быстро завершиться, а вся остальная логика будет
    // обработана в функции ниже ⬇️
  }
);

// Это обычная функция Inngest, которая будет отправлять фактическое письмо
// для каждого полученного события (см. inngest.send() в функции выше)

// Поскольку мы «распределяем» задачи через события, эти функции могут выполняться параллельно
export const sendWeeklyDigest = inngest.createFunction(
  { id: "send-weekly-digest-email", triggers: { event: "app/send.weekly.digest" } },
  async ({ event }) => {
    // 3️⃣ Получаем email и id пользователя из полезной нагрузки события
    const { email, user_id } = event.data;

    // 4️⃣ Наконец, отправляем само письмо:
    await email.send("weekly_digest", email, user_id);

    // 🎇 Вот и всё! — Мы использовали две функции для надёжного выполнения
    // запланированной задачи для большого списка пользователей!
  }
);
```

<LanguageSection show="go">

Вы можете создавать запланированные функции, работающие в любом часовом поясе, используя [`CreateFunction()`](https://pkg.go.dev/github.com/inngest/inngestgo#CreateFunction) из SDK:

```go
package main

import (
	"context"

	"github.com/inngest/inngestgo"
	"github.com/inngest/inngestgo/step"
)

func init() {
	// Эта функция еженедельного дайджеста будет запускаться в 12:00 в пятницу по парижскому времени
	inngestgo.CreateFunction(
		client,
		inngestgo.FunctionOpts{Name: "prepare-weekly-digest"},
		inngestgo.CronTrigger("TZ=Europe/Paris 0 12 * * 5"),
		func(ctx context.Context, input inngestgo.Input[map[string]any]) (any, error) {
			// Загружаем всех пользователей из базы данных:
			users, err := step.Run(ctx, "load-users", func(ctx context.Context) ([]*User, error) {
				return loadUsers()
			})
			if err != nil {
				return nil, err
			}

			// 💡 Поскольку мы хотим отправить еженедельный дайджест каждому пользователю,
			// перебор всех пользователей и отправка писем может занять много времени.

			// Вместо этого мы используем эту запланированную функцию для отправки события в Inngest
			// для каждого пользователя, а затем обрабатываем фактическую отправку письма
			// в отдельной функции, запускаемой этим событием.

			// ✨ Это называется паттерном «fan-out» ✨

			// 1️⃣ Сначала создаём объект события для каждого пользователя из запроса:
			events := make([]inngestgo.Event, len(users))
			for i, user := range users {
				events[i] = inngestgo.Event{
					Name: "app/send.weekly.digest",
					Data: map[string]interface{}{
						"user_id": user.ID,
						"email":   user.Email,
					},
				}
			}

			// 2️⃣ Теперь отправляем все события одним пакетом:
			err = step.SendMany(ctx, "send-digest-events", events)
			if err != nil {
				return nil, err
			}

			// Эта функция может быстро завершиться, а вся остальная логика будет
			// обработана в функции ниже ⬇️
			return nil, nil
		},
	)

	// Это обычная функция Inngest, которая будет отправлять фактическое письмо
	// для каждого полученного события (см. inngest.send() в функции выше)

	// Поскольку мы «распределяем» задачи через события, эти функции могут выполняться параллельно
	inngestgo.CreateFunction(
		client,
		inngestgo.FunctionOpts{Name: "send-weekly-digest-email"},
		inngestgo.EventTrigger("app/send.weekly.digest", nil),
		func(ctx context.Context, input inngestgo.Input[map[string]any]) (any, error) {
			// 3️⃣ Получаем email и id пользователя из полезной нагрузки события
			email := input.Event.Data["email"].(string)
			userID := input.Event.Data["user_id"].(string)

			// 4️⃣ Наконец, отправляем само письмо:
			err := email.Send("weekly_digest", email, userID)
			if err != nil {
				return nil, err
			}

			// 🎇 Вот и всё! — Мы использовали две функции для надёжного выполнения
			// запланированной задачи для большого списка пользователей!
			return nil, nil
		},
	)
}
```

Вы можете создавать запланированные функции, работающие в любом часовом поясе, используя [`create_function()`](/docs/reference/python/functions/create) из SDK:

```py

inngest_client = Inngest(app_id="signup-flow")

# Эта функция еженедельного дайджеста будет запускаться в 12:00 в пятницу по парижскому времени
@inngest_client.create_function(
    fn_id="prepare-weekly-digest",
    trigger=inngest.TriggerCron(cron="TZ=Europe/Paris 0 12 * * 5")
)
async def prepare_weekly_digest(ctx: inngest.Context) -> None:
    # Загружаем всех пользователей из базы данных:
    users = await ctx.step.run(
        "load-users",
        lambda: db.load("SELECT * FROM users")
    )

    # 💡 Поскольку мы хотим отправить еженедельный дайджест каждому пользователю,
    # перебор всех пользователей и отправка писем может занять много времени.

    # Вместо этого мы используем эту запланированную функцию для отправки события в Inngest
    # для каждого пользователя, а затем обрабатываем фактическую отправку письма
    # в отдельной функции, запускаемой этим событием.

    # ✨ Это называется паттерном «fan-out» ✨

    # 1️⃣ Сначала создаём объект события для каждого пользователя из запроса:
    events = [
        {
            "name": "app/send.weekly.digest",
            "data": {
                "user_id": user.id,
                "email": user.email,
            }
        }
        for user in users
    ]

    # 2️⃣ Теперь отправляем все события одним пакетом:
    await ctx.step.send_event("send-digest-events", events)

    # Эта функция может быстро завершиться, а вся остальная логика будет
    # обработана в функции ниже ⬇️


# Это обычная функция Inngest, которая будет отправлять фактическое письмо
# для каждого полученного события (см. inngest.send() в функции выше)

# Поскольку мы «распределяем» задачи через события, эти функции могут выполняться параллельно
@inngest_client.create_function(
    fn_id="send-weekly-digest-email",
    trigger=inngest.TriggerEvent(event="app/send.weekly.digest")
)
async def send_weekly_digest(ctx: inngest.Context) -> None:

    email = ctx.event.data["email"]
    user_id = ctx.event.data["user_id"]

    # 4️⃣ Наконец, отправляем само письмо:
    await email.send("weekly_digest", email, user_id)

    # 🎇 Вот и всё! — Мы использовали две функции для надёжного выполнения
    # запланированной задачи для большого списка пользователей!
```

На бесплатном плане, если ваша функция завершается с ошибкой 20 раз подряд, она будет автоматически приостановлена.

<Callout>
⚠️ **Предупреждение о переходе на летнее/зимнее время (DST):** Расписания вблизи времени перехода на летнее/зимнее время могут вести себя непредсказуемо в местных часовых поясах. В зависимости от часового пояса и точного расписания cron может выполниться ноль, один или два раза в день при переводе часов.

Поведение cron в Inngest следует базовой библиотеке cron и не применяет специальную коррекцию для DST. Чтобы снизить риск, избегайте расписаний в час перехода (например, `2:00 AM` во многих регионах США или `12:00 AM` в некоторых других регионах) и предпочитайте `TZ=UTC`, когда вам нужна стабильная синхронизация выполнения.
</Callout>

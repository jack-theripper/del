# Параллелизм шагов

- При использовании бессерверной платформы код будет выполняться в режиме истинного параллелизма, аналогично многопоточности (без общего состояния)
- Каждый шаг будет повторяться независимо

### Поддержка платформ

**Параллелизм работает на всех провайдерах и платформах.** Истинный параллелизм поддерживается для бессерверных функций; если вы используете одиночный Express-сервер, все параллельные задачи будут выполняться в рамках однопоточного Node-сервера.

## Запуск шагов параллельно

Вы можете запускать шаги параллельно через `Promise.all()`:

- Создайте каждый шаг с помощью [`step.run()`](/docs/reference/functions/step-run) без `await` — это вернёт неразрешённый промис.
- Дождитесь всех шагов через `Promise.all()`. Это запускает все шаги параллельно в отдельных выполнениях.

Типичный сценарий использования — разбиение работы на части:

<CodeGroup>
```ts
import { Inngest } from "inngest";

const inngest = new Inngest({ id: "signup-flow" });

export const fn = inngest.createFunction(
  { id: "post-payment-flow", triggers: { event: "stripe/charge.created" } },
  async ({ event, step }) => {
    // Эти шаги не ожидаются через `await` и выполняются параллельно,
    // когда вызывается Promise.all.
    const sendEmail = step.run("confirmation-email", async () => {
      const emailID = await sendEmail(event.data.email);
      return emailID;
    });

    const updateUser = step.run("update-user", async () => {
      return db.updateUserWithCharge(event);
    });

    // Запускаем оба шага параллельно. После завершения Promise.all вернёт
    // все параллельные состояния здесь.
    //
    // Это гарантирует, что все шаги завершатся как можно быстрее, и мы
    // всё равно будем иметь доступ к данным каждого шага после их завершения.
    const [emailID, updates] = await Promise.all([sendEmail, updateUser]);

    return { emailID, updates };
  }
);
```
</CodeGroup>

После завершения каждого шага Inngest агрегирует состояние каждого шага и повторно вызывает функцию со всеми доступными состояниями.

### Параллелизм шагов в Python

Inngest поддерживает параллельные шаги вне зависимости от того, используете ли вы асинхронный или синхронный код. В обоих случаях можно использовать `step.parallel`:

#### async — с `inngest.Step` и `await ctx.group.parallel()`
```py
@client.create_function(
  fn_id="my-fn",
  trigger=inngest.TriggerEvent(event="my-event"),
)
async def fn(ctx: inngest.Context) -> None:
  user_id = ctx.event.data["user_id"]

  (updated_user, sent_email) = await ctx.group.parallel(
    (
      lambda: step.run("update-user", update_user, user_id),
      lambda: step.run("send-email", send_email, user_id),
    )
  )
```

#### sync — с `inngest.StepSync` и `group.parallel()`
```py
@client.create_function(
  fn_id="my-fn",
  trigger=inngest.TriggerEvent(event="my-event"),
)
def fn(ctx: inngest.ContextSync) -> None:
  user_id = ctx.event.data["user_id"]

  (updated_user, sent_email) = ctx.group.parallel(
    (
      lambda: ctx.step.run("update-user", update_user, user_id),
      lambda: ctx.step.run("send-email", send_email, user_id),
    )
  )
```

## Оптимизация производительности параллельных шагов

Без оптимизированного параллелизма параллельные шаги требуют 2 HTTP-запроса на шаг к вашему приложению. При большом количестве параллельных шагов (например, сотнях) это может привести к:
- Большому количеству HTTP-запросов к вашему приложению
- Увеличению входящего трафика
- Более высокой нагрузке на CPU из-за парсинга запросов

Оптимизированный параллелизм сокращает это до 1 запроса на параллельный шаг, что значительно улучшает производительность для функций с большим количеством параллельных операций.

### TypeScript

Вы можете отключить оптимизированный параллелизм, установив `optimizeParallelism: false` на клиенте или функции.


**Важные замечания:**

- **Поведение `Promise.race`**: При оптимизированном параллелизме (по умолчанию в v4) `Promise.race` ожидает завершения всех параллельных шагов перед разрешением. Если вам нужно поведение с ранним разрешением, используйте `group.parallel()`:

```ts
const winner = await group.parallel(async () => {
  return Promise.race([
    step.run("a", () => "a"),
    step.run("b", () => "b"),
  ]);
});
```

- **Последовательные шаги в параллельных группах**: Шаги, выполняющиеся последовательно внутри разных параллельных ветвей, могут выполняться не в ожидаемом порядке. Например:

```ts
const stepOrder: string[] = [];

export const fn = inngest.createFunction(
  { id: "fn-1", triggers: { event: "event-1" } },
  async ({ step }) => {
    await Promise.all([
      (async () => {
        await step.run("fast.1", async () => {
          stepOrder.push("fast.1");
        });
        await step.run("fast.2", async () => {
          stepOrder.push("fast.2");
        });
      })(),
      (async () => {
        await step.run("slow.1", async () => {
          await sleep(1000);
          stepOrder.push("slow.1");
        });
        await step.run("slow.2", async () => {
          await sleep(1000);
          stepOrder.push("slow.2");
        });
      })(),
    ]);

    // С optimizeParallelism: ['fast.1', 'slow.1', 'fast.2', 'slow.2']
    // Без optimizeParallelism: ['fast.1', 'fast.2', 'slow.1', 'slow.2']
  }
);
```

### Python: оптимизация включена по умолчанию с возможностью отключения

Python всегда использует оптимизированный параллелизм по умолчанию, так как у него нет аналога `Promise.race`. Однако вы можете отключить его на уровне группы, если вам нужно, чтобы последовательные шаги внутри параллельных групп выполнялись независимо.

Используйте параметр `parallel_mode` для управления этим поведением:

```py
import inngest
import asyncio

@inngest_client.create_function(
    fn_id="my-fn",
    trigger=inngest.TriggerEvent(event="my-event"),
)
async def fn(ctx: inngest.Context) -> None:
    async def fast_group() -> None:
        await ctx.step.run("a", lambda: asyncio.sleep(1))
        await ctx.step.run("b", lambda: asyncio.sleep(1))

    async def slow_group() -> None:
        await ctx.step.run("x", lambda: asyncio.sleep(10))
        await ctx.step.run("y", lambda: asyncio.sleep(10))

    # Использование режима RACE заставляет шаги выполняться в ожидаемом порядке: a, b, x, y
    await ctx.group.parallel(
        (fast_group, slow_group),
        parallel_mode=inngest.ParallelMode.RACE
    )
```

**Без указания `parallel_mode`** шаги будут выполняться в порядке: `a`, `x`, `b`, `y` (оптимизированный режим). Всё остаётся корректным, но шаг `b` не запустится до завершения шага `x`.

**С `parallel_mode=inngest.ParallelMode.RACE`** шаги выполняются в ожидаемом порядке: `a`, `b`, `x`, `y`, но с накладными расходами в виде большего числа запросов.

## Разбиение задач на части

Типичный сценарий использования — разбиение работы на части. Например, при использовании API OpenAI вам может потребоваться разбить входные данные пользователя на фрагменты, запустить API для каждого фрагмента, а затем агрегировать все данные:

<CodeGroup>
```ts
import { Inngest } from "inngest";

const inngest = new Inngest({ id: "signup-flow" });

export const fn = inngest.createFunction(
  { id: "summarize-text", triggers: { event: "app/text.summarize" } },
  async ({ event, step }) => {
    const chunks = splitTextIntoChunks(event.data.text);

    const summaries = await Promise.all(
      chunks.map((chunk, index) =>
        step.run(`summarize-chunk-${index}`, () => summarizeChunk(chunk))
      )
    );

    await step.run("summarize-summaries", () => summarizeSummaries(summaries));
  }
);
```
</CodeGroup>

Это позволяет запускать множество независимых шагов, дождаться их завершения, а затем получить результаты всех шагов в нескольких строках кода. Реализация этого в традиционной системе потребовала бы создания множества задач, опроса статуса всех задач и ручного объединения состояния.

## Ограничения

В настоящее время общий объём данных, возвращённых **всеми** шагами, должен быть менее 4 МБ (например, один шаг может вернуть максимум 4 МБ, или 4 шага — максимум по 1 МБ каждый). Функции также ограничены максимум 1000 шагами.

## Параллелизм vs. fan-out

Похожая техника — fan-out ([читайте руководство здесь](/docs/guides/fan-out-jobs)): когда одна функция отправляет события для запуска других функций. Ключевые отличия:

- Оба паттерна выполняют задачи параллельно
- Вы можете получить доступ к результатам параллельных шагов внутри своей функции, тогда как при fan-out это невозможно
- Параллелизм ограничен 1000 шагами, однако с помощью fan-out можно создать столько функций, сколько нужно
- С помощью fan-out можно воспроизводить события, например для локального тестирования функций
- При необходимости можно легко повторить отдельные функции после постоянной ошибки, тогда как при постоянном сбое шага (после повторных попыток) сама функция завершится с ошибкой и остановится
- Fan-out разделяет функциональность на разные функции, тогда как использование step-функций держит всю связанную логику в одной легко читаемой функции

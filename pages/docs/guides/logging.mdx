# Логирование в Inngest

При работе с серверлесс-средами (serverless runtimes) обработка логов может иметь некоторые нюансы.

Одна из основных проблем связана с тем, как серверлесс-провайдеры завершают работу после выхода из функции.
Логгеру может не хватить времени на завершение сброса буферов (flushing), что приводит к потере логов.

Другая (противоположная) проблема связана с тем, как Inngest обрабатывает мемоизацию и выполнение кода через HTTP-вызовы к SDK.
Инструкция логирования вне функции `step` может быть выполнена несколько раз, что приведет к дублированию записей.

```ts {{ title: "example-fn.ts" }}
async ({ event, step }) => {
  console.log("что-то") // это может быть выполнено трижды

  await step.run("fn", () => {
    console.log("что-то другое") // это всегда будет выполнено один раз
  })

  await step.run(...)
}
```

Мы предоставляем легкую обертку над существующими инструментами логирования и экспортируем её в функции Inngest, чтобы смягчить эти проблемы. Таким образом, вам как пользователю не нужно беспокоиться о них, и всё должно работать так, как вы ожидаете.

## Использование

Объект `logger` доступен во всех функциях Inngest в качестве аргумента обработчика. Вы можете использовать его с любым логгером по вашему выбору. Если логгер не указан, `logger` по умолчанию будет использовать `console`.

```ts
inngest.createFunction(
  { id: "my-awesome-function", triggers: { event: "func/awesome" } },
  async ({ event, step, logger }) => { // <== logger доступен как аргумент
    logger.info("запуск функции", { metadataKey: "metadataValue" });

    const val = await step.run("do-something", () => {
      if (somethingBadHappens) logger.warn("произошло что-то плохое");
    });

    return { success: true, event };
  }
);
```

<Tip>
Мы рекомендуем использовать библиотеки логирования, поддерживающие реализацию дочерних логгеров `.child()`, которая автоматически добавляет метаданные рантайма функции в ваши логи. Подробнее об этом читайте в разделе [«Обогащенные логи с метаданными функции»](#enriched-logs-with-function-metadata).
</Tip>

Экспортируемый логгер предоставляет следующие методы интерфейса:

```ts
export interface Logger {
  info(...args: any[]): void;
  warn(...args: any[]): void;
  error(...args: any[]): void;
  debug(...args: any[]): void;
}
```

Это стандартные интерфейсы, соответствующие [рекомендациям RFC5424](https://datatracker.ietf.org/doc/html/rfc5424#section-6.2.1), поэтому большинство выбранных вами логгеров должны работать без проблем.


## Использование предпочитаемого логгера

Хотя `console.log` может быть достаточно для локальной разработки, другие библиотеки логирования предоставляют больше возможностей, подходящих для использования в продакшене.

В следующих примерах в качестве библиотеки логирования используется [winston][winston], включая базовый пример и пример с транспортом Datadog.

<CodeGroup>
```ts {{ title: "Базовый пример Winston" }}
import { Inngest } from "inngest";
import winston from "winston";

const logger = winston.createLogger({
  level: "info",
  exitOnError: false,
  format: winston.format.json(),
  transports: [new winston.transports.Console()],
});

// Передайте `logger` в клиент Inngest, и этот логгер winston будет доступен внутри функций
export const inngest = new Inngest({
  id: "my-awesome-app",
  logger: logger,
  // ...
});
```
```ts {{ title: "Winston с транспортом Datadog" }}
import { Inngest } from "inngest";
import winston from "winston";

/// Предполагаем, что деплой идет на Vercel.
/// У других провайдеров, скорее всего, есть свои предопределенные переменные окружения.
const env = process.env.VERCEL_ENV || "development";
const ddTransportOps = {
  host: "http-intake.logs.datadoghq.com",
  path: `/api/v2/logs?dd-api-key=${process.env.DD_API_KEY}&ddsource=nextjs&service=inngest&ddtags=env:${env}`,
  ssl: true,
};

const logger = winston.createLogger({
  level: "info",
  exitOnError: false,
  format: winston.format.json(),
  transports: [
    new winston.transports.Console(),
    new winston.transports.Http(ddTransportOps),
  ],
});

// Передайте `logger` в клиент Inngest, и этот логгер winston будет доступен внутри функций
export const inngest = new Inngest({
  id: "my-awesome-app",
  logger: logger,
  // ...
});
```
</CodeGroup>

## Обогащенные логи с метаданными функции

Если библиотека логирования поддерживает дочерние логгеры через `.child()`, встроенное промежуточное ПО (middleware) будет использовать это для автоматического добавления метаданных рантайма функции в ваши логи:

- Имя функции
- Имя события
- ID запуска (Run ID)

```ts {{ title: "Пример использования с логгером Winston" }}
await step.run("summarize-content", async ({ step, logger }) => {
  logger.info("вызов Claude", { max_tokens: 1000 });
});
```
```json {{ title: "Пример вывода лога" }}
{"eventName":"inngest/function.invoked","functionName":"Summarize content via GPT-4",
"level":"info","max_tokens":1000,"message":"Querying vector database",
"runID":"01KB7YQXYNPEX3XB257A3RQDRX"}
```

## Кастомизация логгера

Встроенный логгер реализован с использованием [промежуточного ПО (middleware)](/docs/features/middleware). Вы можете создать собственное промежуточное ПО для настройки логгера под свои нужды. См. [пример middleware для логирования](/docs/reference/middleware/examples#logging) для более подробной информации.

# Троттлинг (Throttling)

Троттлинг позволяет задавать, сколько запусков функций может стартовать в течение временного периода. При достижении лимита новые запуски функций сверх порога троттлинга _ставятся в очередь на будущее_. Троттлинг работает по принципу FIFO (первым пришёл — первым вышел). Примеры использования троттлинга:

* Равномерное распределение выполнения функций во времени для снижения пиковой нагрузки.
* Работа с ограничениями частоты запросов сторонних API.

## Как настроить троттлинг

<CodeGroup>
```ts {{ title: "TypeScript" }}
inngest.createFunction(
  {
    id: "unique-function-id",
    throttle: {
      limit: 1,
      period: "5s",
      burst: 2,
      key: "event.data.user_id",
    },
    triggers: { event: "ai/summary.requested" },
  },
  async ({ event, step }) => {
  },
);
```

```go {{ title: "Go" }}
inngestgo.CreateFunction(
	client,
	inngestgo.FunctionOpts{
		ID: "unique-function-id",
		Throttle: &inngestgo.ConfigThrottle{
			Limit:  1,
			Period: 5 * time.Second,
			Key:    inngestgo.StrPtr("event.data.user_id"),
			Burst:  2,
		},
	},
	inngestgo.EventTrigger("ai/summary.requested", nil),
	func(ctx context.Context, input inngestgo.Input[map[string]any]) (any, error) {
		// Эта функция будет троттлиться до 1 запуска каждые 5 секунд для данного
		// payload события с совпадающим user_id, при этом единовременный всплеск
		// позволяет запустить ещё 2 дополнительных запуска в течение 5 секунд,
		// после чего новые запуски не принимаются в течение 5 секунд.
		return nil, nil
	},
)
```

```py {{ title: "Python" }}
@inngest.create_function(
  id="unique-function-id",
  throttle=inngest.Throttle(
    limit=1,
    period=datetime.timedelta(seconds=5),
    key="event.data.user_id",
    burst=2,
  ),
  trigger=inngest.Trigger(event="ai/summary.requested")
)
async def synchronize_data(ctx: inngest.Context):
    # Эта функция будет троттлиться до 1 запуска каждые 5 секунд для данного
    # payload события с совпадающим user_id, при этом единовременный всплеск
    # позволяет запустить ещё 2 дополнительных запуска в течение 5 секунд,
    # после чего новые запуски не принимаются в течение 5 секунд.
```

</CodeGroup>

Вы можете настраивать троттлинг для каждой функции с помощью необязательного параметра `throttle`. Параметры напрямую управляют алгоритмом Generic Cell Rate, используемым в очереди.

### Справочник по настройке

- `limit`: Общее количество запусков, разрешённых в течение заданного `period`.
- `period`: Период, в течение которого применяется лимит.
- `burst`: Количество запусков, разрешённых в заданном окне в виде единовременного всплеска сверх `limit`.
- `key`: Необязательное выражение, возвращающее ключ троттлинга на основе данных события. Позволяет применять уникальные лимиты троттлинга для конкретного пользователя.

<Callout>
  GCRA разбивает заданный `period` на меньшие окна на основе `limit`.
  Без всплесков GCRA пропускает один запрос на каждое окно. Inngest может пытаться запустить несколько ожидающих запусков функций в короткий промежуток времени, поэтому для гарантии максимальной пропускной способности мы запускаем `limit + burst` запусков функций в каждом окне, что позволяет всем запросам стартовать в течение заданного `period`.

Это необходимо, поскольку фоновые задачи не поступают с той же частотой, что и события, их запускающие.
</Callout>

**Информация о конфигурации**

- Использование троттлинга гарантирует, что в течение окна заданного `period` может стартовать не более `limit + burst` запусков.
- Период должен быть от `1s` до `7d` (от 1 секунды до 7 дней). Минимальная гранулярность — одна секунда.
- Троттлинг в настоящее время применяется на уровне функции. Две функции с одинаковым ключом имеют два отдельных лимита.
- Каждый запрос имеет одинаковый вес и считается за одну единицу в ограничителе частоты.

## Как работает троттлинг

Троттлинг использует [алгоритм Generic Cell Rate (GCRA)](https://en.wikipedia.org/wiki/Generic_cell_rate_algorithm) для ограничения *стартов* запусков функций непосредственно в очереди. Когда вы отправляете событие или вызываете функцию с настройкой троттлинга, Inngest проверяет лимит троттлинга функции на наличие свободной ёмкости:

- Если ёмкость есть, запуск функции начинается как обычно.
- Если ёмкости нет, запуск функции начнётся, когда ёмкость появится в будущем.

Обратите внимание, что троттлинг применяется только к стартам запусков функций. Он не применяется к шагам внутри функции. Это позволяет регулировать частоту начала работы функций, *не беспокоясь* о количестве шагов в функции или о том, выполняются ли шаги параллельно. Для ограничения количества одновременно выполняемых шагов используйте [управление параллелизмом](/docs/guides/concurrency).

Троттлинг работает по принципу [FIFO (первым пришёл — первым вышел)](https://en.wikipedia.org/wiki/FIFO_(computing_and_electronics)), то есть первый поставленный в очередь запуск функции будет первым запущен при появлении свободной ёмкости.

## Троттлинг vs. Параллелизм

**Параллелизм (Concurrency)** ограничивает *количество одновременно выполняемых шагов в запусках вашей функции*. Это позволяет управлять общей ёмкостью ваших функций.

**Троттлинг** ограничивает количество *новых запусков функций*. Он не ограничивает количество выполняемых шагов. Например, при лимите троттлинга 1 в минуту в течение одной минуты стартует только один запуск. Однако этот запуск может выполнить сотни шагов, поскольку троттлинг не ограничивает шаги.

## Троттлинг vs. Ограничение частоты (Rate Limiting)

Ограничение частоты также задаёт, сколько функций может стартовать в течение временного периода. Однако в Inngest ограничение частоты игнорирует запуски функций сверх лимита и не ставит их в очередь на будущее. Троттлинг же ставит запуски сверх лимита в очередь на будущее.

Ограничение частоты *потерично* и устанавливает жёсткие ограничения на запуски функций, тогда как троттлинг откладывает запуски сверх лимита до появления свободной ёмкости, сглаживая пики нагрузки.

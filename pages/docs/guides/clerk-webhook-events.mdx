# Обработка событий вебхуков Clerk

![Логотип Clerk и схема, показывающая события вебхуков Clerk](/assets/docs/guides/clerk-webhook-events/featured-image.png)

Сторонние провайдеры аутентификации, такие как [Clerk](https://clerk.com/), — это фантастический способ добавить функции аутентификации, управления пользователями и безопасности в ваше приложение. Они также предоставляют готовые компоненты, которые позволяют быстро настроить аутентификацию. Однако при наличии внешнего источника истины для аутентификации вам часто придется:

* Синхронизировать данные из Clerk с вашей базой данных,
* Выделять ресурсы для новых аккаунтов или
* Запускать другие задачи на основе событий (например, отправку писем).

Эта страница представляет собой руководство по настройке вебхука Clerk в Inngest и использованию событий Clerk внутри функций Inngest.

## Настройка вебхука Clerk

Clerk позволяет [отправлять события на эндпоинт вебхука](https://clerk.com/docs/integrations/webhooks/overview) при возникновении определенных событий. [Эндпоинты вебхуков](/docs/platform/webhooks) Inngest позволяют получать эти события в вашем аккаунте точно так же, как [события, которые вы отправляете](/docs/events) из собственного приложения.

Чтобы настроить вебхук Clerk, откройте панель управления Clerk и перейдите на страницу "Webhooks". Затем нажмите кнопку "Add Endpoint".

![Страница Webhooks в панели управления Clerk. Красная стрелка указывает на кнопку Add Endpoint.](/assets/docs/guides/clerk-webhook-events/webhook-page.webp)

На следующей странице выберите вкладку шаблона "Transformation", выберите шаблон Inngest, а затем нажмите кнопку "Connect to Inngest".

![Страница Webhooks в панели управления Clerk, показывающая шаблон трансформации Inngest. Красные стрелки указывают на вкладку Transformation Template, шаблон Inngest и кнопку Connect to Inngest.](/assets/docs/guides/clerk-webhook-events/webhook-transformation-template.webp)

Появится всплывающее окно для завершения настройки. Выберите "Approve", чтобы создать вебхук.

![Всплывающее окно разрешений Inngest с кнопкой Approve.](/assets/docs/guides/clerk-webhook-events/inngest-permissions-dialog.png)

После того как всплывающее окно исчезнет, на странице Webhooks отобразится статус "Connected" с URL вебхука под ним. Остался еще один шаг для завершения настройки.

![Страница Webhooks в панели управления Clerk, показывающая подключенный аккаунт Inngest. Красная стрелка указывает на кнопку Connected.](/assets/docs/guides/clerk-webhook-events/webhook-endpoint-connected.webp)

Чтобы завершить настройку, прокрутите вниз и выберите "Create".

![Страница Webhooks в панели управления Clerk, показывающая конец страницы для создания нового эндпоинта. Красная стрелка указывает на кнопку Create.](/assets/docs/guides/clerk-webhook-events/webhook-create.webp)

Вы будете перенаправлены на новый эндпоинт. В вашей панели управления Inngest вы увидите новый вебхук, созданный в [продакшн-среде](https://app.inngest.com/env/production/manage/webhooks) вашего аккаунта.

## Создание функции для синхронизации нового пользователя с базой данных

Часто ключевой частью интеграции с провайдером аутентификации, таким как [Clerk](https://clerk.com/), является обработка асинхронных обновлений с помощью вебхука.

Предположим, вам нужно написать функцию, которая будет вставлять нового пользователя в базу данных и запускаться всякий раз, когда происходит событие `clerk/user.created`. Вы должны использовать метод `inngest.createFunction()`, как в примере ниже:

```ts {{ filename: "src/inngest/sync-user.ts" }}
const syncUser = inngest.createFunction(
  { id: 'sync-user-from-clerk', triggers: { event: 'clerk/user.created' } },
  async ({ event }) => {
    // Полезная нагрузка события (data) будет представлять собой JSON-объект пользователя Clerk
    const { user } = event.data;
    const { id, first_name, last_name } = user;
    const email = user.email_addresses.find(e =>
      e.id === user.primary_email_address_id
    ).email;
    await database.users.insert({ id, email, first_name, last_name });
  }
)
```

Объект `event` содержит все соответствующие данные для события. `event.data` будет соответствовать объекту `data` из стандартной [структуры полезной нагрузки](https://clerk.com/docs/integrations/webhooks/overview#payload-structure) вебхука Clerk. В случае события `clerk/user.created`, `event.data` будет JSON-объектом пользователя Clerk.

Как видите, вы можете выбирать, какие события вы хотите обрабатывать в каждой функции. Вы можете написать отдельные функции для `clerk/user.updated` и `clerk/user.deleted`, обрабатывая весь жизненный цикл от начала до конца.

Обратите внимание, что несколько функций могут слушать одно и то же событие. Этот паттерн называется «[fan-out (разветвление)](/docs/guides/fan-out-jobs)».

## Создание функции для отправки приветственного письма

Часто приложениям требуется выполнять дополнительные задачи при создании нового пользователя, например отправлять приветственное письмо с советами и полезной информацией.

Хотя можно добавить эту логику в конец вашей функции синхронизации, как показано в [предыдущем разделе](/docs/guides/clerk-webhook-events#creating-a-function-to-sync-a-new-user-to-a-database), лучше разделять несвязанные задачи на разные функции, чтобы проблемы с одной задачей не влияли на другие. Например, если письмо не удалось отправить, это не должно мешать запуску пробного периода для этого пользователя в Stripe.

Вы можете воспользоваться тем фактом, что в Inngest каждая функция имеет [автоматические повторные попытки](/docs/functions/retries), поэтому повторно запускается только тот код, в котором возникли проблемы.

Код ниже создает еще одну функцию, использующую то же событие `clerk/user.created`, и добавляет логику для отправки приветственного письма:


```ts {{ filename: "src/inngest/send-welcome-email.ts" }}
const sendWelcomeEmail = inngest.createFunction(
  { id: 'send-welcome-email', triggers: { event: 'clerk/user.created' } },
  async ({ event }) => {
    const { user } = event.data;
    const { first_name } = user;
    const email = user.email_addresses.find(e =>
      e.id === user.primary_email_address_id
    ).email;
    await emails.sendWelcomeEmail({ email, first_name });
  }
)
```

Теперь у вас есть функция, которая использует то же самое событие вебхука Clerk для другой цели. События вебхуков Clerk можно использовать для самых разных сценариев жизненного цикла приложения. Например, добавление пользователей в список маркетинговой рассылки, запуск пробного периода Stripe или выделение ресурсов для нового аккаунта.

### Отправка отложенного последующего письма

Чтобы отправить последующее письмо, вы можете использовать метод [`step.run()`](/docs/reference/functions/step-run). Этот метод инкапсулирует определенный код, который будет автоматически перезапущен в случае ошибки, гарантируя, что проблемы в одной части вашей функции не заставят всю функцию запускаться заново. Кроме того, вы расширите функциональность с помощью [`step.sleep()`](/docs/reference/functions/step-sleep).

Код ниже сначала отправляет приветственное письмо, а затем использует `step.sleep()`, чтобы подождать три дня перед отправкой другого письма с предложением пробного периода:

```ts {{ filename: "src/inngest/send-welcome-email.ts" }}
const sendWelcomeEmail = inngest.createFunction(
  { id: 'send-welcome-email', triggers: { event: 'clerk/user.created' } },
  async ({ event, step }) => {
    const { user } = event.data;
    const { first_name } = user;
    const email = user.email_addresses.find(e =>
      e.id === user.primary_email_address_id
    ).email;

    // Обертывание каждой отдельной задачи в step.run() гарантирует, что каждая из них
    // будет автоматически перезапущена при ошибке и не будет запускаться повторно без нужды
    await step.run('welcome-email', async () => {
      await emails.sendWelcomeEmail({ email, first_name })
    });

    // ждем 3 дня перед вторым письмом
    await step.sleep('wait-3-days', '3 days');

    await step.run('trial-offer-email', async () => {
      await emails.sendTrialOfferEmail({ email, first_name })
    });
  }
)
```

# Управление конкурентностью (Concurrency)

Ограничение конкурентности в системах является важным инструментом для правильного управления вычислительными ресурсами и масштабирования рабочих нагрузок. Управление конкурентностью в Inngest позволяет вам контролировать количество _шагов_, которые выполняются одновременно.

<Callout>
**Важно:** Конкурентность ограничивает количество _шагов_, выполняемых в один момент времени, а **не** количество запусков функций. Запуск функции, который находится в режиме ожидания (sleeping), ожидает события или приостановлен между шагами, **не** учитывается в лимите конкурентности. Только шаги, активно выполняющие код, учитываются в этом лимите.

Это означает, что у вас может быть гораздо больше запущенных функций, чем предполагает ваш лимит конкурентности, так как большинство из них, скорее всего, приостановлены или ожидают между шагами.
</Callout>

{/* TODO - Link to updated keys section */}
Конкурентность шагов может быть дополнительно настроена с помощью «ключей» (keys), что применяет лимит к каждому уникальному значению ключа (например, ID пользователя). Опция конкурентности также может быть применена к различным «областям видимости» (scopes), что позволяет разделять лимит конкурентности между _несколькими_ функциями.

По сравнению с традиционными системами очередей и воркеров, Inngest управляет конкурентностью внутри системы, поэтому вам не нужно реализовывать дополнительную логику на уровне воркеров или управлять состоянием.

## Когда использовать конкурентность

Конкурентность наиболее полезна, когда вы хотите ограничить использование ресурсов вашей функцией. Некоторые сценарии использования включают:

- **Ограничение в мультиарендных (multi-tenant) системах** — предотвращение ситуации, когда одна учетная запись, пользователь или арендатор потребляет слишком много ресурсов, создавая очередь для остальных. См.: [Ключи конкурентности (Мультиарендная конкурентность)](#concurrency-keys-multi-tenant-concurrency).
- **Ограничение пропускной способности для операций с базой данных** — предотвращение перегрузки базы данных или аналогичного ресурса потенциально объемными задачами. См.: [Общие лимиты для нескольких функций (scope)](#sharing-limits-across-functions-scope).
- **Базовые лимиты одновременных операций** — ограничение мощности, выделенной для обработки определенной задачи, например, пайплайна импорта. См.: [Базовая конкурентность](#basic-concurrency).
- **Комбинирование вышеперечисленного** — для одной функции можно добавить несколько лимитов конкурентности. См.: [Комбинирование нескольких лимитов конкурентности](#combining-multiple-concurrency-limits).

<Info>
Если вам нужно ограничить функцию определенной скоростью обработки (например, из-за лимитов стороннего API), вам может понадобиться [дросселирование (throttling)](/docs/guides/throttling). Дросселирование применяется на уровне функции, в то время как конкурентность — на уровне шагов.
</Info>

### Базовая конкурентность

Самый простой лимит конкурентности — это одиночный параметр `limit`, установленный в целое число, обозначающее максимальное количество одновременно выполняемых шагов. При достижении лимита новые шаги будут ставиться в очередь, образуя список ожидания для последующей обработки.

<CodeGroup>
```ts
inngest.createFunction(
  {
    id: "generate-ai-summary",
    concurrency: 10,
    triggers: { event: "ai/summary.requested" },
  },
  async ({ event, step }) => {
    // Обработчик вашей функции
  }
);
```

### Ключи конкурентности (Мультиарендная конкурентность)

Используйте выражение `key` для применения `limit` к каждому уникальному полученному значению ключа. Внутри системы Inngest это создает **виртуальную очередь** для каждого уникального значения и ограничивает конкурентность для каждого из них.

<CodeGroup>
```ts

  {
    id: "generate-ai-summary",
    concurrency: [

    ],

Любые другие функции, использующие этот же ключ `"openai"`, будут использовать этот же общий лимит.

### Комбинирование нескольких лимитов конкурентности

Опция конкурентности в каждом SDK поддерживает до двух лимитов. Это наиболее эффективно при комбинировании лимитов с разными областями видимости (`scope`). Вот пример, объединяющий два лимита: один с областью видимости `"account"`, а другой — на уровне функции (`"fn"`). Комбинирование лимитов создаст несколько виртуальных очередей для ограничения конкурентности. В функции ниже:

- Если в виртуальной очереди ключа 'openai' уже выполняется 10 шагов, любые последующие запуски будут заблокированы и будут ждать завершения текущих запусков перед началом выполнения.
- Если в очереди 'openai' выполняется 5 шагов, и один и тот же `event.data.account_id` ставит в очередь 2 запуска, второй запуск будет ограничен виртуальной очередью `event.data.account_id` и будет ожидать перед выполнением.



<CodeGroup>

```ts {{ title: "TypeScript" }}
inngest.createFunction(
  {
    id: "unique-function-id",
    concurrency: [
      {
         // Использовать лимит конкурентности уровня аккаунта для этой функции,
         // используя ключ "openai" в качестве виртуальной очереди. Любая другая
         // функция, использующая тот же ключ "openai", учитывается в этом лимите.
         scope: "account",
         key: `"openai"`,
         limit: 10,
      },
      {
         // Создать еще одну виртуальную очередь конкурентности только для этой функции.
         // Это ограничивает все аккаунты одним выполняемым шагом для этой функции
         // на основе поля `event.data.account_id`.
         // ПРИМЕЧАНИЕ: "fn" — это область видимости по умолчанию, поэтому ее можно опустить.
         scope: "fn",
         key: "event.data.account_id",
         limit: 1,
      },
    ],
    triggers: { event: "ai/summary.requested" },
  },
  async ({ event, step }) => {
  }
);
```

## Как работает конкурентность

**Конкурентность работает путем ограничения количества шагов, выполняемых в один момент времени.** В Inngest выполнение определяется как «SDK, запускающий код». **Вызов **`step.sleep`**, **`step.sleepUntil`**, **`step.waitForEvent`** или **`step.invoke`** не учитывается в лимитах мощности**, так как SDK не выполняет код, пока эти шаги ожидают.

### Понимание выполнения шагов в сравнении с запусками функций

Поскольку ожидание или пауза являются обычным делом, конкурентность _не_ ограничивает количество функций, находящихся в процессе выполнения. Вместо этого она ограничивает количество шагов, активно выполняемых в любой момент времени.

Анимация ниже показывает, как работает конкурентность в Inngest. Вы можете видеть, как различные задачи встают в очередь и проходят через систему. Когда установлен лимит, только указанное количество шагов может выполняться одновременно. По мере завершения шагов начинают выполняться следующие шаги из очереди.

<Lottie animationData={Concurrency} loop={true} />

Ключевой момент заключается в том, что ваш лимит конкурентности применяется к _активному выполнению_, а не к количеству запусков функций в процессе. Рассмотрим функцию с лимитом `concurrency`, равным `10`:

- У вас могут быть **сотни** запусков функций в процессе.
- Но только **10 шагов** могут активно выполнять код одновременно.
- Когда запуск функции вызывает `step.sleep("wait", "1h")`, он освобождает свой слот выполнения.
- Этот слот становится доступным для использования другими шагами.

<Info>
  **Что учитывается в конкурентности:**
  - `step.run()` — пока выполняется код шага.

  **Что НЕ учитывается в конкурентности:**
  - `step.sleep()` / `step.sleepUntil()` — во время паузы.
  - `step.waitForEvent()` — во время ожидания события.
  - `step.invoke()` — во время ожидания завершения вызванной функции.
  - Время между шагами — когда Inngest координирует запуск следующего шага.
</Info>

### Порядок очереди

**Очереди упорядочены от старых задач к новым ([FIFO](https://en.wikipedia.org/wiki/FIFO))** в рамках одной и той же функции. Порядок между различными функциями не гарантируется. Это означает, что внутри конкретной функции Inngest отдает приоритет завершению старых запусков перед началом новых — даже если старые функции продолжают планировать новые шаги для запуска. Однако разные функции соревнуются за мощность, при этом запуски в наиболее загруженной функции с большей вероятностью (но без гарантии) будут запланированы первыми.

### Дополнительная информация

- Порядок ключей не имеет значения. Конкурентность ограничивается любым ключом, который достигает своих пределов.
- Вы можете указать несколько ключей для одной и той же области видимости, если результирующий `key` вычисляется в разные строки.

## Управление конкурентностью для конкретных шагов в функции

Вам может потребоваться установить другой лимит конкурентности для отдельного шага в функции. Например, в AI-потоке у вас может быть 10 шагов предварительной обработки, которые могут выполняться с высокими лимитами, и один вызов AI с гораздо более низкими лимитами.

Чтобы управлять конкурентностью отдельных шагов, вынесите этот шаг в новую функцию с ее _собственными_ элементами управления конкурентностью и вызовите новую функцию с помощью `step.invoke`. Это позволяет комбинировать элементы управления конкурентностью и управлять «потоком выполнения» чистым и модульным способом.

## Как работают глобальные лимиты

<Callout>
  Хотя две функции могут разделять разные лимиты с областью видимости `account`, мы настоятельно рекомендуем использовать глобальную константу с единым общим лимитом.
</Callout>

Вы можете написать две функции, которые определяют разные уровни для лимита конкурентности с областью видимости 'account'. Например, функция A может ограничивать мощность "ai" до 5, в то время как функция B ограничивает мощность "ai" до 50:


<CodeGroup>

```ts {{ title: "TypeScript" }}
inngest.createFunction(
  {
    id: "func-a",
    concurrency: {
      scope: "account",
      key: `"openai"`,
      limit: 5,
    },
    triggers: { event: "ai/summary.requested" },
  },
  async ({ event, step }) => {
  }
);

inngest.createFunction(
  {
    id: "func-b",
    concurrency: {
      scope: "account",
      key: `"openai"`,
      limit: 50,
    },
    triggers: { event: "ai/summary.requested" },
  },
  async ({ event, step }) => {
  }
);
```

```


```py {{ title: "Python" }}

</CodeGroup>
В Inngest это работает и *не* является конфликтом. Вместо этого функция A ограничивается каждый раз, когда в очереди 'openai' выполняется 5 или более функций. Функция B, однако, ограничивается, когда в очереди 50 или более элементов. Это означает, что у функции B больше мощностей, чем у функции A, хотя обе ограничены и соревнуются в одной и той же виртуальной очереди.

Поскольку функции работают по принципу FIFO, над запусками функций тем вероятнее будут работать, чем старше становятся их задачи (по мере роста очереди). Если задачи функции A остаются в очереди дольше, чем задачи функции B, вполне вероятно, что над ними начнут работать, как только освободится мощность. Тем не менее, у функции B почти всегда будет мощность раньше, чем у функции A, и она может блокировать работу функции A.

**Хотя это работает, мы настоятельно рекомендуем использовать глобальные константы для областей видимости уровня `env` или `account`, устанавливая функциям одинаковый лимит.**

## Ограничения

- Конкурентность ограничивает количество шагов, выполняемых в один момент времени. Она *пока* не выполняет ограничение частоты (rate limiting) за определенный период времени.
- Функции могут указывать до 2 ограничений конкурентности одновременно.
- Максимальный лимит конкурентности определяется <a href="/pricing?ref=docs-concurrency" target="_blank" rel="nofollow">планом вашей учетной записи</a>.
- Порядок выполнения внутри одной и той же функции гарантирован (за исключением повторных попыток).
- Порядок выполнения между разными функциями не гарантирован. Функции соревнуются друг с другом за планирование в случайном порядке.

## Справочник по конкурентности

<Properties>
  <Property name="limit" type="number" required>
    Максимальное количество одновременно выполняемых шагов.
    Значение `0` или `undefined` эквивалентно отсутствию лимита.
    Максимальное значение диктуется <a href="/pricing?ref=docs-concurrency" target="_blank" rel="nofollow">планом вашей учетной записи</a>.
  </Property>
  <Property name="scope" type="'account' | 'env' | 'fn'">
    Область видимости для лимита конкурентности, которая влияет на то, управляется ли конкурентность для отдельной функции, во всей среде или во всей вашей учетной записи.

    * `fn` (по умолчанию): лимит конкурентности затрагивает только запуски этой функции.
    * `env`: лимит конкурентности затрагивают все запуски в одной среде, которые разделяют одно и то же вычисленное значение ключа. Это требует установки `key`, который вычисляется в имя виртуальной очереди.
    * `account`: лимит конкурентности затрагивают каждый запуск, разделяющий одно и то же вычисленное значение ключа, во всех средах. Это требует установки `key`, который вычисляется в имя виртуальной очереди.

    Каждый SDK предоставляет эти перечисления в идиоматической для данного языка манере, хотя значения перечислений одинаковы для всех языков.
  </Property>
  <Property name="key" type="string">
    Выражение, которое вычисляется в строку на основе инициирующего события. Строка, возвращаемая выражением, используется как имя очереди конкурентности. Ключ обязателен при установке области видимости уровня `env` или `account`.

    Выражения определяются с использованием Common Expression Language (CEL), а доступ к исходному событию осуществляется через точечную нотацию. Прочтите [наше руководство по написанию выражений](/docs/guides/writing-expressions) для получения дополнительной информации. Примеры:

    * Ограничить конкурентность до `n` (через `limit`) на каждого ID клиента: `'event.data.customer_id'`
    * Ограничить конкурентность до `n` на пользователя для каждого ID импорта: `'event.data.user_id + "-" + event.data.import_id'`
    * Ограничить глобально с помощью конкретной строки: `'"global-quoted-key"'` (обернуто в кавычки, так как выражение вычисляется как язык)
  </Property>
</Properties>

## Дополнительные примеры

### Ограничение параллельных задач импорта для ID клиента

В этой гипотетической системе клиенты могут загружать файлы `.csv`, каждый из которых должен быть обработан и импортирован. Мы хотим ограничить каждого клиента только одной задачей импорта за раз, чтобы две задачи не записывали данные клиента одновременно. Мы делаем это, устанавливая `limit: 1` и ключ конкурентности `key` равным `customerId`, который включен в полезную нагрузку каждого события.

Inngest гарантирует, что конкурентность (`1`) применяется к каждому уникальному значению `event.data.customerId`. Это позволяет разным клиентам выполнять шаги в одно и то же время, но у конкретного клиента не может быть двух шагов, выполняемых одновременно!

<CodeGroup>

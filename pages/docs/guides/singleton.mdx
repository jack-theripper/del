## Когда использовать Singleton-функции

Singleton-функции полезны, когда вы хотите гарантировать, что одновременно выполняется только один экземпляр функции, например:
- Рабочий процесс синхронизации данных со сторонним сервисом
- Вычислительно- или времязатратная функция, которая не должна выполняться несколько раз одновременно (например, AI-обработка)


### Singleton в сравнении с параллелизмом (concurrency):

Тогда как [Параллелизм (Concurrency)](/docs/guides/concurrency), установленный на `1`, гарантирует, что одновременно выполняется только один шаг данной функции, Singleton-функции гарантируют, что одновременно происходит только один запуск данной функции.


### Singleton в сравнении с ограничением частоты (Rate Limiting):

[Ограничение частоты (Rate Limiting)](/docs/guides/rate-limiting) похоже на Singleton-функции, но предназначено для ограничения количества запусков в течение временного периода, тогда как Singleton-функции предназначены для обеспечения того, что только один запуск функции происходит в течение заданного временного окна.

Ограничение частоты полезно для контроля скорости выполнения функции, тогда как Singleton-функции полезны для обеспечения того, что только один запуск функции происходит в течение заданного временного окна.




## Как это работает

Singleton-функции настраиваются с помощью свойства `singleton` в определении функции.

Следующая функция `data-sync` демонстрирует singleton-поведение в рамках отдельных пользователей. В зависимости от `mode`, новые запуски будут либо пропускаться, либо отменять существующий запуск:

```ts
const dataSync = inngest.createFunction({
    id: "data-sync",
    singleton: {
      key: "event.data.user_id",
      mode: "skip",
    },
    triggers: { event: "data-sync.start" },
  },
  async ({ event }) => {
    // ...
  },
);
```

Подробнее см. в [справочной документации](/docs/reference/functions/singleton).

### Использование `key`

Когда задан `key`, правило уникальных запусков применяется для каждого уникального значения выражения `key`. Например, если ваш `key` установлен как `event.data.user_id`, для каждого пользователя будет применяться своё индивидуальное singleton-правило, гарантируя, что для каждого пользователя одновременно выполняется только один запуск функции. Подробнее о написании выражений — в [нашем руководстве по выражениям](/docs/guides/writing-expressions).


### Два режима: Skip и Cancel

Singleton-функции можно настроить так, чтобы они либо пропускали новый запуск, либо отменяли существующий и запускали новый.

Свойство `mode` настраивает поведение Singleton-функции:

- `"skip"` — Пропускает новый запуск, если уже выполняется другой запуск.
- `"cancel"` — Отменяет существующий запуск и запускает новый.

<Callout variant="info" skipSearchCrawler>
  **Поведение режима Cancel**: Запуск нескольких функций с одним и тем же ключом в очень быстрой последовательности может привести к тому, что некоторые запуски будут пропущены, а не отменены, — аналогично эффекту debounce. Это предотвращает избыточные накладные расходы на отмену при быстрой серии событий.
</Callout>

#### Когда использовать режим "cancel" vs. режим "skip"?

Используйте режим `"skip"`, когда хотите предотвратить дублирование работы и сохранить текущую выполняющуюся функцию. Используйте режим `"cancel"`, когда хотите гарантировать, что всегда обрабатывается самое последнее событие, даже если это означает отмену выполняющегося запуска.

<CodeGroup>
```ts {{ title: "Режим Skip" }}
const dataSync = inngest.createFunction({
    id: "data-sync",
    singleton: {
      key: "event.data.user_id",
      mode: "skip",
    },
    triggers: { event: "data-sync.start" },
  },
  async ({ event }) => {
    const userId = event.data.user_id;

    // Этот длительный процесс синхронизации не будет прерван
    // Если для этого пользователя будет запущена другая синхронизация, она будет пропущена
    const data = await syncUserDataFromExternalAPI(userId);
    const processed = await processLargeDataset(data);
    await updateDatabase(processed);
  },
);
```
```ts {{ title: "Режим Cancel" }}
const latestDataSync = inngest.createFunction({
    id: "latest-data-sync",
    singleton: {
      key: "event.data.user_id",
      mode: "cancel",
    },
    triggers: { event: "data-sync.start" },
  },
  async ({ event }) => {
    const userId = event.data.user_id;

    // Если запускается более новая синхронизация, этот запуск будет отменён,
    // гарантируя обработку только самых актуальных данных
    const payload = await fetchLatestUserData(userId);
    await applyRealTimeUpdates(payload);
  },
);
```
</CodeGroup>


## Совместимость с другими функциями управления потоком

Singleton-функции можно комбинировать с другими функциями управления потоком, учитывая следующее:

| Управление потоком | Совместимость | Примечания |
| --- | --- | --- |
| [Debounce](/docs/guides/debounce) | ✅ | Можно использовать совместно без проблем. |
| [Ограничение частоты (Rate limiting)](/docs/guides/rate-limiting) | ✅ | Похожая функциональность, но ограничение частоты работает в рамках заданного временного окна, а не продолжительности выполнения функции. |
| [Троттлинг (Throttling)](/docs/guides/throttling) | ✅ | Похожая функциональность, но троттлинг ставит события в очередь с течением времени, а не отбрасывает/отменяет их. |
| [Параллелизм (Concurrency)](/docs/guides/concurrency) | ❌ | Singleton-функции неявно имеют параллелизм равный 1. Настройку параллелизма можно задать, но следует использовать с осторожностью. |
| [Пакетирование (Batching)](/docs/guides/batching) | ❌ | Singleton несовместим с пакетированием; регистрация функции завершится ошибкой, если установлены оба параметра. |


## Часто задаваемые вопросы

### Как Singleton-функции работают с повторными попытками (retries)?

Если singleton-функция завершилась с ошибкой и выполняет повторную попытку, новые входящие запуски всё равно должны пропускаться.

# Многошаговые функции (Multi-Step Functions)

Используйте многошаговые функции Inngest для безопасной координации событий, задержки выполнения на часы (или до года), повторного запуска [отдельных шагов](/docs/learn/inngest-steps) и условного выполнения кода на основе результатов предыдущих шагов и входящих событий.

Важно отметить, что многошаговые функции описываются кодом, а не конфигурацией. Это позволяет создавать читаемую и понятную функциональность, которую легко поддерживать.

## Преимущества многошаговых функций

Создание функций, состоящих из нескольких шагов, позволяет:

- Запускать блоки кода с возможностью повтора для максимальной надежности.
- Приостанавливать выполнение и ждать события, соответствующего определенным правилам, прежде чем продолжить.
- Делать паузу на определенное время или до указанного момента.

Этот подход упрощает написание надежного распределенного кода. Оборачивая асинхронные действия (например, вызовы API) в блоки с поддержкой повторов, мы гарантируем надежность при взаимодействии со множеством сервисов.


## Как написать многошаговую функцию

<LanguageSelector>

<LanguageSection show="typescript">
Рассмотрим простую [функцию Inngest](/docs/learn/inngest-functions), которая отправляет приветственное письмо при регистрации пользователя:

<CodeGroup>
```ts
import { Inngest } from "inngest";

const inngest = new Inngest({ id: "my-app" });

export default inngest.createFunction(
  { id: "activation-email", triggers: { event: "app/user.created" } },
  async ({ event }) => {
    await sendEmail({ email: event.user.email, template: "welcome" });
  }
);
```
</CodeGroup>

Эта функция обладает всеми преимуществами Inngest: код надежен и поддерживает повторные попытки. В случае ошибки данные будут восстановлены. Это отлично работает для функций с одной задачей.

Однако появилось новое требование: если пользователь не создал ни одного поста на нашей платформе в течение 24 часов после регистрации, мы должны отправить ему еще одно письмо. Вместо того чтобы усложнять логику обработчика, мы можем превратить эту функцию в многошаговую.

### 1. Преобразование в функцию с шагами

Сначала преобразуем эту функцию в многошаговую:
- Добавим аргумент `step` в обработчик функции Inngest.
- Обернем вызов `sendEmail()` в метод [`step.run()`](/docs/reference/functions/step-run).

```ts
export default inngest.createFunction(
  { id: "activation-email", triggers: { event: "app/user.created" } },
  async ({ event, step }) => {
    await step.run("send-welcome-email", async () => {
      return await sendEmail({ email: event.user.email, template: "welcome" });
    });
  }
);
```

Основное отличие в том, что мы обернули вызов `sendEmail()` в `step.run()`. Так мы сообщаем Inngest, что это отдельный шаг в нашей функции. Этот шаг может быть перезапущен независимо, точно так же, как и одношаговая функция.

### 2. Добавление шага: ожидание события

После отправки приветственного письма мы хотим подождать не более 24 часов, пока пользователь создаст пост. Если он этого не сделает, мы отправим ему напоминание.

В другом месте нашего приложения событие `app/post.created` отправляется всякий раз, когда пользователь создает новый пост. Мы можем использовать его для запуска второго письма.

Для этого воспользуемся методом [`step.waitForEvent()`](/docs/reference/functions/step-wait-for-event). Этот инструмент будет ждать возникновения подходящего события, а затем вернет данные этого события. Если событие не произойдет в течение указанного времени, метод вернет `null`, что мы и используем для принятия решения об отправке напоминания.

<CodeGroup>
```ts
export default inngest.createFunction(
  { id: "activation-email", triggers: { event: "app/user.created" } },
  async ({ event, step }) => {
    await step.run("send-welcome-email", async () => {
      return await sendEmail({ email: event.user.email, template: "welcome" });
    });

    // Ждем события "app/post.created"
    const postCreated = await step.waitForEvent("wait-for-post-creation", {
      event: "app/post.created",
      match: "data.user.id", // поле "data.user.id" должно совпадать
      timeout: "24h", // ждать не более 24 часов
    });
  }
);
```
</CodeGroup>

Теперь у нас есть переменная `postCreated`, которая будет содержать `null`, если пользователь не создал пост в течение 24 часов, или данные события, если создал.

### 3. Настройка условного действия

Наконец, мы можем использовать переменную `postCreated`, чтобы отправить напоминание, если пост не был создан. Добавим еще один блок кода с `step.run()`:

<CodeGroup>
```ts
export default inngest.createFunction(
  { id: "activation-email", triggers: { event: "app/user.created" } },
  async ({ event, step }) => {
    await step.run("send-welcome-email", async () => {
      return await sendEmail({ email: event.user.email, template: "welcome" });
    });

    // Ждем события "app/post.created"
    const postCreated = await step.waitForEvent("wait-for-post-creation", {
      event: "app/post.created",
      match: "data.user.id", // поле "data.user.id" должно совпадать
      timeout: "24h", // ждать не более 24 часов
    });

    if (!postCreated) {
      // Если пост не был создан, отправляем напоминание
      await step.run("send-reminder-email", async () => {
        return await sendEmail({
          email: event.user.email,
          template: "reminder",
        });
      });
    }
  }
);
```
</CodeGroup>

Вот и всё! Мы написали многошаговую функцию, которая отправляет приветственное письмо, а затем — напоминание, если пользователь не создал пост в течение 24 часов.

Самое главное, что для этого нам не пришлось писать файлы конфигурации. Мы используем всю мощь JavaScript для написания функций и возможности инструментов Inngest для координации событий и шагов.

## Нюансы (Gotchas)

### Моя функция запускается дважды

Inngest будет взаимодействовать с вашей функцией несколько раз в течение одного запуска и использовать ваши инструменты для интеллектуальной мемоизации состояния.

По этой причине размещение бизнес-логики вне вызова `step.run()` является плохой идеей, так как этот код будет выполняться при каждом взаимодействии Inngest с вашей функцией.

### Я хочу запустить асинхронный код

`step.run()` принимает асинхронную функцию (`async`):

```ts
await step.run("do-something", async () => {
  // ваш код
});
```

Каждый вызов `step.run()` — это один шаг с поддержкой повторов, своего рода «легкая транзакция». Поэтому каждый шаг должен иметь один побочный эффект. Например, приведенный ниже код проблематичен:

```ts
await step.run("create-alert", async () => {
  const alertId = await createAlert();
  await sendAlertLinkToSlack(alertId);
});
```

Если `createAlert()` выполнится успешно, а `sendAlertLinkToSlack()` завершится ошибкой, код будет перезапущен, и оповещение (alert) будет создаваться заново при каждой попытке.

Вместо этого следует разделять асинхронные действия на несколько шагов, чтобы они повторялись независимо.

```ts
const alertId = await step.run("create-alert", () => createAlert());

await step.run("send-alert-link", () => sendAlertLinkToSlack(alertId));
```

### Моя переменная не обновляется

Поскольку Inngest взаимодействует с вашей функцией несколько раз, мемоизируя состояние по ходу дела, код внутри вызовов `step.run()` не вызывается при каждом обращении.

Убедитесь, что любые переменные, необходимые для всей функции, *возвращаются* из вызовов `step.run()`:

```ts
// Это правильный способ установки переменных внутри step.run :)
const userId = await step.run("get-user", () => getRandomUserId());

console.log(userId); // 123
```

Для сравнения, вот **два примера неработающего кода** (если вы используете шаги для обновления переменных в замыкании функции):

```ts
// ЭТО НЕПРАВИЛЬНО! step.run() выполняется только один раз и пропускается при
// последующих вызовах, поэтому userId не будет определен.
let userId;

// НЕ ДЕЛАЙТЕ ТАК! Вместо этого возвращайте данные из step.run()
await step.run("get-user", async () => {
  userId = await getRandomUserId();
});

console.log(userId); // undefined
```

### `sleepUntil()` работает не так, как ожидалось

Используйте `sleepUntil()` только с датами, которые будут статичными при различных вызовах вашей функции.

Всегда используйте `sleep()`, если вы хотите подождать определенное время, отсчитываемое от *текущего момента*.

```ts
// ❌ Плохо
const tomorrow = new Date();
tomorrow.setDate(tomorrow.getDate() + 1);
await step.sleepUntil("wait-until-tomorrow", tomorrow);

// ✅ Хорошо
await step.sleep("wait-a-day", "1 day");
```

```ts
// ✅ Хорошо
const userBirthday = await step.run("get-user-birthday", async () => {
  const user = await getUser();
  return user.birthday; // Date
});

await sleepUntil("wait-for-user-birthday", userBirthday);
```

### Неожиданное поведение циклов

При использовании циклов внутри функций рекомендуется рассматривать каждую итерацию как отдельный шаг или набор шагов.

При [запуске функций](/docs/learn/how-functions-are-executed) обработчик функции выполняется заново с самого начала для каждого нового шага, а ранее завершенные шаги мемоизируются. Это означает, что итерации циклов будут выполняться при каждом перезапуске, но код, инкапсулированный в `step.run()`, не будет запускаться повторно.

Если код внутри цикла не инкапсулирован в шаг, он будет выполняться многократно, что может привести к запутанному поведению, проблемам при отладке или [логировании](/docs/guides/logging). Вот почему рекомендуется инкапсулировать недетерминированный код в `step.run()` при работе с циклами.

Узнайте больше о [работе с циклами в Inngest](/docs/guides/working-with-loops).

### `sleep_until()` работает не так, как ожидалось

Используйте `sleep_until()` только с датами, которые будут статичными при различных вызовах вашей функции.

Всегда используйте `sleep()`, если вы хотите подождать определенное время, отсчитываемое от *текущего момента*.

```py
# ❌ Плохо
tomorrow = datetime.now() + timedelta(days=1)
await ctx.step.sleepUntil("wait-until-tomorrow", tomorrow);

# ✅ Хорошо
await ctx.step.sleep("wait-a-day", "1 day");
```

```py
# ✅ Хорошо
user_birthday = await ctx.step.run("get-user-birthday", async () => {
  user = await get_user();
  return user.birthday; # Дата
});

await ctx.step.sleep_until("wait-for-user-birthday", user_birthday);
```

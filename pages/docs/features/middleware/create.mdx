# Создание промежуточного ПО

Создание промежуточного ПО означает определение жизненных циклов и хуков внутри них для выполнения кода. Жизненные циклы — это действия, такие как запуск функции или отправка событий, а хуки внутри них — это место выполнения кода, обычно с шагами _до_ и _после_.

<LanguageSelector>

    <LanguageSection show="go">

      <Callout>
        Поддержка промежуточного ПО в Go SDK планируется.
      </Callout>

    </LanguageSection>

<LanguageSection show="typescript">
    Промежуточное ПО создаётся расширением класса `Middleware.BaseMiddleware`.

    **`class MyMiddleware extends Middleware.BaseMiddleware`**


    ```ts
    import { Middleware } from "inngest";

    // Create a new middleware
    class MyMiddleware extends Middleware.BaseMiddleware {
      // Override hook methods here
    }

    // Register it on the client
    const inngest = new Inngest({
      id: "my-app",
      middleware: [MyMiddleware],
    });
    ```

</LanguageSection>

<LanguageSection show="python">

    Промежуточное ПО создаётся с помощью класса `inngest.Middleware`.

    **`class MyMiddleware(inngest.Middleware):`**

    ```py
    import inngest

    class MyMiddleware(inngest.Middleware):
        def __init__(
            self,
            client: inngest.Inngest,
            raw_request: object,
        ) -> None:
            #  ...

        async def before_send_events( self, events: list[inngest.Event]) -> None:
            print(f"Sending {len(events)} events")

        async def after_send_events(self, result: inngest.SendEventsResult) -> None:
            print("Done sending events")

    inngest_client = inngest.Inngest(
        app_id="my_app",
        middleware=[MyMiddleware],
    )
    ```

</LanguageSection>

## Инициализация

<LanguageSection show="typescript">
Как видно выше, начинаем с класса, расширяющего `Middleware.BaseMiddleware`. Новый экземпляр создаётся для каждого запроса, поэтому можно безопасно использовать свойства экземпляра (`this`) для хранения состояния запроса.

```ts
import { Middleware } from "inngest";

class MyMiddleware extends Middleware.BaseMiddleware {
  // A new instance is created for every request
  // Override hook methods here
}
```
</LanguageSection>

<LanguageSection show="python">
    Как видно выше, начинаем с метода `__init__`, вызываемого при инициализации клиента.

    ```py
    import inngest

        class MyMiddleware(inngest.Middleware):
            def __init__(
                self,
                client: inngest.Inngest,
                raw_request: object,
            ) -> None:
                # This runs when the client is initialized
                # Use this to set up anything your middleware needs
                #  ...
    ```
</LanguageSection>


<LanguageSection show="typescript">
    Методы хуков могут быть синхронными или `async` функциями. Для одноразовой настройки, например установки соединения с базой данных, используйте статический хук `onRegister`, выполняемый при регистрации класса промежуточного ПО с клиентом или функцией.

    ```ts
    class MyMiddleware extends Middleware.BaseMiddleware {
      static onRegister({ client }: Middleware.OnRegisterArgs) {
        // One-time setup, e.g. initialize a database connection
      }
    }
    ```
</LanguageSection>


<LanguageSection show="python">
    Регистрация функций, жизненные циклы и хуки могут быть синхронными или `async` функциями. Это позволяет обработчику инициализации выполнять асинхронную работу, например установку соединения с базой данных.

    ```py
    import inngest

        class MyMiddleware(inngest.Middleware):
            def __init__(
                self,
                client: inngest.Inngest,
                raw_request: object,
            ) -> None:
                #  ...connect to database
    ```
</LanguageSection>

Все функции жизненного цикла и хуков могут быть синхронными или `async` функциями — SDK всегда ожидает завершения функции промежуточного ПО перед переходом к следующей.

<Callout>
Поскольку приложение может использовать несколько клиентов Inngest, рекомендуется всегда инициализировать зависимости внутри функции/метода инициализатора, а не в глобальной области видимости.
</Callout>

## Определение жизненных циклов и хуков

<LanguageSection show="typescript">
    Переопределяйте нужные методы хуков прямо в классе промежуточного ПО. Полный список доступных хуков смотрите в [справочнике жизненного цикла промежуточного ПО](/docs/reference/typescript/v4/middleware/lifecycle).

    ```ts
    import { Middleware } from "inngest";

    class MyMiddleware extends Middleware.BaseMiddleware {
      // Observable hook: runs before the function handler on the first attempt
      onRunStart({ ctx, functionInfo }: Middleware.OnRunStartArgs) {
        console.log(`Starting ${functionInfo.id}`);
      }

      // Wrapping hook: wraps function execution
      async wrapFunctionHandler({ next }: Middleware.WrapFunctionHandlerArgs) {
        console.log("before");
        const result = await next();
        console.log("after");
        return result;
      }

      // Transform hook: modify function input (e.g. dependency injection)
      transformFunctionInput(args: Middleware.TransformFunctionInputArgs) {
        return {
          ...args,
          ctx: {
            ...args.ctx,
            // Add custom properties here
          },
        };
      }
    }
    ```

    Неопределённые хуки не создают накладных расходов: SDK полностью их пропускает. Новый экземпляр создаётся для каждого запроса, поэтому можно безопасно использовать свойства экземпляра (`this`) для хранения состояния запроса без риска утечки между запусками.

    Подробнее о хуках:

    - [Жизненный цикл](/docs/reference/typescript/v4/middleware/lifecycle) — порядок промежуточного ПО и все доступные хуки
    - [Примеры](/docs/reference/typescript/v4/middleware/examples) — практические примеры промежуточного ПО

</LanguageSection>

<LanguageSection show="python">
    Вы могли заметить, что наше промежуточное ПО определяет методы `before_send_events` и `after_send_events`. Эти методы, называемые хуками, позволяют промежуточному ПО встраиваться в определённые этапы жизненного цикла выполнения функций и шагов.

    ```py
    import inngest

    class MyMiddleware(inngest.Middleware):
        def __init__(
            self,
            client: inngest.Inngest,
            raw_request: object,
        ) -> None:
            #  ...

        async def before_send_events( self, events: list[inngest.Event]) -> None:
            # called before an event is sent from within a Function or Step
            print(f"Sending {len(events)} events")

        async def after_send_events(self, result: inngest.SendEventsResult) -> None:
            # called after an event is sent from within a Function or Step
            print("Done sending events")

    ```

    Полный список доступных хуков смотрите в [справочнике Python SDK](/docs/reference/python/middleware/lifecycle).

</LanguageSection>

## Добавление конфигурации

Часто промежуточному ПО требуется дополнительная настройка или параметры от разработчиков. Для этого рекомендуется создать функцию, принимающую параметры и возвращающую промежуточное ПО.

<LanguageSection show="typescript">
    ```ts {{ title: "inngest/middleware/myMiddleware.ts" }}
    import { Middleware } from "inngest";

    export function createMyMiddleware(logEventOutput: string) {
      return class MyMiddleware extends Middleware.BaseMiddleware {
        onRunComplete({ ctx, functionInfo, output }: Middleware.OnRunCompleteArgs) {
          if (ctx.event.name === logEventOutput) {
            console.log(
              `${logEventOutput} output: ${JSON.stringify(output)}`
            );
          }
        }
      };
    }
    ```
    ```ts
    import { createMyMiddleware } from "./middleware/myMiddleware";

    export const inngest = new Inngest({
      id: "my-client",
      middleware: [createMyMiddleware("app/user.created")],
    });
    ```

    <Callout>
    Позвольте TypeScript выводить возвращаемый тип функции вместо явной типизации — это помогает Inngest понимать изменения во входных и выходных аргументах. Подробнее смотрите в [жизненном цикле промежуточного ПО](/docs/reference/typescript/v4/middleware/lifecycle).
    </Callout>

</LanguageSection>

<LanguageSection show="python">
    Добавление конфигурации в пользовательское промежуточное ПО можно реализовать с помощью метода класса `factory()`, используя [паттерн фабрики](https://en.wikipedia.org/wiki/Factory_method_pattern).

    Например, добавим параметр конфигурации `secret_key` в промежуточное ПО `MyMiddleware`:

    ```py
    import inngest

    class MyMiddleware(inngest.Middleware):
        def __init__(
            self,
            client: inngest.Inngest,
            raw_request: object,
        ) -> None:
            #  ...

        @classmethod
        def factory(
            cls,
            secret_key: typing.Union[bytes, str],
        ) -> typing.Callable[[inngest.Inngest, object], MyMiddleware]:
            def _factory(
                client: inngest.Inngest,
                raw_request: object,
            ) -> MyMiddleware:
                return cls(
                    client,
                    raw_request,
                    secret_key,
                )

            return _factory

        async def before_send_events( self, events: list[inngest.Event]) -> None:
            # called before an event is sent from within a Function or Step
            print(f"Sending {len(events)} events")

        async def after_send_events(self, result: inngest.SendEventsResult) -> None:
            # called after an event is sent from within a Function or Step
            print("Done sending events")
    ```

    Теперь промежуточное ПО можно регистрировать следующим образом:

    ```py
    inngest_client = inngest.Inngest(
        app_id="my_app",
        middleware=[MyMiddleware.factory(_secret_key)],
    )
    ```
</LanguageSection>

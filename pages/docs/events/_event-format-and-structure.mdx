# Формат и структура события

События Inngest представляют собой обычный JSON, что делает их простыми в создании и чтении. Ниже приведён базовый пример со всеми обязательными и необязательными полями полезной нагрузки:

```js
await inngest.send({
  name: "api/user.signup",
  data: { method: "google_auth" },
  user: { id: "1JDydig4HHBJCiaGu2a9" },
  ts: new Date().valueOf(), // = 1663702869305
  v: "2022-09-20.1",
})
```

## Обязательные поля

- `name: String` — Название события. Рекомендуется использовать строчные буквы и точечную нотацию. Также рекомендуется использовать префиксы (например, `<prefix>/some.event`) для удобной организации событий.
- `data: Object` — Все данные, связанные с событием. Вы можете передать любые данные — они будут сериализованы как JSON. Вложенные данные поддерживаются, однако рекомендуется сохранять полезную нагрузку простой и, что важнее, согласованной.

## Необязательные поля

- `user: Object` — Любые данные, идентифицирующие пользователя, или атрибуты, связанные с событием. Все поля добавляются или обновляются в записи «Пользователь» в Inngest Cloud для группировки событий и упрощения отладки (см. раздел «Преимущества объекта user» ниже).
- `ts: Number` — Целое число **временной метки**, представляющее время (в миллисекундах), когда произошло событие. ПРИМЕЧАНИЕ: Inngest автоматически устанавливает это значение равным точному времени получения события, поэтому поле нужно только при использовании исторических событий или при необходимости точных значений.
- `v: String` — Идентификатор **версии** конкретного формата полезной нагрузки события. Версии полезны для фиксации изменений формата полезной нагрузки (схемы события). Рекомендуется формат `YYYY-MM-DD.N`, где `N` — целое число, увеличиваемое при каждом изменении. Поле необязательное, но очень полезное.

### Преимущества объекта `user` — отладка на основе пользователей

Объект `user` — это специальное поле в Inngest Cloud. Отправка данных в этом объекте позволяет использовать **единую отладку на основе пользователей и журналы аудита**.

Inngest создаёт и хранит единую идентификационную запись (профиль) для каждого уникального пользователя, для которого отправляются события. Это работает как «upsert» для любых данных, переданных в объекте `user`. Есть два ключевых способа использования этой функции:

- **Идентификаторы** — `id`, `email`, `phone` или любое поле, заканчивающееся на `_id`, используются для сопоставления и группировки событий в единую идентификационную запись. (Примеры: `stripe_customer_id`, `zendesk_id`)
- **Атрибуты** — Любое поле, не являющееся идентификатором, сохраняется как атрибут для вашего использования. Возможные варианты атрибутов: `billing_plan`, `signup_source` или `last_login_at`.

<aside>
  <p>🔐 <strong>Безопасность</strong> — Атрибуты каждого пользователя полностью шифруются в нашей базе данных с использованием шифрования на уровне строк. Каждый пользователь получает собственный ключ шифрования, который безвозвратно удаляется при удалении пользователя из нашей системы — это помогает соблюдать требования GDPR и CCPA.</p>

  <p>Данные пользователя доступны только в рамках запускаемых вами функций и из вашей аутентифицированной учётной записи. Они никогда не передаются другим пользователям или сотрудникам Inngest.</p>
</aside>

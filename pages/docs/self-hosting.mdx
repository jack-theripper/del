## Зачем размещать Inngest самостоятельно?

Хотя самый простой способ начать работу с Inngest — использовать нашу облачную платформу, включая щедрый [бесплатный тариф](/pricing?ref=docs-self-hosting), мы понимаем, что разработчики могут хотеть самостоятельного хостинга по разным причинам. Если вас беспокоят вопросы безопасности или конфиденциальности данных, ознакомьтесь с нашей [документацией по безопасности](/docs/learn/security?ref=docs-self-hosting), в том числе с информацией о [сквозном шифровании](/docs/features/middleware/encryption-middleware?ref=docs-self-hosting).

## Архитектура системы Inngest

Чтобы лучше понять, как развернуть Inngest самостоятельно, важно разобраться в архитектуре системы и её компонентах.

![Диаграмма архитектуры системы Inngest](/assets/docs/self-hosting/system-architecture-2024-09-23.png)

Система состоит из следующих сервисов:

* **Event API** — Принимает события от SDK через HTTP-запросы. Аутентифицирует клиентские запросы с помощью [Event Keys](/docs/events/creating-an-event-key?ref=docs-self-hosting). Event API публикует полезные данные событий во внутренний поток событий.
* **Event stream** — Выступает буфером между _Event API_ и _Runner_.
* **Runner** — Обрабатывает входящие события и выполняет несколько действий:
  * Планирование новых «запусков функций» (функциональных заданий) в зависимости от типа события, создание начального состояния запуска в базе данных _State store_. Запуски добавляются в очереди согласно конфигурации управления потоком функции.
  * Возобновление функций, приостановленных через [`waitForEvent`](/docs/features/inngest-functions/steps-workflows/wait-for-event?ref=docs-self-hosting), с совпадающими выражениями.
  * Отмена запущенных функций с совпадающими выражениями [`cancelOn`](/docs/features/inngest-functions/cancellation/cancel-on-events?ref=docs-self-hosting).
  * Запись поступивших событий в базу данных для истории и возможного воспроизведения.
* **Queue** — Многоарендная многоуровневая очередь, разработанная для справедливого распределения нагрузки и различных методов [управления потоком](/docs/guides/flow-control?ref=docs-self-hosting) (конкурентность, троттлинг, приоритизация, debouncing, ограничение частоты запросов) и [пакетной обработки](/docs/guides/batching?ref=docs-self-hosting).
* **Executor** — Отвечает за выполнение функций: от первоначального запуска и выполнения шагов до записи промежуточного состояния запуска в _State store_ и повторных попыток после сбоев.
* **State store (база данных)** — Хранит данные для ожидающих и выполняющихся запусков функций. Данные включают исходные триггерные события, результаты шагов и ошибки шагов.
* **Database** — Хранит системные данные и историю, включая приложения, функции, события и результаты запусков функций.
* **API** — GraphQL и REST API для программного доступа к системным ресурсам и управления ими.
* **Dashboard UI** — Интерфейс для управления приложениями, функциями и просмотра истории запусков.

Исходный код Inngest и всех сервисов [доступен на GitHub](https://github.com/inngest/inngest).

## Как развернуть Inngest самостоятельно

Чтобы начать самостоятельный хостинг Inngest, вам нужно только установить Inngest CLI. Inngest CLI — это единственный бинарный файл, включающий все сервисы Inngest и работающий в любом окружении. Либо вы можете скачать бинарный файл напрямую из [релизов GitHub](https://github.com/inngest/inngest/releases).

<CodeGroup>
```plaintext {{ title: "npm" }}
npm install -g inngest-cli
```
```plaintext {{ title: "Docker" }}
docker pull inngest/inngest
```
```plaintext {{ title: "curl" }}
curl -sfL https://cli.inngest.com/install.sh
```
</CodeGroup>

После установки CLI вы можете запустить сервер Inngest с помощью команды `inngest start`.

<CodeGroup>
```plaintext {{ title: "shell" }}
inngest start --event-key abcd --signing-key 1234
```
```plaintext {{ title: "Docker" }}
docker run -p 8288:8288 -p 8289:8289 -e INNGEST_EVENT_KEY=abcd -e INNGEST_SIGNING_KEY=1234 inngest/inngest inngest start
```
</CodeGroup>

Это запустит сервер Inngest на порту `8288` по умолчанию с использованием стандартной конфигурации, включая SQLite для хранения данных. Кроме того, шлюз Connect будет доступен на порту `8289`.

### Конфигурация

Настройка сервера может выполняться через флаги командной строки, переменные окружения или файл конфигурации.

По умолчанию сервер:

* Запускается на `localhost:8288` для обслуживания Event API, API и Dashboard UI. `localhost:8289` используется для `connect`-воркеров ([смотрите документацию для дополнительной конфигурации connect](/docs/setup/connect#self-hosted-inngest)).
* Использует in-memory Redis для очереди и хранилища состояний. (Смотрите раздел [Использование внешних сервисов](#using-external-services) для дополнительной информации.)
* Использует SQLite для хранения данных. База данных по умолчанию находится по пути `./.inngest/main.db`. Снимки очереди и хранилища состояний периодически сохраняются в базу данных SQLite, в том числе перед завершением работы.
* Отключает опрос синхронизации приложений для проверки новых функций или обновлённых конфигураций (смотрите флаг `--poll-interval`).

Для безопасной настройки сервера создайте ключи событий и ключи подписи в любом удобном формате и запустите сервер Inngest с их использованием. Их также можно передать через переменные окружения (смотрите ниже):

<Callout>
Ключ подписи должен быть корректной шестнадцатеричной строкой с чётным количеством символов. Вы можете сгенерировать безопасный ключ подписи с помощью:
```plaintext
openssl rand -hex 32
```
</Callout>

```plaintext
inngest start --event-key <ВАШ_EVENT_KEY> --signing-key <ВАШ_SIGNING_KEY>
```

Затем эти же ключи можно использовать как переменные окружения при запуске приложения (`INNGEST_EVENT_KEY` и `INNGEST_SIGNING_KEY`). [Смотрите ниже](#configuring-inngest-sdks-to-use-self-hosted-server) пример команды запуска Node.js.

### Настройка Inngest SDK для работы с самостоятельно развёрнутым сервером

По умолчанию Inngest SDK использует URL-адреса управляемой платформы Inngest. Чтобы подключиться к самостоятельно развёрнутому серверу, задайте переменные окружения [`INNGEST_DEV`](/docs/sdk/environment-variables#inngest-dev) и [`INNGEST_BASE_URL`](/docs/sdk/environment-variables#inngest-base-url). Как упоминалось выше, вам также нужно задать переменные окружения `INNGEST_EVENT_KEY` и `INNGEST_SIGNING_KEY` для безопасного подключения приложения к серверу Inngest.

Например, для подключения к самостоятельно развёрнутому серверу на `localhost:8288` в Node.js-приложении задайте следующие переменные окружения:

```plaintext
INNGEST_EVENT_KEY=<ВАШ_EVENT_KEY> \
  INNGEST_SIGNING_KEY=<ВАШ_SIGNING_KEY> \
  INNGEST_DEV=0 \
  INNGEST_BASE_URL=http://localhost:8288 \
  node ./server.js
```

### Использование внешних сервисов

Inngest можно настроить для использования внешних сервисов для очереди, хранилища состояний, а в ближайшем будущем — и базы данных.

**Внешний Redis-сервер**

Для упрощения начальной настройки сервер Inngest запускает in-memory Redis для очереди и хранилища состояний. Поскольку он работает в том же процессе, что и сервер Inngest, использование собственного Redis-сервера может улучшить производительность и надёжность системы. Вы можете запустить собственный Redis-сервер или использовать облачный Redis-сервис, например AWS ElastiCache, Redis Cloud и т.д.

Чтобы использовать внешний Redis-сервер, задайте URI Redis-сервера в флаге `redis-uri`.

**Внешняя база данных Postgres**

По умолчанию сервер Inngest использует SQLite для хранения данных. Это удобно для развёртывания без зависимостей, но не поддерживает масштабирование за пределы одного узла. Вы можете запустить собственную базу данных Postgres или использовать облачный сервис Postgres, например AWS RDS, Neon, Supabase и т.д.

Чтобы использовать внешнюю базу данных Postgres, задайте URI строки подключения Postgres в флаге `postgres-uri`.

## Пример Docker Compose

```yaml
services:
  inngest:
    image: inngest/inngest
    command: "inngest start"
    ports:
      - "8288:8288" # APIs and Dashboard
      - "8289:8289" # Connect WebSocket gateway
    environment:
      - INNGEST_EVENT_KEY=your_event_key_here # Must be hex string with even number of chars
      - INNGEST_SIGNING_KEY=your_signing_key_here # Must be hex string with even number of chars
      - INNGEST_POSTGRES_URI=postgres://inngest:password@postgres:5432/inngest
      - INNGEST_REDIS_URI=redis://redis:6379
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - inngest-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8288/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  postgres:
    image: postgres:17
    environment:
      - POSTGRES_DB=inngest
      - POSTGRES_USER=inngest
      - POSTGRES_PASSWORD=password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - inngest-network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U inngest -d inngest"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    networks:
      - inngest-network
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:
  postgres_data:
  redis_data:

networks:
  inngest-network:
    driver: bridge

```

## Helm-чарт

Inngest предоставляет Helm-чарт для готовых к продакшену развёртываний Kubernetes. Основные возможности:

**Быстрая настройка и готовность к продакшену**
- Устанавливается из OCI Registry ([смотрите документацию](https://github.com/inngest/inngest-helm?tab=readme-ov-file#installation))
- Поддерживает Kubernetes 1.20+ и Helm 3.0+ с простой установкой
- Встроенные PostgreSQL и Redis для лёгкой настройки с поддержкой внешних баз данных
- Единообразное именование ресурсов во всех окружениях независимо от имени Helm-релиза
- Примеры развёртывания для разработки, продакшена и гибридных сценариев

**Автомасштабирование и производительность**
- Автомасштабирование на основе KEDA с использованием метрик Prometheus Inngest
- Настраиваемое количество реплик, лимиты ресурсов и настройки воркеров очереди
- Горизонтальное автомасштабирование подов на основе глубины очереди

**Безопасность и доступ**
- Принцип «безопасность прежде всего»: выполнение без привилегий root и файловая система только для чтения
- Учётные данные базы данных хранятся в Kubernetes Secrets
- Поддержка сетевых политик для дополнительной изоляции кластера
- Встроенная поддержка ingress с выпуском SSL-сертификатов через cert-manager
- Интеграция с Let's Encrypt для продакшен HTTPS-эндпоинтов

Чарт, документацию и примеры конфигурации можно найти здесь:

# Настройка OpenTelemetry с Inngest <VersionBadge version="TypeScript only" />

[Extended Traces](/docs/platform/monitor/traces#extended-traces) Inngest позволяет перенаправлять OpenTelemetry-трассировки вашего приложения в
Inngest Traces для единого опыта наблюдаемости и отладки как в DevServer, так и на платформе.

В этом руководстве описано, как **настроить OpenTelemetry с Inngest Traces** и как **создавать пользовательские спаны**.

## Настройка Inngest Extended Traces с существующим OpenTelemetry-клиентом

Если ваше приложение уже использует OpenTelemetry-клиент, `extendedTracesMiddleware()` Inngest следует настроить корректно, чтобы
захватывать трассировки приложения без двойной регистрации инструментаций Node.js.

Вот настройка OpenTelemetry для Node.js-приложения (_экспорт трассировок через OTLP_) с использованием Inngest Extended Traces:

```ts tracing.ts
import { NodeSDK } from "@opentelemetry/sdk-node";
import { OTLPTraceExporter } from "@opentelemetry/exporter-trace-otlp-http";

import { Inngest } from "inngest";
import { InngestSpanProcessor, extendedTracesMiddleware } from "inngest/experimental";


export const inngest = new Inngest({
  id: "nodejs-open-telemetry-example",
  name: "NodeJS Open Telemetry Example",
  // set to "off" if your OTel client has some `instrumentations` configured
  middleware: [extendedTracesMiddleware({ behaviour: "auto" })],
});

// Configure OTLP endpoint for Jaeger using the HTTP exporter
// Jaeger typically accepts OTLP HTTP on http://localhost:4318/v1/traces
// Override via OTEL_EXPORTER_OTLP_ENDPOINT or OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
const traceExporter = new OTLPTraceExporter({
  url:
    process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT ||
    process.env.OTEL_EXPORTER_OTLP_ENDPOINT ||
    "http://localhost:4318/v1/traces",
});

const sdk = new NodeSDK({
  traceExporter: traceExporter,
  spanProcessors: [new InngestSpanProcessor(inngest)],
  // Set service name for Jaeger
  serviceName: "nodejs-open-telemetry-example",
});

sdk.start();

console.log("OpenTelemetry SDK started");
console.log(
  `Tracing endpoint: ${
    process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT ||
    process.env.OTEL_EXPORTER_OTLP_ENDPOINT ||
    "http://localhost:4318/v1/traces"
  }`
);
```

Эта настройка наиболее надёжна, так как не требует строгого порядка импортов для совместной работы OTel и Inngest Tracing.
С этой настройкой OTel-трассировки вашего приложения (например, Express API) продолжат экспортироваться в ваш OTLP-совместимый приёмник (например, Jaeger), в то время как
Inngest Extended Traces будут перенаправляться в Inngest DevServer или платформу.

<Callout>

**Какой режим `behaviour` для Extended Traces следует использовать?**

Если ваша текущая настройка OpenTelemetry включает автоинструментации (например, `@opentelemetry/auto-instrumentations-node`), установите режим `behaviour` в `"off"`, чтобы избежать дублирования трассировок.

Если вы хотите получить трассировки запросов к базе данных и HTTP-запросов в Inngest Traces, установите `behaviour` в `"auto"`.

</Callout>


## Настройка Inngest Extended Traces БЕЗ существующего OpenTelemetry-клиента

Настройка Inngest Extended Traces без существующего OpenTelemetry-клиента требует только добавления `extendedTracesMiddleware()`
в ваш клиент Inngest следующим образом:

 ```ts client.ts
// Create your client the same as you would normally
import { Inngest } from "inngest";
import { extendedTracesMiddleware } from "inngest/experimental";

export const inngest = new Inngest({
  id: "my-app",
  middleware: [extendedTracesMiddleware()]
});
```

Эта конфигурация обогатит ваши Inngest Traces спанами запросов к базе данных и HTTP-запросов.


## Создание пользовательских спанов с Inngest Extended Traces

После правильной настройки Inngest Extended Traces в вашем приложении ваш рабочий процесс Inngest получит дополнительный аргумент `tracer`:

```ts workflow.ts
import { inngest } from './client'

export const userOnboarding = inngest.createFunction(
  { id: "user-onboarding", triggers: { event: "user.onboarding" } },
  async ({ event, step, tracer }) => {
    //  ...
  }
);
```

Объект `tracer` является экземпляром `Tracer` из `@opentelemetry/api`, позволяя создавать пользовательские трассировки следующим образом:

```ts workflow.ts
import { inngest } from './client'
import { sendEmail } from './emails'

export const userOnboarding = inngest.createFunction(
  { id: "user-onboarding", triggers: { event: "user.onboarding" } },
  async ({ event, step, tracer }) => {
    await step.run("create-user", async () => {
        // ...
    });

    await step.run("send-welcome-email", async () => {
        tracer.startActiveSpan("call-email-service", async (span) => {
        span.setAttributes({ name, email });

        await sendEmail({ name, email })

        span.end();
      });
    });
  }
);
```

С помощью `tracer.startActiveSpan()` мы создаём пользовательский спан `call-email-service` для отслеживания производительности внешнего сервиса `sendEmail()`.

Наш пользовательский спан корректно отображается в трассировках запуска рабочего процесса Inngest:

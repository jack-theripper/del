# Повышение производительности

Inngest предлагает два взаимодополняющих подхода для снижения задержки в ваших функциях: **Checkpointing** для ускорения выполнения шагов и **Connect** для постоянных соединений с низкой задержкой между вашим приложением и Inngest.

---

## Checkpointing (Контрольные точки)

Checkpointing — это оптимизация производительности функций Inngest, которая выполняет шаги немедленно, не дожидаясь внутренней оркестрации.

### Различия в моделях выполнения

[Стандартная модель выполнения Inngest](/docs/learn/how-functions-are-executed) подразумевает полную передачу управления платформе Inngest, где после завершения каждого шага выполняется HTTP-запрос для сохранения состояния выполнения, что приводит к задержкам между шагами.

Checkpointing использует SDK для оркестрации шагов и запускает их на стороне клиента, что приводит к немедленному выполнению последующих синхронных шагов (например, `step.run()`) в функции. По мере выполнения шагов запросы на создание контрольных точек отправляются в Inngest для отслеживания прогресса при встрече с асинхронным шагом (например, `step.sleep()`).

При стандартной модели выполнения Inngest оркеструет вашу функцию, совершая сетевые обмены между каждым шагом. Это надежно, но добавляет задержку.

Checkpointing меняет это: SDK оркеструет шаги на стороне клиента (_на вашем сервере_) и выполняет их немедленно. По мере завершения шагов в Inngest отправляются сообщения контрольных точек для отслеживания прогресса. Результатом является значительно меньшая задержка — идеально для AI-воркфлоу в реальном времени.

### Сбои и повторные попытки

Что происходит, если что-то идет не так? Если шаг завершается сбоем и требует повторной попытки, движок выполнения возвращается к стандартной оркестрации для корректной обработки. Вы получаете скорость, когда все работает, и безопасность, когда возникают проблемы.

### Настройка Checkpointing

<LanguageSelector>

<LanguageSection show="typescript">

Checkpointing включен по умолчанию. Дополнительная конфигурация для долгоживущих или всегда включенных серверов не требуется.

Для серверлесс-платформ настройте `maxRuntime` немного меньше максимальной длительности вашей функции:

```ts
import { Inngest } from "inngest";

export const inngest = new Inngest({
  id: "my-app",
  checkpointing: {
    maxRuntime: '50s',
  },
});
```

Чтобы отключить checkpointing, установите `checkpointing: false` на клиенте или для конкретной функции.

</LanguageSection>

<LanguageSection show="go">

Чтобы включить checkpointing:

1. Установите пакет checkpoint:

```shell
go get github.com/inngest/inngestgo/pkg/checkpoint
```

2. Установите `Checkpoint` в опциях вашей функции:

```go
import (
  "github.com/inngest/inngestgo"
  "github.com/inngest/inngestgo/pkg/checkpoint"
)

_, err := inngestgo.CreateFunction(
  client,
  inngestgo.FunctionOpts{
    ID:         "my-function",
    Name:       "My Function",
    Checkpoint: checkpoint.ConfigSafe,
  },
  // ... триггеры и обработчик
)
```

## Connect

API `connect` позволяет вашему приложению создавать исходящее постоянное соединение с Inngest (_подход в стиле воркеров_). Это еще один способ снизить задержку за счет поддержания открытого WebSocket-соединения вместо выполнения HTTP-запросов для каждого шага.

### Преимущества производительности

По сравнению со стандартным подходом [`serve`](/docs/learn/serving-inngest-functions), `connect` предлагает:

- **Минимальная задержка** — Постоянные соединения устраняют накладные расходы на установку новых HTTP-соединений для каждого шага.
- **Упрощение длительных шагов** — Выполнение шагов не ограничено таймаутами HTTP платформы.
- **Эластичное горизонтальное масштабирование** — Легко добавляйте мощности, запуская дополнительные воркеры.
- **Идеально для контейнерных сред** — Развертывание в Kubernetes или ECS без необходимости использования балансировщика нагрузки для входящего трафика.

### Как это работает

API `connect` устанавливает постоянное WebSocket-соединение с Inngest. Каждое соединение может обрабатывать выполнение нескольких функций и шагов одновременно. Каждое приложение может создавать несколько соединений с Inngest, что обеспечивает горизонтальное масштабирование.

Ключевые особенности включают:

- **Автоматическое переподключение** — Соединение автоматически восстановится, если оно будет закрыто.
- **Плавное завершение работы** — Соединение плавно закрывается, когда приложение получает сигнал завершения (`SIGTERM`). Новые шаги не будут приниматься, но существующим шагам будет позволено завершиться.
- **Максимальная конкурентность на уровне воркера (Скоро)** — Каждый воркер может настроить максимальное количество одновременных шагов, которые он может обрабатывать, что позволяет Inngest распределять нагрузку между несколькими воркерами.

<Info>
  **WebSocket-соединение и откат к HTTP** — Пока WebSocket-соединение открыто, воркер отправляет и получает все результаты шагов через WebSocket. Когда соединение закрывается, воркер возвращается к использованию HTTP API для отправки оставшихся результатов шагов.
</Info>

# Интеграция Model Context Protocol (MCP)

Сервер разработки Inngest включает в себя полноценный сервер Model Context Protocol (MCP), который предоставляет ИИ-помощникам прямой доступ к вашим функциям Inngest, событиям и документации. Это открывает мощные возможности для разработки с помощью ИИ: тестирование, отладка и создание приложений Inngest.

## Что такое MCP?

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/introduction) — это стандарт для подключения ИИ-помощников к внешним инструментам и источникам данных. Интеграция Inngest MCP предоставляет ИИ-инструментам, таким как Claude Code и Cursor:

- **Полную видимость функций** — просмотр списка и анализ всех зарегистрированных функций.
- **Запуск событий** — отправка событий для тестирования функций и воркфлоу.
- **Мониторинг в реальном времени** — отслеживание выполнения функций и отладка сбоев.
- **Доступ к документации** — поиск и чтение документации Inngest в оффлайн-режиме.
- **Прямой вызов функций** — синхронное выполнение функций с немедленным получением результата.

<Callout>
Сервер Inngest MCP работает локально вместе с вашим сервером разработки и не требует внешних зависимостей, API-ключей или интернет-соединения.
</Callout>

## Быстрый старт

### 1. Запустите сервер разработки Inngest

MCP-сервер становится доступен автоматически при запуске сервера разработки Inngest:

```bash
npx --ignore-scripts=false inngest-cli@latest dev
```

Эндпоинт MCP будет доступен по адресу `http://127.0.0.1:8288/mcp`

### 2. Подключите ваш ИИ-инструмент

<CodeGroup>

```bash {{ title: "Claude Code" }}
# Добавьте в вашу конфигурацию MCP
claude mcp add --transport http inngest-dev http://127.0.0.1:8288/mcp
```

```json {{ title: "Cursor" }}
// Добавьте в .cursor/mcp.json
{
  "mcpServers": {
    "inngest-dev": {
      "url": "http://127.0.0.1:8288/mcp"
    }
  }
}
```

```json {{ title: "Claude Desktop" }}
// Добавьте в конфигурацию Claude Desktop
{
  "mcpServers": {
    "inngest-dev": {
      "command": "curl",
      "args": [
        "-X", "POST",
        "http://127.0.0.1:8288/mcp",
        "-H", "Content-Type: application/json",
        "-d", "@-"
      ]
    }
  }
}
```

</CodeGroup>

### 3. Начните разработку с ИИ

После подключения вы можете просить вашего ИИ-помощника:

```
Выведи список всех моих функций Inngest и их триггеров
```

```

```

## Лучшие практики

### Тестирование функций
- **Начинайте с простого:** Тестируйте отдельные функции перед проверкой сложных воркфлоу.
- **Используйте понятные события:** Четкие названия событий помогают при отладке.
- **Мониторьте выполнение:** Всегда проверяйте статус запуска после отправки событий.
- **Тестируйте сценарии с ошибками:** Намеренно вызывайте сбои для проверки обработки ошибок.

### Отладка воркфлоу
- **Проверяйте детали шагов:** Используйте `get_run_status`, чтобы увидеть пошаговое выполнение.
- **Изучайте контекст ошибок:** Сообщения об ошибках включают стек-трейсы и контекст.
- **Проверяйте поток данных:** Контролируйте входные и выходные данные на каждом шаге.
- **Используйте опрос (polling) для асинхронности:** Отслеживайте длительные воркфлоу с помощью `poll_run_status`.

### Использование документации
- **Ищите перед созданием:** Используйте `grep_docs`, чтобы найти подходящие примеры.
- **Обращайтесь к паттернам:** Ищите похожие случаи использования в документации.
- **Сверяйтесь с API:** Используйте `read_doc` для получения полной документации API.

## Доступные инструменты MCP

MCP-сервер Inngest предоставляет 8 мощных инструментов, разделенных на три категории:

### Инструменты управления событиями

#### `send_event`
Отправляет событие для запуска функций и мгновенно сообщает, какие запуски были созданы.

**Параметры:**
- `name` (string, обязательный): Название события (например, 'app/user.created').
- `data` (object, необязательный): Данные полезной нагрузки события.
- `user` (object, необязательный): Информация о контексте пользователя.
- `eventIdSeed` (string, необязательный): Сид для детерминированных ID событий.

**Пример использования:**
```
Отправь тестовое событие 'app/user.signup' с email пользователя 'test@example.com'
```

#### `list_functions`
Позволяет обнаружить все зарегистрированные функции с их названиями, ID и информацией о триггерах.

**Возвращает:** Полный инвентарь функций с деталями триггеров (события, cron-задачи и т.д.).

#### `invoke_function`
Напрямую выполняет функцию и ожидает ее полного завершения — идеально подходит для тестирования конкретных функций.

**Параметры:**
- `functionId` (string, обязательный): Слаг, ID или название функции.
- `data` (object, необязательный): Входные данные для функции.
- `user` (object, необязательный): Контекст пользователя.
- `timeout` (int, необязательный): Таймаут ожидания в секундах (по умолчанию: 30).

**Пример использования:**
```
Вызови функцию 'process-payment' с суммой 100 и валютой USD
```

### Инструменты мониторинга выполнения

#### `get_run_status`
Предоставляет подробную информацию о конкретном запуске функции, включая детали пошагового выполнения.

**Параметры:**
- `runId` (string, обязательный): ID запуска из send_event или логов.

**Возвращает:** Полную информацию о запуске, включая статус, шаги, выходные данные и детали ошибок.

#### `poll_run_status`
Отслеживает несколько запусков функций до их завершения — необходимо для интеграционного тестирования воркфлоу.

**Параметры:**
- `runIds` (array, обязательный): Массив ID запусков для мониторинга.
- `timeout` (int, необязательный): Общий таймаут опроса в секундах (по умолчанию: 30).
- `pollInterval` (int, необязательный): Миллисекунды между опросами (по умолчанию: 1000).

**Пример использования:**
```
Опрашивай эти 3 запуска до их завершения и покажи сводку
```

### Инструменты документации

#### `grep_docs`
Ищет по встроенной документации Inngest с использованием сопоставления с образцом.

**Параметры:**
- `pattern` (string, обязательный): Шаблон поиска (поддерживаются регулярные выражения).
- `limit` (int, необязательный): Максимальное количество результатов (по умолчанию: 10).

**Пример использования:**
```
Найди в документации примеры 'rate limiting'
```

#### `read_doc`
Читает полное содержимое конкретного файла документации.

**Параметры:**
- `path` (string, обязательный): Путь к документу относительно директории docs.

#### `list_docs`
Предоставляет обзор всей доступной документации с разбивкой по категориям.

**Возвращает:** Структуру документации с количеством документов по категориям и SDK.

## Примеры использования

### Тестирование воркфлоу регистрации пользователя

```typescript
// Ваша функция
export const handleSignup = inngest.createFunction(
  { id: "handle-signup", triggers: { event: "app/user.signup" } },
  async ({ event, step }) => {
    const user = await step.run("create-user", async () => {
      return createUser(event.data.email);
    });
    
    await step.run("send-welcome-email", async () => {
      return sendWelcomeEmail(user.email);
    });
    
    return { success: true, userId: user.id };
  }
);
```

С помощью MCP вы можете протестировать этот воркфлоу, попросив своего ИИ-помощника:

```
Отправь тестовое событие регистрации с email 'alice@example.com' и отслеживай выполнение функции
```

ИИ выполнит следующие действия:
1. Использует `send_event` для запуска функции.
2. Использует `poll_run_status` для мониторинга выполнения.
3. Покажет вам пошаговый прогресс и финальные результаты.
4. Сообщит о любых ошибках с подробным контекстом.

### Отладка упавших функций

```
Получи статус запуска 01J5QH90... и объясни, что пошло не так
```

ИИ использует `get_run_status` для получения:
- Полного трейса выполнения.
- Пошагового статуса.
- Сообщений об ошибках и стек-трейсов.
- Результатов функции и промежуточных данных.

### Интеграционное тестирование

```
Отправь события для запуска полного воркфлоу обработки заказа и убедись, что все шаги завершились успешно
```

ИИ может:
1. Отправлять несколько скоординированных событий.
2. Мониторить все результирующие запуски функций.
3. Проверять завершение всего воркфлоу.
4. Сообщать о любых сбоях или таймаутах.

## Устранение неполадок

### Общие проблемы

**MCP-сервер не найден**
- Убедитесь, что сервер разработки Inngest запущен: `npx --ignore-scripts=false inngest-cli@latest dev`
- Проверьте доступность эндпоинта MCP: `curl http://127.0.0.1:8288/mcp`
- Проверьте конфигурацию вашего MCP-клиента.

**Функции не отображаются в списке**
- Подтвердите, что функции правильно зарегистрированы на сервере разработки.
- Проверьте логи сервера разработки на наличие ошибок регистрации.
- Убедитесь, что ваше приложение правильно синхронизировано с сервером разработки.

**Запуски не найдены после отправки событий**
- Подождите немного, пока событие обработается (встроена задержка 500 мс).
- Проверьте, совпадает ли название события с триггерами вашей функции.
- Убедитесь, что функция действительно зарегистрирована и активна.

**Таймауты опроса (polling timeouts)**
- Увеличьте значения таймаута для длительных функций.
- Проверьте логи функций на наличие зависших операций.
- Убедитесь, что функции действительно завершаются, а не висят бесконечно.

### Дополнительная конфигурация

**Пользовательские таймауты:**
Таймаут опроса по умолчанию составляет 30 секунд. Для более длительных функций:

```
Опрашивай этот запуск с таймаутом 60 секунд вместо стандартного
```

**Сидирование ID событий (Event ID seeding):**
Для детерминированного тестирования вы можете предоставить сиды для ID событий:

```
Отправь тестовое событие с сидом 'test-123' для воспроизводимого тестирования
```

**Поиск по документации:**
Используйте регулярные выражения для продвинутого поиска по документации:

```
Ищи в документации 'step\\.run.*retry', чтобы найти примеры повторных попыток для шагов
```
